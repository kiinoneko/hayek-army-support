# Runbook: 議会だより・要約資料照合 v0.1

## 1. Runbookの位置づけ

議会だより・要約資料照合Runbookは、原文発言と要約資料の差分を確認するための作業手順書である。

- 要約資料は便利だが、原文そのものではない。
- 議会だより、広報紙、審議会要旨、公式要約は、編集・省略・要約・紙幅制約を含む可能性がある。
- 発言評価、矛盾指摘、政策態度分析は、可能な限り原文資料に戻って確認する。
- `quote_id`は原文発言と要約掲載文を区別して管理する。
- `source_id`なしに原文照合済みとしない。
- 要約と原文の差分は、直ちに不正、隠蔽、改ざんと断定しない。
- 差分が重要な場合は`warning_id`又は`gap_id`として管理する。
- 外部発信や評価断定には必要に応じて`decision_id`を要求する。

## 2. 対象

本Runbookは、以下の資料を対象とする。

- 議会だより
- 広報紙
- 議会公式サイトの要約
- 一般質問要約
- 委員会要旨
- 審議会要旨
- 協議会要旨
- 検討会要旨
- 公式説明資料
- 国会会議録要約
- 政府審議会資料
- 省庁会議資料
- 首長会見要旨
- 報道向け発表資料
- SNS等での本人要約
- 議事録
- 会議録
- 議会録画
- 会見動画
- 発言通告書
- 答弁書

## 3. 使用する主なID

| ID | 用途 |
|---|---|
| `case_id` | 照合案件を管理する |
| `source_id` | 原文資料、要約資料を管理する |
| `quote_id` | 原文発言、要約掲載文を管理する |
| `actor_id` | 発言者、質問者、答弁者、編集責任線を管理する |
| `policy_id` | 関連政策、議案、事業、予算、契約を管理する |
| `gap_id` | 原文未確認、要約未確認、照合不能を管理する |
| `warning_id` | 要約差分、文脈注意、紙幅制約等を管理する |
| `stop_id` | 原文未確認、外部発信停止、断定停止を管理する |
| `decision_id` | 外部発信、評価断定、法務官引渡を管理する |
| `change_id` | 照合結果、要約、評価の変更履歴を管理する |
| `incident_id` | 誤引用、誤要約、人物誤認等を管理する |

## 4. 基本原則

- 原文、要約、解釈、評価を分ける。
- 議会だより等の要約資料を、原文発言として扱わない。
- 要約資料に載っていない発言を、発言不存在と断定しない。
- 原文にあるが要約にない場合は、未掲載又は省略として記録する。
- 要約が原文より弱い、強い、短い、違う場合は差分として記録する。
- 差分があるだけで隠蔽、改ざん、虚偽と断定しない。
- 発言者、発言日、会議名、媒体、ページ、発言箇所を確認する。
- 外部発信前に`source_id`、`quote_id`、`actor_id`、`warning_id`を確認する。

## 5. 原文資料確認手順

| 手順 | 内容 |
|---|---|
| 1 | 議事録、会議録、録画、答弁書等の原文資料を確認する |
| 2 | `source_id`を発行する |
| 3 | 発言者を`actor_id`で確認する |
| 4 | 発言日時、会議名、議題、発言区分を確認する |
| 5 | 原文発言を`quote_id`化する |
| 6 | 前後文脈を確認する |
| 7 | 関連`policy_id`へ接続する |
| 8 | 原文未確認箇所を`gap_id`化する |

## 6. 要約資料確認手順

| 手順 | 内容 |
|---|---|
| 1 | 議会だより、広報紙、公式要約等を確認する |
| 2 | `source_id`を発行する |
| 3 | 掲載ページ、掲載欄、見出しを確認する |
| 4 | 要約掲載文を`quote_id`候補として記録する |
| 5 | 要約作成主体、編集主体、発行主体を確認する |
| 6 | 原文発言との対応関係を確認する |
| 7 | 要約に含まれない論点を確認する |
| 8 | 紙幅制約、編集方針、掲載ルールがある場合は`warning_id`化する |

## 7. 原文・要約照合手順

| 照合項目 | 確認内容 |
|---|---|
| 発言者 | 原文と要約で同一か |
| 発言日 | 対象会議・対象号が一致するか |
| 論点 | 原文の主要論点が要約に残っているか |
| 数値 | 数値が省略・変更されていないか |
| 主語 | 誰の発言か曖昧になっていないか |
| 断定度 | 原文より強く又は弱くなっていないか |
| 条件 | 但し書き、前提条件が落ちていないか |
| 答弁 | 質問だけでなく答弁も対応しているか |
| 文脈 | 前後のやり取りが必要か |
| 未掲載 | 原文にあるが要約にない発言があるか |

## 8. 差分分類

| 差分分類 | 内容 | 対応 |
|---|---|---|
| 一致 | 原文と要約が概ね一致 | 通常記録 |
| 省略 | 原文の一部が掲載されていない | `warning_id`候補 |
| 弱体化 | 原文より表現が弱い | `warning_id`候補 |
| 強調 | 原文より表現が強い | `warning_id`候補 |
| 文脈欠落 | 前後条件が落ちている | `warning_id`候補 |
| 数値欠落 | 重要数値が省略されている | `warning_id`又は`gap_id` |
| 主語曖昧 | 誰の発言か曖昧 | `gap_id`候補 |
| 論点転換 | 原文の焦点と要約の焦点が違う | `warning_id`候補 |
| 未掲載 | 原文にあるが要約にない | `warning_id`候補 |
| 誤要約疑い | 意味が変わっている可能性 | `stop_id`候補 |

## 9. actor_id / quote_id / policy_id 接続

- 原文発言の`quote_id`と、要約掲載文の`quote_id`を分ける。
- 両者を同一発言として混同しない。
- `actor_id`は発言者本人を明記する。
- 要約資料の発行主体、編集主体も必要に応じて`actor_id`又は`source_id`備考へ記録する。
- `policy_id`は関連する政策、事業、議案、予算、契約へ接続する。
- `policy_id`だけで政策効果、失敗、不正を断定しない。

## 10. gap_id / warning_id / stop_id 判定

| 状態 | 発行ID | 対応 |
|---|---|---|
| 原文資料が未確認 | `GAP_` 又は `STOP_` | 原文確認まで正式照合停止 |
| 要約資料のみ確認 | `WARN_` 又は `STOP_` | 原文扱いしない |
| 発言者不明 | `GAP_` 又は `STOP_` | `actor_id`確認 |
| 掲載箇所不明 | `GAP_` | ページ、欄を確認 |
| 文脈欠落 | `WARN_` | 前後文脈を明記 |
| 省略・弱体化・強調 | `WARN_` | 差分分類として記録 |
| 意味変更疑い | `STOP_` | 原文照合又は法務官確認 |
| 外部発信予定 | `DEC_` | `USER_DECISION` |

## 11. 議会だより特有の注意

- 議会だよりは紙幅制約により要約される。
- 一般質問の掲載文は、議員本人又は議会広報側の編集が入る場合がある。
- 要約文は原文ではない。
- 掲載されていない論点がある場合も、直ちに隠蔽とは断定しない。
- 議会だよりと議事録の差分は、原文確認後に記録する。
- 要約が住民向け説明として十分かどうかは、別途評価対象とする。
- 外部発信する場合は、原文と要約を並べて示す。

## 12. 国会・政府・審議会要約資料の注意

- 国会、政府、中央省庁、審議会、検討会の要旨資料も対象に含める。
- 議事要旨は逐語録ではない場合がある。
- 配布資料、議事要旨、議事録、録画を区別する。
- 委員名、発言者名が伏せられている場合は、推定で`actor_id`化しない。
- 要約資料だけで政府方針、審議会結論、委員発言を断定しない。
- 政府資料は更新・差替えがあるため取得日、版、URLを記録する。

## 13. 外部発信前チェック

公開前チェックリストを使用する。

- 原文`source_id`と要約`source_id`の両方を確認する。
- 原文`quote_id`と要約`quote_id`を分けているか確認する。
- `actor_id`を確認する。
- `policy_id`を確認する。
- `gap_id`を隠していないか確認する。
- `warning_id`を反映しているか確認する。
- `stop_id`がある場合は公開しない。
- 要約差分を直ちに隠蔽、改ざん、不正と断定しない。
- 外部発信には必要に応じて`decision_id`を取得する。

## 15. 禁止事項

- 議会だよりを原文として扱わない。
- 要約資料だけで発言原文を確定しない。
- 要約資料を原文発言として引用しない。
- `source_id`なしで原文照合済みと記録しない。
- 原文未確認のまま矛盾、不正、隠蔽、改ざん、虚偽と断定しない。
- 要約の省略だけで悪意、隠蔽、改ざん、不正を断定しない。
- 要約に未掲載であることだけを理由に、発言不存在と断定しない。
- 発言者未確認のまま`actor_id`を確定しない。
- 発言者名が伏せられている資料から、推定だけで`actor_id`を確定しない。
- `quote_id`を`source_id`なしに正式化しない。
- 原文`quote_id`と要約`quote_id`を混同しない。
- 原文`quote_id`と要約`quote_id`を同一IDとして扱わない。
- 要約と評価を同じ欄に混在させない。
- 紙幅制約、編集方針、媒体特性を無視して評価断定しない。
- `gap_id`を隠して外部発信しない。
- `warning_id`を無視して評価しない。
- `gap_id`又は`stop_id`が残る案件を、確認済み案件として外部発信しない。
- `stop_id`がある状態で公開しない。
- `decision_id`なしに外部発信、法務官引渡、評価断定へ進まない。
- PDF原本、個人情報、黒塗り前資料、APIキー、secretをGitHubに追加しない。

## 16. PR条件

- 本Runbookを追加又は変更するPRでは、対象範囲、ID接続、外部発信停止条件に影響がないか確認する。
- 原文、要約、解釈、評価の分離原則を弱める変更を含めない。
- `quote_id`、`actor_id`、`source_id`、`policy_id`、`gap_id`、`warning_id`、`stop_id`、`decision_id`の扱いを変更する場合は、変更理由をPR本文に明記する。
- 要約差分を不正、隠蔽、改ざんと即断しない方針を維持する。
- 外部発信や評価断定に関わる変更は、必要に応じて`decision_id`又はレビュー担当者の確認を要求する。
- 既存本文を不要に改変しない。
- 既存ファイルを削除しない。
- archive配下を変更しない。
- PDF原本を追加しない。
- 個人情報を追加しない。
- 黒塗り前資料を追加しない。
- APIキー、パスワード、トークンを追加しない。
- mainへ直接コミットしない。
- PRでは自動マージを行わない。
