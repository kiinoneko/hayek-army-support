# Runbook: 原文資料相互検証 v0.1

## 1. 位置づけ

このRunbookは、`source_id` が存在する資料について、原文、ページ、日時、発言者、資料名、金額、表、引用箇所を相互検証するための運用手順である。

`source_id` は資料候補又は根拠資料を追跡するためのIDであり、`source_id` が存在するだけでは、本文確認済み、引用確認済み、数値確認済み、公開使用可とは扱わない。

本Runbookでは、確認状態を次のように分けて扱う。

- 資料の存在を確認した状態
- 原文本文を確認した状態
- 該当ページ、表、発言箇所を確認した状態
- 金額、件数、年度、単位を確認した状態
- 引用文と原文の一致を確認した状態
- 別資料と照合した状態
- 反証又は矛盾資料を確認した状態
- 公開使用可と判断できる状態

確認状態を分けることで、未確認情報を断定せず、確認済みの範囲だけを成果物、台帳、公開前チェックへ引き渡す。

## 2. 適用対象

本Runbookは、次の資料又は情報を扱う場合に適用する。

- 自治体、議会、省庁、公的機関が公開する原資料
- 予算書、決算書、決算カード、事務事業評価、財政資料
- 議事録、会議録、委員会記録、会見録、答弁書
- 議会だより、広報紙、会議要旨、説明資料、概要資料
- 入札、契約、委託、補助金、指定管理に関する資料
- 統計表、CSV、Excel、PDF内の表、HTML表
- 動画、音声、文字起こし、動画切り取り、字幕
- SNS投稿、プレスリリース、報道、団体資料、個人発信
- 情報公開請求で取得した資料のうち、公開使用を検討するもの
- 外部発信、カード化、公開用PDF、広報PR、法務官引渡の根拠にするもの

PDF、画像、個人情報、secret、API key、黒塗り前情報そのものは、GitHubへ追加しない。必要な場合は、保存場所、取得日、確認状態、公開使用可否を台帳で管理する。

## 3. source_id確認状態との関係

`source_id` は、資料を識別し、再確認できるようにするための入口である。`source_id` の登録と、URL確認、本文確認、ページ確認、引用確認、数値確認、別資料照合、反証確認、公開使用可は別の確認状態として扱う。

| 状態 | 意味 | 使用上の扱い |
| --- | --- | --- |
| source_id登録済み | 資料候補又は根拠資料としてIDを登録した。 | これだけでは断定根拠にしない。 |
| URL確認済み | URL、保存場所、資料名、公開主体を確認した。 | 本文内容、数値、引用の根拠とは扱わない。 |
| 本文確認済み | 原文本文を開き、対象内容の存在を確認した。 | 概要根拠にできる場合があるが、ページ、数値、引用は別途確認する。 |
| ページ確認済み | 該当ページ、表、見出し、セル、発言箇所を確認した。 | 引用、数値、要約の前提にできる。 |
| 引用確認済み | 引用文と原文の一致、引用範囲、省略、改変有無を確認した。 | 直接引用に使える。 |
| 数値確認済み | 金額、件数、割合、年度、単位、算式を確認した。 | 数値を成果物へ反映できる。 |
| 別資料照合済み | 別資料、原資料、要約資料と照合した。 | 公開使用可の判断材料にできる。 |
| 反証確認済み | 反証資料、別解釈、訂正版、差替版、前後文脈を確認した。 | 断定度と公開使用可の判断材料にできる。 |
| 公開使用可 | 原文、ページ、数値、引用、反証、注意事項を確認し、停止条件がない。 | 公開前チェックへ進める。 |

原文未確認の場合は、重要断定、金額確定、発言者名を伴う批判、公開用PDF化へ進めない。必要に応じて `gap_id`、`warning_id`、`stop_id` を接続する。

## 4. 原文確認項目

原文確認では、少なくとも次の項目を確認する。

| 確認項目 | 確認内容 |
| --- | --- |
| 資料名 | 表紙、タイトル、ファイル名、ページ見出しが一致するか。 |
| 公開主体 | 作成者、公開者、議会、自治体、部署、団体を確認したか。 |
| 公開日・作成日 | 公開日、作成日、会議日、告示日、更新日を確認したか。 |
| 取得日 | 作業者が資料へアクセス又は取得した日を記録したか。 |
| 対象年度・期間 | 会計年度、事業年度、対象期間、基準日を確認したか。 |
| ページ・箇所 | ページ番号、表番号、見出し、項目、セル、発言箇所を特定したか。 |
| 発言者 | 発言者、質問者、答弁者、司会、編集主体を区別したか。 |
| 文脈 | 前後の発言、脚注、注記、但し書き、条件を確認したか。 |
| 版・差替 | 版番号、改訂、差替、更新、リンク移転の可能性を確認したか。 |
| 公開使用制約 | 個人情報、secret、著作権上の引用範囲、黒塗り前情報を確認したか。 |

確認できない項目がある場合は、未確認のまま使用可能範囲を広げず、台帳に未確認項目として記録する。

## 5. 要約資料・議会だより・動画切り取り・SNS投稿の扱い

要約資料、議会だより、動画切り取り、SNS投稿は、原文又は全体資料の代替として扱わない。

| 資料種別 | 扱い | 注意点 |
| --- | --- | --- |
| 要約資料 | 原資料の要約又は説明として扱う。 | 原文の発言、数値、制度根拠をそのまま示すものとは限らない。 |
| 議会だより | 編集済みの住民向け資料として扱う。 | 紙幅制約、省略、表現調整があり得る。発言原文とは分ける。 |
| 動画切り取り | 発言断片又は編集済み素材として扱う。 | 開始時刻、終了時刻、前後文脈、字幕の正確性を確認する。 |
| SNS投稿 | 発信者の投稿又は引用投稿として扱う。 | 本人確認、投稿日時、削除、編集、スクリーンショット由来の改変可能性を確認する。 |
| 報道・第三者記事 | 補助資料として扱う。 | 原資料、公式発表、発言録、統計表へ戻って確認する。 |

要約資料のみを確認した状態では、本文確認済みとはしない。議会だよりやSNS投稿の表現を根拠に、発言者の原文発言、意図、違法性、不正、虚偽を断定しない。

## 6. 数値確認

金額、件数、割合、人数、順位、増減率、契約額、補助額などを扱う場合は、次を確認する。

1. 数値の出所となる `source_id` を確認する。
2. 原資料のページ、表番号、行、列、セル、注記を確認する。
3. 単位が円、千円、百万円、件、人、％などのどれかを確認する。
4. 税込・税抜、当初予算・補正予算・決算、一般会計・特別会計を区別する。
5. 年度、対象期間、基準日、集計時点を確認する。
6. 転記した数値と原資料の数値を照合する。
7. 計算値を使う場合は、分子、分母、算式、丸め、端数処理を記録する。
8. 複数資料で数値が異なる場合は、差異、資料の版、更新日、対象範囲を記録する。
9. 金額確定を行う前に、原文、表、単位、年度、注記を再確認する。

数値確認が未了の場合は、金額確定、増減率断定、順位断定、損害額断定、費用対効果断定へ進めない。

## 7. 引用確認

引用を行う場合は、原文確認とは別に引用確認を行う。

| 確認項目 | 確認内容 |
| --- | --- |
| 引用元 | `source_id`、資料名、ページ、発言箇所、表番号を記録したか。 |
| 引用範囲 | 引用開始位置、終了位置、省略箇所、前後文脈を確認したか。 |
| 文字一致 | 文字、数字、句読点、括弧、単位、固有名詞が原文と一致するか。 |
| 省略表示 | 中略、前略、後略を明示し、意味を変えていないか。 |
| 発言者 | 発言者、質問者、答弁者、引用者、編集者を混同していないか。 |
| 翻刻・字幕 | 音声、動画、字幕、文字起こしの場合、聞き取り不確実箇所を分けたか。 |
| 公開制約 | 個人情報、secret、非公開情報、過度な転載を含まないか。 |

引用確認が未了の場合は、直接引用として使わず、要約又は確認待ちとして扱う。発言者名を伴う批判に使う場合は、引用確認に加えて前後文脈と反証確認を行う。

## 8. 別資料照合

重要な事実、金額、発言、制度根拠を扱う場合は、可能な範囲で別資料と照合する。

- 原資料と要約資料を照合する。
- 予算書と決算書を照合する。
- 契約書、入札結果、仕様書、支出命令、事業評価を照合する。
- 議事録、会議録、録画、議会だよりを照合する。
- 公告、告示、要綱、条例、規則、説明資料を照合する。
- 統計表と本文説明を照合する。
- 最新版と取得済み版を照合する。

別資料の内容が一致しない場合は、直ちに不正、虚偽、隠蔽、改ざんとは断定しない。対象年度、版、集計範囲、単位、編集方針、公開日、更新日、資料種別の違いを確認する。

## 9. 反証確認

外部発信、批判、問題提起、法務官引渡、公開用PDF化を行う前に、反証又は別解釈の可能性を確認する。

反証確認では、次を確認する。

- 同じ事実を否定又は修正する公式資料がないか。
- 最新版、訂正版、差替版、FAQ、補足説明がないか。
- 発言の前後文脈に、条件、留保、訂正、撤回がないか。
- 金額や件数が別会計、別年度、別事業、別単位の数値ではないか。
- 要約資料の省略が、紙幅制約や編集方針によるものではないか。
- SNS投稿や動画切り取りが、全体文脈を欠いていないか。
- 当事者側の説明、議会答弁、監査結果、第三者報告書がないか。

反証確認が未了の場合は、断定度を下げるか、`warning_id` 又は `stop_id` を接続する。

## 10. gap_id / warning_id / stop_id への接続

相互検証中に未確認、注意、停止条件が見つかった場合は、次のIDへ接続する。

| ID | 接続する状態 | 対応 |
| --- | --- | --- |
| `gap_id` | 原資料未取得、ページ不明、年度不明、数値差異、発言箇所不明、反証未確認がある。 | 未確認事項として記録し、追加確認又は情報公開請求へつなぐ。 |
| `warning_id` | 要約資料のみ確認、更新可能性あり、引用注意、文脈注意、数値単位注意がある。 | 成果物、台帳、公開前チェックへ注意事項として引き渡す。 |
| `stop_id` | 原文未確認のまま重要断定が必要、金額確定が必要、発言者名を伴う批判が必要、個人情報又はsecret混入疑いがある。 | 公開、公開用PDF化、広報PR、外部発信、法務官引渡を停止する。 |

原文未確認の場合は、重要断定、金額確定、発言者名を伴う批判、公開用PDF化に進めない。この状態は原則として `stop_id` 候補として扱い、軽微な内部メモに限る場合でも `warning_id` 又は `gap_id` で確認状態を残す。

## 11. ID台帳への記録

相互検証の結果は、ID台帳へ記録する。少なくとも次の項目を残す。

| 項目 | 記録内容 |
| --- | --- |
| `source_id` | 対象資料のID。 |
| `case_id` | 関連案件ID。複数ある場合は併記する。 |
| 資料名 | 正式名、版、号、会議名、表題。 |
| 公開主体 | 自治体、議会、部署、省庁、団体、発信者。 |
| URL又は保存場所 | 資料本体のURL、保存場所、取得元。 |
| 取得日 | 作業者がアクセス、取得、保存、確認した日。 |
| 公開日・作成日 | 資料側の公開日、作成日、会議日、更新日。 |
| 原資料区分 | 原資料、要約資料、加工資料、動画、SNS、報道等。 |
| 原文確認状態 | 未確認、URL確認済み、本文確認済み、ページ確認済み等。 |
| 数値確認状態 | 未確認、照合済み、算式確認済み、差異あり、再確認必要等。 |
| 引用確認状態 | 未確認、直接引用可、要約のみ、引用不可、要確認等。 |
| 照合先ID | 別資料の `source_id`、`quote_id`、`actor_id`、`policy_id` 等。 |
| 関連ID | `gap_id`、`warning_id`、`stop_id`、`decision_id` 等。 |
| 備考 | 差異、未確認事項、反証、更新可能性、公開使用制約。 |

台帳には、確認済みの事実と作業者の推測を分けて記録する。推測を記録する場合は、推測であること、確認に必要な次工程、停止条件を明示する。

## 12. 公開使用可の条件

公開使用可と扱うには、用途に応じて次の条件を満たす必要がある。

- `source_id`、資料名、公開主体、URL又は保存場所、取得日が記録されている。
- 原文本文を確認している。
- 該当ページ、表、発言箇所、見出し、セル等を特定している。
- 金額、件数、割合、年度、単位を使う場合は数値確認が完了している。
- 直接引用を使う場合は引用確認が完了している。
- 発言者名を伴う批判を行う場合は、発言者、日時、会議名、前後文脈、反証確認が完了している。
- 要約資料、議会だより、動画切り取り、SNS投稿を原文と混同していない。
- 別資料照合又は反証確認で重要な矛盾が残っていない。
- `gap_id`、`warning_id`、`stop_id` がある場合は、成果物へ反映又は停止判断されている。
- 個人情報、secret、API key、黒塗り前情報、公開不可資料を含めていない。

公開使用可は、全用途に一律で付与する状態ではない。内部メモで使用可、要約で使用可、数値で使用可、直接引用で使用可、公開用PDFで使用可を分けて記録する。

## 13. 禁止事項

次の行為は禁止する。

- `source_id` があるだけで、本文確認済み、引用確認済み、数値確認済み、公開使用可として扱うこと。
- 原文未確認のまま、重要断定、金額確定、発言者名を伴う批判、公開用PDF化へ進むこと。
- 要約資料、議会だより、動画切り取り、SNS投稿を原文として扱うこと。
- ページ、表、単位、年度、注記を確認せずに数値を確定すること。
- 引用範囲、前後文脈、発言者を確認せずに直接引用すること。
- 別資料との差異だけで、不正、虚偽、隠蔽、改ざんと断定すること。
- 反証確認を行わずに、外部発信、批判、法務官引渡を進めること。
- 個人情報、secret、API key、黒塗り前情報、非公開資料そのものをGitHubへ追加すること。
- 確認状態、未確認事項、注意事項、停止条件を台帳に残さずに公開前チェックへ進むこと。

## 14. PR前確認

PR前に、作業者は次を確認する。

- 新規追加又は更新した資料に、不要なPDF、画像、個人情報、secret、API keyが含まれていない。
- `source_id` の存在だけで確認済み扱いにしていない。
- 原文未確認の情報を、重要断定、金額確定、発言者名を伴う批判、公開用PDF化へ進めていない。
- 要約資料、議会だより、動画切り取り、SNS投稿を原文と分けている。
- 数値を使う箇所は、原資料、ページ、表、単位、年度、注記を確認している。
- 引用を使う箇所は、引用範囲、原文一致、発言者、前後文脈を確認している。
- 別資料照合と反証確認の結果を台帳又は備考へ記録している。
- 未確認、注意、停止条件を `gap_id`、`warning_id`、`stop_id` に接続している。
- 公開使用可の範囲を、内部メモ、要約、数値、直接引用、公開用PDFで分けている。
- 既存のREADME.md、archive、00_index、04_templates、既存05_toolsファイルを変更していない。
