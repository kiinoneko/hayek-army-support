# Runbook: source_id根拠資料管理 v0.1

## 1. 位置づけ

このRunbookは、後方支援軍が扱う根拠資料に `source_id` を付与し、根拠資料の確認状態、取得日、引用可否、数値確認状態を一貫して管理するための運用手順である。

`source_id` は、事実、数値、引用、年度、公開主体、資料の所在を後から再確認できる状態にするための基礎IDである。根拠資料を確認しないまま、成果物、分析、外部発信、公開前チェックへ進めてはならない。

このRunbookは、次を標準化する。

- `source_id` の正式化
- 根拠資料の登録と確認状態の分類
- 原資料、要約資料、加工資料の区別
- 本文、ページ、引用、数値の確認手順
- URL、取得日、公開主体、更新・削除時の扱い
- `gap_id` / `warning_id` / `stop_id` / `case_id` との接続
- 公開前チェックへ引き渡す条件

## 2. source_idの定義

`source_id` とは、後方支援軍が根拠として参照する資料、データ、Webページ、PDF、議事録、統計、契約資料、補助金資料、公開発言資料などに付与する一意の識別子である。

`source_id` は、資料の存在を示すだけではなく、次の情報を追跡するために使用する。

- どの案件に関係する資料か。
- 誰が公開又は作成した資料か。
- どこで取得又は保存した資料か。
- いつ取得した資料か。
- どの年度、期間、制度、事業を対象とする資料か。
- 原文本文を確認したか。
- 該当ページ、該当箇所、引用箇所を確認したか。
- 金額、件数、割合、年度、単位を確認したか。
- 更新、削除、リンク切れの可能性があるか。

`source_id` が付与されていても、本文確認、ページ確認、引用確認、数値確認が未了であれば、正式な根拠として使える範囲は限定される。

## 3. source_idを付与すべき資料

次の資料には、原則として `source_id` を付与する。

- 自治体、議会、省庁、公的機関が公開する資料
- 予算書、決算書、決算カード、事務事業評価、財政資料
- 議事録、会議録、委員会記録、答弁資料
- 補助金、委託、契約、指定管理、入札に関する資料
- 情報公開請求で取得した資料
- 統計表、CSV、Excel、データベース出力
- 公式Webページ、公告、告示、要綱、要領、規則、条例
- 報道、団体資料、専門家記事など、補助根拠として参照する資料
- 外部発信、分析、カード、台帳、Runbook、SOPの根拠になる資料
- `gap_id` / `warning_id` / `stop_id` の発生根拠になる資料

同じ名称の資料でも、年度、版、公開日、URL、本文差分が異なる場合は、別の資料として扱うか、台帳上で差異を明記する。

## 4. source_idを付与しないもの

次のものには、原則として `source_id` を付与しない。

- 作業者の未整理メモ
- 根拠として使わない一時的な検索語句
- AI回答そのもの
- 出典不明の伝聞
- 検証不能なSNS断片
- 個人情報、secret、APIキー、パスワード、トークンを含む非公開情報
- GitHubへ追加しないPDF原本そのもの
- 黒塗り前資料そのもの
- 単なる作業ログ、コマンド履歴、内部チェックリスト

ただし、上記に該当する情報が後から根拠資料の候補になる場合は、正式な公開主体、URL又は保存場所、取得日、本文確認状態を確認したうえで、別途 `source_id` 候補として登録する。

## 5. 原資料・要約資料・加工資料の区別

根拠資料は、次の3種類を混同しない。

| 区分 | 意味 | 使用上の注意 |
| --- | --- | --- |
| 原資料 | 公開主体又は作成主体が直接作成・公開した一次資料。 | 最優先で確認する。数値、引用、年度、制度根拠の正式根拠候補になる。 |
| 要約資料 | 原資料の内容を短く整理した公式概要、説明資料、議会だより、報道、解説資料。 | 原資料の代替として断定しない。要約表現と原文表現を区別する。 |
| 加工資料 | 後方支援軍又は第三者が集計、転記、結合、注釈、黒塗り、図表化した資料。 | 加工前の原資料 `source_id` を必ず接続し、加工内容を備考に残す。 |

原資料が確認できる場合は、要約資料や加工資料だけで重要な断定を行わない。原資料が未取得の場合は、`gap_id` 又は `warning_id` を接続し、確認未了であることを明示する。

## 6. 本文確認状態の分類

本文確認状態は、次の分類で記録する。

| 本文確認状態 | 意味 | 使用可否の目安 |
| --- | --- | --- |
| 未確認 | 資料候補を認識しただけで、URL、本文、保存状態を確認していない。 | 正式根拠として使用しない。 |
| URL確認済み | URL、公開主体、資料名の存在を確認した。 | 本文・数値・引用の根拠にはしない。 |
| PDF取得済み | PDF又はファイルを取得又は保存した。 | 取得済みであっても本文未確認なら断定しない。 |
| 本文確認済み | 本文を開き、対象内容が存在することを確認した。 | 概要根拠として使用可能。ページ確認が必要な場合は次工程へ進む。 |
| ページ確認済み | 該当ページ、表、項目、発言箇所を特定した。 | 引用、数値、カード化の前提にできる。 |
| 引用確認済み | 引用文と原文の一致、引用範囲、改変有無を確認した。 | 直接引用に使用可能。 |
| 数値確認済み | 金額、件数、割合、単位、年度、算式を確認した。 | 数値を成果物へ反映可能。 |
| 更新確認必要 | 改訂、差替、最新版、リンク変更の可能性がある。 | 公開前に再確認する。 |
| 削除・リンク切れ | 資料が削除、非公開化、移転、リンク切れになっている。 | 新規根拠化を停止し、代替資料又は保存場所を確認する。 |

複数の状態に該当する場合は、成果物で必要な最も厳しい確認状態を満たしているか確認する。

## 7. ページ・箇所・引用確認の手順

ページ、箇所、引用を確認する場合は、次の順序で実施する。

1. `source_id`、資料名、公開主体、URL又は保存場所を確認する。
2. 資料が原資料、要約資料、加工資料のどれかを分類する。
3. 対象とする事実、数値、発言、制度、年度を明確にする。
4. PDF、HTML、CSV、Excelなどの本文を開く。
5. 該当ページ、見出し、表番号、項目名、発言者、日付を特定する。
6. 引用する場合は、原文の文字列、句読点、省略箇所、改行、注記を確認する。
7. 引用文と要約文を分けて記録する。
8. 台帳の「ページまたは該当箇所」と「引用可否」を更新する。
9. 引用範囲に個人情報、secret、黒塗り前情報が含まれないか確認する。

引用に自信がない場合は、直接引用を避け、要約として扱う。要約として扱う場合も、要約の根拠になるページ又は該当箇所を記録する。

## 8. 数値確認の手順

金額、件数、割合、年度、単位、順位、増減率などを扱う場合は、次を確認する。

1. 数値の出所となる `source_id` を確認する。
2. 対象年度、会計年度、集計期間、基準日を確認する。
3. 単位が円、千円、百万円、件、人、％などのどれかを確認する。
4. 税込・税抜、一般会計・特別会計、当初予算・補正予算・決算を区別する。
5. 表、列、行、脚注、注釈、注記を確認する。
6. 転記した数値と原資料の数値を照合する。
7. 計算値を使う場合は、分子、分母、算式、丸め処理を記録する。
8. 複数資料の数値が異なる場合は、差異を備考に残し、必要に応じて `gap_id` を作成する。
9. 数値が外部発信又は法的評価に影響する場合は、公開前チェックで再確認する。

数値確認が完了していない数値は、断定表現に使わない。「未確認」「暫定」「要再確認」などの状態を明示する。

## 9. URL・取得日・公開主体の記録

`source_id` 台帳には、少なくとも次を記録する。

- URL又は保存場所
- 取得日
- 公開主体
- 公開日又は作成日
- 対象年度
- 資料名
- 資料種別

取得日は、作業者が実際に資料へアクセス、取得、保存、確認した日を `YYYY-MM-DD` 形式で記録する。公開主体は、資料を作成又は公開した自治体、議会、省庁、団体、法人などを記録し、単なる掲載サイト名と混同しない。

URLが長い場合や検索結果ページしかない場合でも、可能な限り資料本体のURLを記録する。保存場所を記録する場合は、GitHubへPDF原本、個人情報、secretを追加しない。

## 10. 資料が更新・削除された場合の扱い

資料が更新、差替、削除、移転、リンク切れになった場合は、次のように扱う。

- 既存の `source_id` を無断で上書きしない。
- 版、公開日、取得日、URL、本文差分を確認する。
- 更新後資料を新しい根拠として使う場合は、新規 `source_id` 又は版差分を台帳で明示する。
- 旧資料が必要な場合は、保存場所、取得日、当時の確認状態を残す。
- 削除・リンク切れの場合は、本文確認状態を「削除・リンク切れ」にする。
- 最新版確認が必要な場合は、本文確認状態を「更新確認必要」にする。
- 重要な根拠が失われた場合は、`gap_id` 又は `stop_id` を作成する。

更新・削除の事実だけで資料内容を推定してはならない。更新後の本文を確認するまでは、新旧差分を断定しない。

## 11. source_id台帳の最小項目

`source_id` 台帳には、少なくとも次の項目を設ける。

| 項目 | 記録内容 |
| --- | --- |
| source_id | 資料を一意に識別するID。 |
| case_id | 接続する案件ID。複数案件で使う場合は併記する。 |
| 資料名 | 資料の正式名又は識別可能な名称。 |
| 公開主体 | 資料を作成又は公開した主体。 |
| URLまたは保存場所 | 資料本体のURL、取得元、又は保存場所。 |
| 取得日 | 資料へアクセス、取得、保存、確認した日。 |
| 公開日または作成日 | 資料側に示された公開日、作成日、告示日、会議日など。 |
| 対象年度 | 資料が対象とする年度、期間、基準日。 |
| 資料種別 | PDF、HTML、CSV、Excel、議事録、統計、契約資料など。 |
| 原資料 / 要約資料 / 加工資料 | 根拠資料の区分。 |
| 本文確認状態 | 未確認、URL確認済み、本文確認済み等の状態。 |
| ページまたは該当箇所 | ページ、表、項目、見出し、発言箇所、セル範囲など。 |
| 引用可否 | 直接引用可、要約のみ、引用不可、要確認など。 |
| 数値確認状態 | 未確認、照合済み、算式確認済み、差異あり、再確認必要など。 |
| 備考 | 差替可能性、注意点、関連ID、確認者、未確認事項など。 |

台帳の表を作成する場合は、見出し行、区切り行、列数を一致させ、Markdown表を壊さない。

## 12. gap_id / warning_id / stop_id との接続

`source_id` 管理中に未確認事項、注意事項、停止条件が見つかった場合は、次のIDへ接続する。

| 接続先 | 接続する状態 | 対応 |
| --- | --- | --- |
| `gap_id` | 根拠不足、数値差異、年度不明、資料欠落、原資料未取得がある。 | 未確認事項として記録し、追加確認又は情報公開請求へつなぐ。 |
| `warning_id` | 使用は可能だが、要約資料のみ、更新可能性あり、引用注意などの制約がある。 | 成果物と公開前チェックへ注意事項として引き渡す。 |
| `stop_id` | 本文未確認のまま断定が必要、重要資料がリンク切れ、secretや個人情報混入の疑いがある。 | 公開、PR、外部発信、法務官引渡を停止する。 |

`gap_id` / `warning_id` / `stop_id` を作成した場合は、対象の `source_id`、ページ又は該当箇所、確認状態、再開条件を併記する。

## 13. case_idとの接続

すべての正式な `source_id` は、可能な限り `case_id` と接続する。

`case_id` との接続では、次を確認する。

- その資料がどの案件の根拠になるか。
- 複数案件にまたがる資料か。
- 案件の対象年度、自治体、制度、事業と資料の対象が一致するか。
- 案件本文、台帳、カード、公開前チェックで同じ `source_id` を参照しているか。
- 案件終了後も、取得日、確認状態、備考から再確認できるか。

案件に関係しない資料を正式根拠として大量登録しない。案件候補段階の資料は、備考に候補であることを明記する。

## 14. 公開前チェックへの引き渡し

公開前チェックへ進む前に、`source_id` 管理担当は次を引き渡す。

- 使用した `source_id` 一覧
- 各資料の原資料 / 要約資料 / 加工資料の区分
- 本文確認状態
- ページ又は該当箇所
- 引用可否
- 数値確認状態
- 取得日、公開主体、URL又は保存場所
- `gap_id` / `warning_id` / `stop_id` の有無
- 更新確認が必要な資料の有無
- 削除・リンク切れ資料の有無

公開前チェックでは、本文未確認、ページ未確認、引用未確認、数値未確認を隠してはならない。確認未了のまま公開する判断が必要な場合は、ユーザー判断又は停止条件として扱う。

## 15. 禁止事項

`source_id` 管理では、次を禁止する。

- 本文未確認の資料を「確認済み」と記録する。
- URLを見ただけで本文、ページ、引用、数値を確認済みにする。
- PDFを保存しただけで本文確認済みにする。
- 要約資料を原資料として扱う。
- 加工資料の加工前根拠を記録しない。
- 旧版と新版を同じ根拠として無断で混同する。
- 引用文を原文と異なる形で直接引用として扱う。
- 数値の単位、年度、脚注、算式を確認せずに断定する。
- PDF原本、画像、個人情報、secret、APIキーをGitHubへ追加する。
- README.md、archive、既存ファイルを目的外に変更する。
- `gap_id` / `warning_id` / `stop_id` を隠して成果物を公開する。

## 16. PR前確認条件

このRunbook又は `source_id` 関連成果物をPRに出す前に、次を確認する。

- 変更ファイルが予定どおりである。
- 新規Markdownファイルの場合、差分が追加のみである。
- README.mdを変更していない。
- archiveを変更していない。
- PDF、画像、個人情報、secret、APIキーを追加していない。
- 章番号が重複していない。
- Markdown表の列数が一致している。
- 既存Runbook本文をコピーして混入させていない。
- `source_id` 台帳の最小項目が不足していない。
- 本文確認状態の分類が省略されていない。
- 公開前チェックへ引き渡す未確認事項、注意事項、停止条件が明記されている。
