# Runbook: 公開前チェック実行 v0.1

## 1. Runbookの位置づけ

公開前チェック実行Runbookは、後方支援軍の成果物をGitHub、PDF、外部発信、情報公開請求、法務官引渡へ進める前の最終確認手順として扱う。

公開前チェックでは、内容の正しさだけでなく、資料区分、個人情報、secret、黒塗り、ID接続、停止条件、ユーザー判断を確認する。

このRunbookは、次の事故や手戻りを防ぐための確認手順である。

- `source_id`なしの事実確定を防ぐ。
- `gap_id`又は`warning_id`を隠した外部発信を防ぐ。
- `stop_id`がある案件の公開を防ぐ。
- `decision_id`なしの外部発信、評価断定、法務官引渡を防ぐ。
- GitHub上の壊れた差分、章番号重複、表崩れ、旧本文混入、誤ファイル変更を防ぐ。

## 2. 対象

このRunbookは、次の成果物を対象にする。

- Runbook
- SOP
- 台帳仕様
- 案件台帳
- `source_id`台帳
- `gap_id`台帳
- `warning_id` / `stop_id`仕様
- `decision_id`台帳
- 情報公開請求文案
- 議会発言カード
- 決算カード
- 補助金・委託契約カード
- 外部発信用文章
- PR本文
- GitHub差分
- PDF化資料
- 黒塗り資料
- 法務官引渡資料

## 3. 使用する主なID

| ID | 用途 | 公開前チェックでの確認観点 |
| --- | --- | --- |
| `case_id` | 案件を識別するID | 対象案件と成果物が接続されているか確認する。 |
| `source_id` | 根拠資料を識別するID | 事実、金額、年度、引用に根拠資料が接続されているか確認する。 |
| `quote_id` | 引用箇所を識別するID | 引用文と要約文が混同されていないか確認する。 |
| `actor_id` | 関係主体を識別するID | 団体名、部署名、人物属性を誤って扱っていないか確認する。 |
| `policy_id` | 制度、施策、方針を識別するID | 対象制度や施策を取り違えていないか確認する。 |
| `gap_id` | 未確認事項を識別するID | 未確認事項を確認済みとして扱っていないか確認する。 |
| `warning_id` | 注意事項を識別するID | 読者や利用者へ明示すべき注意点を隠していないか確認する。 |
| `stop_id` | 停止条件を識別するID | 公開、PR、マージ、外部発信を止めるべき条件が残っていないか確認する。 |
| `request_id` | 情報公開請求を識別するID | 請求対象、請求先、解消する未確認事項が接続されているか確認する。 |
| `decision_id` | ユーザー判断を識別するID | 外部発信、評価断定、提出、引渡の判断が取得済みか確認する。 |
| `change_id` | 変更内容を識別するID | 差分、変更理由、影響範囲を確認できるか確認する。 |
| `incident_id` | 事故、誤り、混入を識別するID | 誤引用、誤金額、secret混入などの事故候補を記録する。 |

## 4. チェック段階

公開前チェックは、次の段階に分けて実施する。

1. 作成前チェック
2. 作成後チェック
3. GitHub反映前チェック
4. PR作成前チェック
5. マージ前チェック
6. PDF化前チェック
7. 外部発信前チェック
8. 情報公開請求提出前チェック
9. 法務官引渡前チェック

## 5. 作成前チェック

作成前には、少なくとも次を確認する。

- 対象案件の`case_id`を確認する。
- 使用資料の`source_id`候補を確認する。
- 作成する成果物の種類を確認する。
- 公開予定の有無を確認する。
- 個人情報を扱う可能性を確認する。
- PDF原本、黒塗り前資料、secretを扱う可能性を確認する。
- ユーザー判断が必要な箇所を確認する。

## 6. 作成後チェック

作成後には、少なくとも次を確認する。

- 事実、推定、評価、提案が分離されているか。
- 金額、年度、単位、分母、分子が明記されているか。
- 引用文と要約文が混同されていないか。
- 原文確認済みと要約確認済みが区別されているか。
- `source_id`が不足していないか。
- `gap_id`が必要な未確認事項を放置していないか。
- `warning_id`が必要な注意事項を放置していないか。
- `stop_id`が必要な停止条件を見逃していないか。

## 7. GitHub反映前チェック

GitHubへ反映する前には、少なくとも次を確認する。

- PDF原本を追加していないか。
- 個人情報を追加していないか。
- 黒塗り前資料を追加していないか。
- APIキー、パスワード、トークン、secretを追加していないか。
- archive配下を不用意に変更していないか。
- 既存ファイルを誤って削除していないか。
- 新規ファイル追加なのか、既存ファイル修正なのかを確認したか。
- 既存ファイル修正の場合、全文新規追加扱いになっていないか。
- ファイル名、パス、拡張子が正しいか。

## 8. PR作成前チェック

PR作成前には、少なくとも次を確認する。

- 変更ファイル数を確認する。
- 目的外ファイルが含まれていないか。
- 差分が過大ではないか。
- 章番号が重複していないか。
- 同じ見出しが二重に出ていないか。
- 表が壊れていないか。
- `| --- | --- |`だけの壊れた行が残っていないか。
- 旧本文と新本文が混在していないか。
- 出力テンプレートが二重化していないか。
- 禁止事項が二重化していないか。
- PR条件が二重化していないか。
- mainへ直接コミットしようとしていないか。

## 9. マージ前チェック

マージ前には、少なくとも次を確認する。

- 変更ファイル数が想定どおりか。
- 追加削除行数が妥当か。
- コンフリクトがないか。
- GitHub上でマージ準備完了になっているか。
- 「SQUASHしてマージ」を選ぶべきか確認したか。
- 自動マージを有効化していないか。
- main直コミットではないか。
- レビューすべき差分を最後まで見たか。
- 壊れた表、章番号重複、旧本文混入がないか。
- PDF原本、個人情報、黒塗り前資料、secretが含まれていないか。

## 10. PDF化前チェック

PDF化前には、少なくとも次を確認する。

- 公開用PDFか内部用PDFかを明確にする。
- 黒塗りが必要な情報を確認する。
- 黒塗り前資料をGitHubへ入れない。
- 個人情報、住所、電話番号、メールアドレス、口座情報、印影、署名を確認する。
- 原本PDFと加工済みPDFを混同しない。
- 公開用PDFには`gap_id`、`warning_id`、`stop_id`の扱いを明示する。

## 11. 外部発信前チェック

外部発信前には、少なくとも次を確認する。

- `source_id`があるか。
- `gap_id`を隠していないか。
- `warning_id`を反映しているか。
- `stop_id`が残っていないか。
- 評価断定に必要な`decision_id`を取得したか。
- 違法、不正、癒着、汚職、虚偽、隠蔽、改ざんと断定していないか。
- 事実、推定、評価、意見、提案を分けているか。
- 引用元と取得日を明記しているか。
- 公開対象者、公開範囲、公開手段を確認したか。

## 12. 情報公開請求提出前チェック

情報公開請求提出前には、少なくとも次を確認する。

- 請求対象文書名が具体化されているか。
- 請求先を推定で断定していないか。
- 公開資料で確認できる範囲を先に確認したか。
- `request_id`を作成したか。
- 解消する`gap_id`を明示したか。
- 提出判断の`decision_id`を取得したか。
- 請求文案に不要な評価断定、攻撃的表現、政治的断定が入っていないか。

## 13. 法務官引渡前チェック

法務官引渡前には、少なくとも次を確認する。

- 法的評価が必要な理由を明示したか。
- 事実と評価を分けたか。
- 原資料の`source_id`を接続したか。
- 未確認事項を`gap_id`で明示したか。
- 注意事項を`warning_id`で明示したか。
- `stop_id`がある場合は引渡条件を明記したか。
- 法務官引渡の`decision_id`を取得したか。

## 14. stop_id判定

| 判定対象 | `stop_id`候補になる状態 | 再開条件の例 |
| --- | --- | --- |
| `source_id`なし | 事実、金額、年度、引用に根拠資料が接続されていない。 | 根拠資料を確認し、`source_id`を接続する。 |
| 本文未確認 | 原文、最新版、差分を確認していない。 | 本文確認の範囲と確認者を記録する。 |
| 金額未確認 | 金額、単位、年度、税込・税抜、合計の根拠が不明である。 | 原資料で金額を確認し、必要に応じて`gap_id`を作成する。 |
| 章番号重複 | 同一階層で同じ章番号が複数存在する。 | 章番号を修正し、目次や参照先を確認する。 |
| 表崩れ | Markdown表、PDF表、台帳表が読めない状態である。 | 表記を修正し、プレビューで確認する。 |
| 旧本文混入 | 旧版の本文、テンプレート、禁止事項、PR条件が混在している。 | 旧本文を削除し、最新版だけが残っていることを確認する。 |
| PDF原本混入 | GitHubへ原本PDFを追加しようとしている。 | 原本PDFを除外し、必要な場合は扱いをユーザーへ確認する。 |
| 個人情報混入 | 氏名、住所、電話番号、メールアドレス、口座情報、印影、署名などが含まれる。 | 削除又は黒塗りの要否を確認し、公開範囲を判断する。 |
| 黒塗り前資料混入 | 黒塗り前の資料がGitHubや外部発信に含まれる。 | 黒塗り済み資料と取り違えていないか確認する。 |
| APIキー、secret混入 | APIキー、パスワード、トークン、secretが含まれる。 | 直ちに公開を停止し、削除、無効化、事故対応を検討する。 |
| 外部発信予定 | 公開対象、公開範囲、公開手段、判断者が未確定である。 | `decision_id`を取得し、発信条件を確認する。 |
| 法的断定が必要 | 違法、不正、癒着、汚職、虚偽、隠蔽、改ざんなどの断定が必要になる。 | 法的評価の必要性を整理し、ユーザー判断又は法務官引渡を確認する。 |
| ユーザー判断未取得 | 外部発信、提出、引渡、評価断定に必要な判断が未取得である。 | `decision_id`を取得するまで停止する。 |

## 15. incident_id判定

次の事象が判明した場合は、`incident_id`候補として記録し、影響範囲、再発防止、ユーザー報告の要否を確認する。

- 誤引用
- 誤金額
- 誤年度
- 誤単位
- 誤団体名
- 誤ファイル追加
- 誤マージ
- secret混入
- 個人情報混入
- 黒塗り漏れ
- 未確認事項を確認済みとして扱った場合
- `stop_id`を無視して外部発信した場合

## 17. 禁止事項

次の行為は行わない。

- `source_id`なしに事実を確定しない。
- `gap_id`を隠して外部発信しない。
- `warning_id`を無視して評価しない。
- `stop_id`がある状態で公開しない。
- `decision_id`なしに外部発信、情報公開請求提出、法務官引渡、評価断定へ進まない。
- PDF原本をGitHubに追加しない。
- 個人情報をGitHubに追加しない。
- 黒塗り前資料をGitHubに追加しない。
- APIキー、パスワード、トークン、secretをGitHubに追加しない。
- mainへ直接コミットしない。
- 自動マージを有効化しない。
- 壊れた表、章番号重複、旧本文混入を残したままPRを作成しない。
- 壊れた差分をマージしない。

## 18. PR条件

このRunbookに基づくPRでは、少なくとも次を条件として確認する。

- 新規ファイル追加のみ。
- 既存ファイルを変更しない。
- 既存ファイルを削除しない。
- archive配下を変更しない。
- PDF原本を追加しない。
- 個人情報を追加しない。
- 黒塗り前資料を追加しない。
- APIキー、パスワード、トークン、secretを追加しない。
- mainへ直接コミットしない。
- PRでは自動マージを行わない。
- PR作成後は、変更ファイル数、追加削除行数、コンフリクト有無を確認する。
- マージ前に、壊れた表、章番号重複、旧本文混入がないことを確認する。
