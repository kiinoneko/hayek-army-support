# Runbook: 案件起票・初動分類 v0.1

## 1. Runbookの位置づけ

案件起票・初動分類Runbookは、後方支援軍が新規案件を扱う前の入口手順である。

- 新規案件では、最初に `case_id` を作成又は確認する。
- 案件区分、対象範囲、使用資料、必要ID、停止条件、ユーザー判断事項を整理する。
- いきなり評価断定、外部発信、情報公開請求、法務官引渡へ進まない。
- `source_id` なしに事実確定しない。
- `gap_id`、`warning_id`、`stop_id` が必要な場合は初動で明示する。
- `decision_id` なしに提出、公開、マージ、法務官引渡へ進まない。

## 2. 対象

このRunbookは、以下の案件を対象に含める。

- 自治体政策案件
- 議会発言確認案件
- 議会だより・要約資料照合案件
- 決算カード作成案件
- 補助金・委託契約確認案件
- 社会福祉協議会・外郭団体確認案件
- 情報公開請求作成案件
- 公開前チェック案件
- GitHub PR作成・マージ案件
- PDF化案件
- 外部発信案件
- 法務官引渡案件
- 事故対応案件
- 台帳整備案件

## 3. 使用する主なID

| ID | 用途 |
| --- | --- |
| `case_id` | 案件全体を管理する |
| `source_id` | 根拠資料を管理する |
| `quote_id` | 発言・引用を管理する |
| `actor_id` | 関係主体を管理する |
| `policy_id` | 政策・事業・制度を管理する |
| `gap_id` | 未確認事項を管理する |
| `warning_id` | 注意事項を管理する |
| `stop_id` | 停止条件を管理する |
| `request_id` | 情報公開請求候補を管理する |
| `decision_id` | ユーザー判断を管理する |
| `change_id` | 変更履歴を管理する |
| `incident_id` | 誤り、事故、混入を管理する |

## 4. 初動原則

- まず案件を分類する。
- 次に対象範囲を定義する。
- その後、使用資料と不足資料を整理する。
- 事実、推定、評価、提案を分ける。
- 公開資料で確認できないことを推定で埋めない。
- 金額、発言、契約、成果物、責任線を混同しない。
- 不足資料がある場合は `gap_id` にする。
- 注意事項がある場合は `warning_id` にする。
- 断定停止が必要な場合は `stop_id` にする。
- ユーザー判断が必要な場合は `decision_id` 候補にする。

## 5. 案件区分

| 案件区分 | 内容 | 初動対応 |
| --- | --- | --- |
| 軽量案件 | 文章整備、軽微なRunbook追加、単純な整理 | 通常確認のみ |
| 通常案件 | 公金、議会発言、政策、団体、契約、資料照合を扱う | `source_id`、`gap_id` を確認 |
| 重要案件 | 外部発信、情報公開請求、法的評価、人物・団体評価を含む | `stop_id`、`decision_id` 確認 |
| 停止案件 | secret、個人情報、黒塗り前資料、法的断定、未確認重大事項を含む | 作業停止、ユーザー判断 |
| 事故案件 | 誤公開、誤引用、誤金額、secret混入等 | `incident_id` 候補化 |

## 6. 作成前チェック

作成前に、以下を確認する。

- 案件名
- 案件目的
- 対象自治体・対象団体・対象人物
- 対象年度
- 使用資料
- 公開予定の有無
- 情報公開請求予定の有無
- 法的評価の要否
- PDF原本、個人情報、黒塗り前資料、secretの有無
- GitHub反映の有無
- ユーザー判断が必要な箇所

## 7. `case_id` 作成基準

- 新規に独立した作業単位であれば `case_id` を作成する。
- 既存案件の一部であれば既存 `case_id` に接続する。
- 同一資料の確認でも目的が違う場合は別案件にできる。
- 外部発信、情報公開請求、法務官引渡を伴う場合は必ず `case_id` を明示する。
- `case_id` なしに案件台帳へ正式登録しない。

## 8. `source_id` 初動確認

- 使用資料候補を列挙する。
- 資料名、年度、公開元、URL、取得日、確認状態を記録する。
- 本文確認済みと存在確認のみを分ける。
- PDF本文未確認、画像表未確認、ページ不明は `gap_id` 又は `warning_id` とする。
- `source_id` なしに事実、金額、引用を確定しない。

## 9. `actor_id` / `policy_id` 初動確認

- 人物、部署、団体、委託先、補助対象団体、指定管理者が出る場合は `actor_id` 候補を確認する。
- 政策、事業、制度、施設、議案、補助金、委託契約が出る場合は `policy_id` 候補を確認する。
- `actor_id` だけで責任や違法性を断定しない。
- `policy_id` だけで政策失敗や不正を断定しない。

## 10. `gap_id` / `warning_id` / `stop_id` 初動判定

| 状態 | 発行ID | 初動対応 |
| --- | --- | --- |
| 根拠資料未確認 | `GAP_` | 資料確認 |
| 本文未確認 | `STOP_` 又は `GAP_` | 本文確認まで断定停止 |
| 金額未確認 | `GAP_` | 金額資料確認 |
| 発言原文未確認 | `GAP_` 又は `STOP_` | 原文確認 |
| 要約資料のみ | `WARN_` | 原文照合 |
| 契約書未確認 | `GAP_` 又は `STOP_` | 契約条件断定停止 |
| 成果物未確認 | `GAP_` 又は `STOP_` | 履行済み断定停止 |
| 個人情報あり | `STOP_` | 公開停止 |
| 黒塗り前資料あり | `STOP_` | GitHub追加禁止 |
| secretあり | `STOP_` | 作業停止、事故対応 |
| 外部発信予定 | `DEC_` | ユーザー判断 |
| 法的断定が必要 | `STOP_` | 法務官確認又はユーザー判断 |

## 11. `request_id` 候補化基準

- 公開資料で確認できない重要資料は `request_id` 候補とする。
- いきなり情報公開請求へ進まない。
- 先に公開資料、公式サイト、議会資料、国・県・市町村資料、団体資料を確認する。
- 請求文案作成後も提出判断には `decision_id` が必要。
- 請求対象文書名を具体化する。
- 請求先を推定で断定しない。

## 12. `decision_id` 候補化基準

以下の場合は、`decision_id` 候補にする。

- 外部発信する場合
- 情報公開請求を提出する場合
- 法務官へ引き渡す場合
- 評価断定する場合
- GitHubでマージする場合
- PDF公開する場合
- 不確実性が残る案件を公開する場合
- 重要案件又は停止案件を再開する場合

## 14. 禁止事項

- `case_id` なしに正式案件化しない。
- `source_id` なしに事実、金額、発言を確定しない。
- 本文未確認の資料を確認済みとして扱わない。
- 要約資料だけで原文発言を確定しない。
- 金額だけで不正、癒着、無駄、利益誘導と断定しない。
- `gap_id` を隠して外部発信しない。
- `warning_id` を無視して評価しない。
- `stop_id` がある状態で公開しない。
- `decision_id` なしに外部発信、情報公開請求提出、法務官引渡、評価断定へ進まない。
- PDF原本、個人情報、黒塗り前資料、APIキー、secretをGitHubに追加しない。
- mainへ直接コミットしない。
- 自動マージを有効化しない。

## 15. PR条件

- 新規ファイル追加のみ。
- 既存ファイルを変更しない。
- 既存ファイルを削除しない。
- archive配下を変更しない。
- PDF原本を追加しない。
- 個人情報を追加しない。
- 黒塗り前資料を追加しない。
- APIキー、パスワード、トークン、secretを追加しない。
- mainへ直接コミットしない。
- PRでは自動マージを行わない。
- PR作成後は、変更ファイル数、追加削除行数、コンフリクト有無を確認する。
