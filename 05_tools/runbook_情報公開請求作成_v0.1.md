# 情報公開請求作成Runbook v0.1

## 1. Runbookの位置づけ

情報公開請求作成Runbookは、未確認事項を請求可能な文書名へ変換するための作業手順書である。

- 情報公開請求は情報収集の開始点ではなく、公開資料確認後の補助手段である。
- 先に公式サイト、議会資料、予算書、決算書、主要施策成果報告書、WAM、公開済み資料を確認する。
- `gap_id` があるだけで直ちに請求してはならない。
- 請求文案作成と提出判断を分離する。
- 提出判断は必ずUSER_DECISIONとし、必要に応じて `decision_id` を発行する。
- AIは文案を作成できるが、提出を代行したり、提出済みにしたりしてはならない。

## 2. 対象

本Runbookは、次の作業を対象とする。

- 情報公開請求
- 公文書開示請求
- 自治体への照会
- 議会事務局への確認依頼
- 補助金交付資料の確認
- 委託契約資料の確認
- 指定管理資料の確認
- 事業実績報告書の確認
- 成果物・報告書の確認
- 議事録・会議録・録画記録の確認
- 社会福祉協議会、外郭団体、補助金交付先、委託先に関する自治体保有資料の確認

## 3. 使用する主なID

| ID | 用途 |
|---|---|
| `case_id` | 請求作成案件を管理する |
| `source_id` | 既存公開資料、根拠資料を管理する |
| `gap_id` | 未確認事項、資料不足を管理する |
| `request_id` | 情報公開請求、照会、確認依頼を管理する |
| `decision_id` | 提出判断、文案確定判断を管理する |
| `warning_id` | 請求範囲、対象年度、文書名の注意を管理する |
| `stop_id` | 自動提出、根拠不足、公開リスクを停止する |
| `change_id` | 請求文案の変更履歴を管理する |
| `incident_id` | 誤提出、誤請求、個人情報混入等を管理する |

## 4. 請求前の基本原則

- 先に公開資料を確認する。
- 既存公開資料で確認できる事項を重複請求しない。
- 請求対象は「知りたいこと」ではなく「行政機関が保有する文書名」に変換する。
- 目的、対象年度、対象団体、対象事業、必要文書を明確にする。
- 請求範囲を過度に広げない。
- 個人情報、第三者情報、非公開情報の混入に注意する。
- 請求文案は下書きと提出用最終案を区別する。
- 提出には `decision_id` を必要とする。
- 情報公開請求を政治的断定や不正断定の代替にしない。

## 5. 公開資料確認手順

| 手順 | 内容 |
|---|---|
| 1 | 自治体公式サイトを確認する |
| 2 | 議会資料、議事録、会議録、委員会資料を確認する |
| 3 | 予算書、決算書、主要施策成果報告書を確認する |
| 4 | 補助金一覧、契約一覧、指定管理資料を確認する |
| 5 | 社協案件ではWAM現況報告書、計算書類、事業報告書を確認する |
| 6 | 既存資料を`source_id`化する |
| 7 | 残った不足事項を`gap_id`化する |
| 8 | `gap_id`を請求文書候補へ変換する |

## 6. gap_idからrequest_idへの変換基準

| gap内容 | request_id化の判断 | 請求対象文書候補 |
|---|---|---|
| 補助金額が不明 | `request_id`候補 | 補助金交付決定通知、交付申請書、実績報告書、精算書 |
| 委託契約額が不明 | `request_id`候補 | 委託契約書、仕様書、設計書、支払実績、検査調書 |
| 成果物が不明 | `request_id`候補 | 成果報告書、事業報告書、納品物、検査記録 |
| 職員数・FTEが不明 | `request_id`候補 | 職員配置資料、事業報告、現況報告 |
| 議員・首長発言の原文が不明 | `request_id`候補 | 議事録、会議録、録画記録、答弁書 |
| 予算と決算の差額理由が不明 | `request_id`候補 | 決算説明資料、事業別決算資料、主要施策成果報告書 |
| 事業目的・KPIが不明 | `request_id`候補 | 事業計画書、評価書、実施要綱、仕様書 |
| 法令・要綱根拠が不明 | `request_id`候補 | 条例、規則、要綱、交付要綱、決裁文書 |

- `gap_id` は未確認事項であり、根拠ではない。
- `request_id` は外部アクション候補であり、提出済みを意味しない。
- `gap_id` から `request_id` へ進む場合、必要に応じてUSER_DECISIONを求める。

## 7. 請求対象文書の特定手順

| 手順 | 内容 |
|---|---|
| 1 | 知りたい事項を整理する |
| 2 | 対象年度を決める |
| 3 | 対象団体・対象事業を決める |
| 4 | 行政が保有している可能性が高い文書名へ変換する |
| 5 | 文書名を具体化する |
| 6 | 請求範囲を絞る |
| 7 | 個人情報や第三者情報を必要以上に求めない |
| 8 | 請求文案に「対象年度」「対象団体」「対象事業」を明記する |

## 8. 請求先確認手順

| 確認項目 | 内容 |
|---|---|
| 自治体名 | 請求先自治体を確認する |
| 担当部署 | 文書を保有している可能性が高い部署を確認する |
| 議会事務局 | 議事録・発言・議会資料の場合に確認する |
| 教育委員会 | 教育関係資料の場合に確認する |
| 福祉担当部署 | 社協、福祉、補助金、委託契約の場合に確認する |
| 契約担当部署 | 委託契約、入札、仕様書の場合に確認する |
| 情報公開窓口 | 提出先制度窓口を確認する |

- 請求先が不明な場合は照会文案を作成する。
- 請求先を推定で断定しない。
- 公式サイトで提出方法、様式、手数料、受付方法を確認する。
- 提出方法の判断はUSER_DECISION対象とする。

## 9. 請求範囲の設計

| 設計項目 | 注意 |
|---|---|
| 対象年度 | 例：令和6年度、2024年度等 |
| 対象期間 | 契約日、交付決定日、事業実施期間等 |
| 対象団体 | 社協、外郭団体、委託先等 |
| 対象事業 | 事業名、補助金名、委託名等 |
| 文書種類 | 契約書、仕様書、実績報告書等 |
| 除外事項 | 個人情報、口座番号、印影等を必要に応じ除外 |
| 優先順位 | 文書量が多い場合は優先文書を明記 |

| 3 | 不足している文書を列挙する |
| 4 | `request_id`を仮発行する |
| 5 | 請求対象文書を文書名として書く |
| 6 | 対象年度、対象団体、対象事業を明記する |
| 7 | 個人情報を不要に求めない |
| 8 | 請求文案を下書きとして作成する |
| 9 | 提出用最終案にする前にUSER_DECISIONを求める |
| 10 | 提出前チェックを行う |

## 12. decision_idが必要な場面

- 情報公開請求を提出する。
- 請求文案を提出用最終案にする。
- 請求先を確定する。
- 請求範囲を大きく広げる。
- 個人情報や第三者情報に関係し得る文書を求める。
- 外部照会又は確認依頼を送る。
- 不開示決定、不存在回答、一部開示への対応方針を決める。
- 再請求、補正、異議申立て等の方針を検討する。
- 受領資料をGitHub又は外部資料へ反映する。
- 黒塗り後資料を公開可能と判断する。

## 14. 提出後管理

| 状態 | 処理 |
|---|---|
| 下書き | USER_DECISION前。提出不可 |
| 提出待ち | `decision_id`取得後、提出作業待ち |
| 提出済み | 提出日、受付番号、提出先を記録 |
| 回答待ち | 回答期限を記録 |
| 補正依頼あり | 補正内容を記録し、必要に応じて`decision_id` |
| 回答受領 | 回答資料を`source_id`化する |
| 一部開示 | 黒塗り、非開示部分、理由を記録 |
| 不開示 | 理由を記録し、次対応をUSER_DECISION |
| 不存在 | 不存在理由を記録し、追加請求可否を検討 |
| 終了 | `remaining_gaps`、`remaining_warnings`を残して終了 |

## 15. 回答受領後の処理

| 手順 | 内容 |
|---|---|
| 1 | 回答文書を受領した日を記録する |
| 2 | 回答資料を`source_id`化する |
| 3 | 黒塗り、個人情報、公開区分を確認する |
| 4 | `gap_id`が解消したか確認する |
| 5 | 解消しない場合は`gap_id`を継続する |
| 6 | 数値や発言を正式化する前に本文確認を行う |
| 7 | 必要に応じて決算カード、案件台帳、`policy_id`へ反映する |
| 8 | 外部公開前に公開前チェックリストを通す |
| 9 | 反映した変更は`change_id`へ記録する |

## 16. 不開示・不存在・一部開示時の処理

- 不開示、一部開示、不存在は失敗ではなく、行政側の回答事実として記録する。
- 回答理由、根拠条項、対象文書、対象年度を記録する。
- 不存在の場合、文書名が不適切だったのか、文書自体が作成されていないのかを分ける。
- 一部開示の場合、黒塗り箇所を推定で断定しない。
- 次に再請求、補正、別文書請求、照会を行う場合はUSER_DECISIONを必要とする。
- 回答文書はPDF原本としてGitHubに置かない。

## 17. 社協・外郭団体案件の注意

- 社協自体の資料と、自治体が保有する社協関連資料を分ける。
- WAMで確認できる計算書類と、自治体側の補助金・委託契約資料を分ける。
- 町補助金、町委託料、国県補助金、制度収入を混同しない。
- 社協の職員数、FTE、常勤換算は資料本文で確認する。
- 自治体が保有していない資料まで一括請求しない。
- 自治体が社協から受領した実績報告書、補助金申請書、委託契約書、仕様書、検査調書等を文書名候補とする。

## 18. 補助金・委託契約案件の注意

| 確認事項 | 文書候補 |
|---|---|
| 補助金の根拠 | 交付要綱、交付決定通知、申請書 |
| 補助金額 | 交付決定通知、精算書、実績報告書 |
| 委託契約額 | 委託契約書、契約一覧、支払実績 |
| 業務内容 | 仕様書、設計書、業務計画書 |
| 成果確認 | 完了報告書、検査調書、成果物 |
| 支払確認 | 支出命令書、支払調書 |
| 随意契約理由 | 随意契約理由書、契約審査資料 |

## 19. 発言・議事録案件の注意

- 発言を請求する場合は、議事録、会議録、録画記録、答弁書、発言通告書等を確認する。
- 発言原文がない状態で`quote_id`を正式化しない。
- 動画だけでなく議事録又は会議録の有無を確認する。
- 発言者、発言日、会議名、議題、前後文脈を確認する。
- 要約と原文を混同しない。
- 発言の評価や矛盾指摘は、原文確認後に行う。

## 21. 禁止事項

- 公開資料を確認せずに情報公開請求へ進まない。
- `gap_id`だけで請求文案を確定しない。
- `request_id`を提出済みの意味で使わない。
- `decision_id`なしに請求を提出済みにしない。
- AI判断だけで請求先、請求範囲、提出可否を確定しない。
- 請求文案の下書きと提出用最終案を混同しない。
- 個人情報、口座番号、印影等を不要に求めない。
- PDF原本、黒塗り前資料、個人情報をGitHubに追加しない。
- 回答資料を本文確認せず`source_id`正式化しない。
- 不開示や不存在を行政の不正と断定しない。
- 発言原文未確認のまま`quote_id`を正式化しない。
- 法的断定を後方支援軍だけで行わない。
- 外部提出を自動化しない。
