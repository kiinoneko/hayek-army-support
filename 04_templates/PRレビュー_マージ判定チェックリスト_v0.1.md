# PRレビュー・マージ判定チェックリスト v0.1

## 目的

このチェックリストは、後方支援軍におけるGitHub Pull Request（PR）のレビュー・マージ判定を標準化するためのものです。

ドクトリン、SOP、RACI、Evals、ID仕様、案件台帳、テンプレート、公開資料化文書などをPRで追加・変更する際に使用します。

PRは後方支援軍における軍議であり、`main`ブランチは正式軍令です。`main`へ入れる前に、削除、コンフリクト、秘密情報、個人情報、黒塗り前資料、PDF原本、`archive`変更、余計な既存ファイル変更がないかを確認します。

---

## 1. PRレビューの位置づけ

- PRは変更提案であり、`main`へ入るまでは正式軍令ではない。
- `main`は正式運用版である。
- `branch`は草案である。
- Squash mergeは、複数の試行錯誤を1つの正式変更として記録するために使う。
- 後方支援軍の重要変更は原則としてPRで行う。
- `main`直コミットは原則禁止とする。
- マージ判断はユーザー判断を原則とする。

---

## 2. 基本判定

| 判定 | 意味 | 処理 |
|---|---|---|
| マージ可 | 条件を満たし、`main`へ入れてよい | Squash merge |
| 差し戻し | 修正が必要 | Codexへ修正指示 |
| 保留 | 内容確認又はユーザー判断が必要 | USER_DECISION待ち |
| マージ禁止 | コンフリクト、秘密情報、個人情報等の重大問題 | STOP |

---

## 3. マージ可条件

以下をすべて確認する。

- マージコンフリクトがない。
- 変更ファイルが意図したファイルのみである。
- 追加のみ又は妥当な差分である。
- 削除がない、又は削除理由が明確でUSER_DECISION済みである。
- `archive`配下を変更していない。
- PDF原本を追加していない。
- 個人情報を追加していない。
- 黒塗り前資料を追加していない。
- APIキー、パスワード、トークンを追加していない。
- 既存ファイルを不要に変更していない。
- ファイル名が意図通りである。
- 日本語運用文書として読める。
- ID接頭辞が既存仕様と整合している。
- 禁止事項が含まれている。
- PRタイトルが内容と一致している。

---

## 4. 差し戻し条件

以下のいずれかに該当する場合は、原則として差し戻す。

- マージコンフリクトがある。
- 意図しない既存ファイル変更がある。
- `archive`配下を変更している。
- 削除があるが理由が不明である。
- 追加ファイル名が誤っている。
- 内容が英語中心で、運用文書として読みにくい。
- ID接頭辞が既存仕様と不整合である。
- `source_`、`gap_`、`warning_`、`policy_`、`change_`など小文字接頭辞をサンプルIDとして使っている。
- `SRC_`、`GAP_`、`WARN_`、`STOP_`、`ACTOR_`、`QUOTE_`、`POLICY_`、`REQ_`、`DEC_`、`HANDOFF_`、`INC_`、`CHG_`と整合していない。
- 「引継ぎ」と「引渡」が混在している。
- PDF原本、個人情報、黒塗り前資料、APIキーを追加している。
- 禁止事項が抜けている。
- USER_DECISION事項をAIが自動進行する内容になっている。

---

## 5. 保留条件

以下のいずれかに該当する場合は、USER_DECISION又は追加確認まで保留する。

- 変更内容が大きく、ユーザー確認が必要である。
- 複数ファイルが変更されている。
- 既存仕様との関係が不明である。
- `v0.1`、`v0.2`、`v0.3`等の版管理が不明である。
- 既存ファイルを上書きするか、新規ファイルにするか判断が必要である。
- 公開区分が不明である。
- 法務官確認が必要である。
- 外部公開に影響する。
- `archive`変更を含む。
- 重要な仕様変更で`change_id`が必要である。

---

## 6. コンフリクト時の対応

- マージコンフリクトがあるPRはマージしない。
- `main`の最新状態を正とする。
- Codexにコンフリクト解消を指示する。
- 既存ファイルを不要に変更しない。
- 目的ファイルのみの変更へ整理する。
- 修正後、再度PR画面でマージ準備完了を確認する。

### 修正指示テンプレート

```text
PRにマージコンフリクトがあります。
mainの最新状態を正として、コンフリクトを解消してください。

方針：
- mainの既存ファイルを正としてください。
- 既存ファイルを不要に変更しないでください。
- archive配下を変更しないでください。
- PDF原本、個人情報、黒塗り前資料、APIキーを追加しないでください。
- mainへ直接コミットしないでください。
- このPRの目的ファイルのみを追加又は修正する形に整理してください。
- 自動マージはしないでください。
```

## 7. ファイル変更数チェック

| 表示 | 判定 |
|---|---|
| 1個のファイルが変更 | 原則確認しやすい |
| 2〜3個のファイルが変更 | 内容確認。意図通りなら可 |
| 4個以上のファイルが変更 | 原則保留又は差し戻し |
| 既存ファイル変更あり | 変更理由確認 |
| archive変更あり | 原則STOP |
| 削除あり | 原則保留又は差し戻し |

---

## 8. 追加・削除チェック

| 差分 | 判定 |
|---|---|
| `+のみ -0` | 新規追加なら原則良好 |
| `+あり -あり` | 既存変更。理由確認 |
| `-あり` | 削除理由確認。USER_DECISION対象 |
| 大量削除 | STOP |
| archive削除 | STOP |
| README等既存基幹ファイル変更 | 保留又は差し戻し |

## 10. archive配下チェック

- `archive`配下は旧版保存領域である。
- `archive`配下を自動変更しない。
- 旧版削除、上書き、移動はUSER_DECISION対象とする。
- `archive`変更が含まれるPRは原則STOP又は保留とする。

---

## 11. ID体系チェック

- `case_id`は`case_`系。
- `source_id`は`SRC_`系。
- `gap_id`は`GAP_`系。
- `warning_id`は`WARN_`系。
- `stop_id`は`STOP_`系。
- `actor_id`は`ACTOR_`系。
- `quote_id`は`QUOTE_`系。
- `policy_id`は`POLICY_`系。
- `request_id`は`REQ_`系。
- `decision_id`は`DEC_`系。
- `handoff_id`は`HANDOFF_`系。
- `incident_id`は`INC_`系。
- `change_id`は`CHG_`系。

補足：

- ID種別名は小文字でよい。
- 実際に発行するIDの接頭辞は既存仕様に合わせる。
- 個人名、住所、連絡先をID文字列に入れない。
- `actor_id`は個人攻撃や断定の根拠ではない。

---

## 12. 日本語・英語チェック

| 項目 | 判定 |
|---|---|
| ファイル本文が日本語中心か |  |
| PR説明文だけ英語か |  |
| ID名・コードだけ英語か |  |
| 英語本文が多く運用文書として読みにくいか |  |

- ID名、コード、PRタイトルの英語は許容する。
- PR説明文の英語は許容する。
- 運用文書本文が英語中心の場合は、ユーザー判断又は差し戻し対象とする。
- 日本語運用を優先する場合は、Codexに日本語化を指示する。

---

## 13. USER_DECISIONが必要な場面

以下はAI判断で自動進行せず、USER_DECISIONを待つ。

- PRをマージする。
- `archive`配下を変更する。
- 既存ファイルを削除する。
- 重要仕様を上書きする。
- 外部公開に関係する文書を追加する。
- 情報公開請求文案を提出用最終案にする。
- 法務官へ引き渡す。
- 中核軍へ引き渡す。
- 前線部隊へ引き渡す。
- STOPを解除する。
- インシデント対応を終了扱いにする。

---

## 14. Squash mergeルール

- 後方支援軍の通常PRは原則Squash mergeとする。
- Squash mergeは試行錯誤のコミットを1つの正式変更として`main`へ入れるために使う。
- マージコミットメッセージはPRタイトルに合わせる。
- コンフリクトがある場合はSquash mergeしない。
- 内容が保留又は差し戻しの場合はSquash mergeしない。

## 16. 禁止事項

- コンフリクトがあるPRをマージしない。
- `archive`配下をUSER_DECISIONなしに変更しない。
- PDF原本をGitHubに追加しない。
- 個人情報をGitHubに追加しない。
- 黒塗り前資料をGitHubに追加しない。
- APIキー、パスワード、トークンをGitHubに追加しない。
- 削除差分を確認せずマージしない。
- 既存ファイル変更を確認せずマージしない。
- STOP条件があるPRをマージしない。
- AI判断だけで重要PRをマージしない。
- PR説明文だけを見て本文確認を省略しない。
