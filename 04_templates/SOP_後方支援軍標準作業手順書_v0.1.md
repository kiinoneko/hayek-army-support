# SOP 後方支援軍標準作業手順書 v0.1

## 1. 目的

本手順書は、後方支援軍がユーザーから受領した案件を、再現性・追跡可能性・安全性を保ちながら整理し、調査・資料化・論点確認・判断保留まで進めるための標準作業手順を定める。

## 2. 適用範囲

本手順書は、後方支援軍が扱う資料探索、情報整理、比較検討、論点抽出、ユーザー判断支援に適用する。

ただし、以下の行為へは自動進行しない。

- 情報公開請求
- 法的断定
- 外部発信
- 政策評価断定

これらが必要となる可能性がある場合は、必ずユーザーへ確認し、最終判断はユーザーに留保する。

## 3. 基本原則

1. ユーザーの依頼内容を起点とし、勝手な目的変更を行わない。
2. 根拠資料と推論を分離し、後から検証できる形で記録する。
3. 不明点、矛盾、比較上の注意点を明示する。
4. STOP / WARNING / USER_DECISION に該当する事項は、作業を自動継続せずユーザー確認へ戻す。
5. 最終判断は常にユーザーに留保する。
6. 個人情報、PDF原本、既存ファイル、archive配下の内容は、明示的な許可なく変更しない。

## 4. 標準作業フロー

### 4.1 案件受付

1. ユーザーから案件を受け付ける。
2. 依頼文から以下を確認する。
   - 目的
   - 対象範囲
   - 希望成果物
   - 期限または優先度
   - 使用してよい資料範囲
   - 変更禁止対象
3. 不明点が成果物の品質または安全性に影響する場合は、作業前に確認する。

### 4.2 case_id作成

1. 案件ごとに一意の `case_id` を作成する。
2. 推奨形式は以下とする。

```text
CASE-YYYYMMDD-短い案件名-連番
```

3. `case_id` には、案件受付日、案件概要、連番を含める。
4. 以後の資料、メモ、カード、確認事項は `case_id` に紐づける。

### 4.3 案件区分判定

1. 案件を以下のいずれか、または複数に区分する。
   - 資料探索
   - 事実整理
   - 年表化
   - 論点整理
   - 比較検討
   - 手続確認
   - 文案作成
   - ユーザー判断支援
2. 区分に応じて、必要な資料粒度、確認手順、リスク確認を決める。
3. 法的断定、政策評価断定、外部発信、情報公開請求に近接する場合は、後述の STOP / WARNING / USER_DECISION 確認を必須とする。

## 5. 役割別作業手順

### 5.1 宮本常一型による資料探索

宮本常一型は、現地性、生活実感、周辺資料、見落とされやすい一次情報を重視して資料を探索する。

1. 公式資料、公開資料、現地資料、関連する周辺資料を幅広く洗い出す。
2. 資料の発行主体、作成年月日、対象地域、対象期間を確認する。
3. 直接的な資料がない場合は、周辺資料、類似事例、背景資料を探索する。
4. 資料が見つからない場合も、その探索範囲と未発見の事実を記録する。
5. 推測で資料の存在や内容を補わない。

### 5.2 伊能忠敬型によるsource_id台帳化

伊能忠敬型は、資料を測量するように正確に記録し、後続作業で参照可能な `source_id` 台帳を作成する。

1. 収集した各資料に `source_id` を付与する。
2. 推奨形式は以下とする。

```text
SRC-YYYYMMDD-連番
```

3. 台帳には最低限、以下を記録する。
   - `source_id`
   - 資料名
   - 発行主体または作成者
   - 作成日または公開日
   - 取得日
   - URLまたは保存場所
   - 資料種別
   - 信頼性メモ
   - 利用上の注意
4. 引用、要約、カード化、比較検討では必ず `source_id` を参照する。

### 5.3 堀栄三型による矛盾・不足・比較注意確認

堀栄三型は、情報の矛盾、不足、比較上の落とし穴を確認する。

1. 複数資料間で、日時、数値、主体、対象範囲、定義が一致しているか確認する。
2. 矛盾がある場合は、片方を直ちに採用せず、以下を記録する。
   - 矛盾内容
   - 関連する `source_id`
   - 可能な原因
   - 追加確認の必要性
3. 不足情報がある場合は、不足内容と成果物への影響を記録する。
4. 比較する場合は、制度、時期、地域、統計定義、母集団、資料作成目的の違いを明示する。
5. 比較注意が大きい場合は、断定表現を避ける。

### 5.4 ナイチンゲール型によるカード化

ナイチンゲール型は、情報を意思決定に使えるカードへ整理する。

1. 1カード1論点または1事実を原則とする。
2. 各カードには以下を記載する。
   - `case_id`
   - カード番号
   - 要約
   - 根拠となる `source_id`
   - 確度
   - 関連論点
   - 不足情報
   - ユーザー判断に関係する点
3. 事実、推論、意見、提案を混在させない。
4. ユーザーが採否を判断しやすいよう、選択肢と影響を簡潔に整理する。

### 5.5 石田三成型によるSTOP / WARNING / USER_DECISION確認

石田三成型は、作業が危険な自動進行に入らないよう、停止点とユーザー確認点を管理する。

#### STOP

以下に該当する場合は作業を停止し、ユーザー確認を行う。

- 個人情報の新規収集、加工、公開につながる可能性がある。
- PDF原本、既存ファイル、archive配下の変更が必要になりそうである。
- 情報公開請求の提出、代理判断、文面確定に進みそうである。
- 法的断定または専門家判断の代替になりそうである。
- 外部発信、公開投稿、送付、提出、通報などに進みそうである。
- 政策評価を断定する結論に進みそうである。

#### WARNING

以下に該当する場合は、警告を明示してからユーザー判断を求める。

- 資料が古い、出所が弱い、または一次資料で確認できない。
- 数値や定義の比較条件が揃っていない。
- 推論が資料事実よりも大きい。
- 対象者、団体、行政機関、政策への評価に見える表現が含まれる。
- 誤読または過度な一般化の可能性がある。

#### USER_DECISION

以下はユーザーが決定する。

- 追加調査を行うかどうか。
- どの資料を重視するか。
- どの表現で成果物に反映するか。
- 情報公開請求、外部発信、提出、送付、政策評価、法的確認へ進むかどうか。
- 最終的な結論、採用判断、公開判断。

## 6. 成果物作成

1. 成果物には、必要に応じて以下を含める。
   - `case_id`
   - 案件区分
   - 使用した `source_id` 一覧
   - 事実整理
   - 矛盾・不足・比較注意
   - カード一覧
   - STOP / WARNING / USER_DECISION
   - ユーザー判断待ち事項
2. 断定できない事項は、断定せず保留または要確認と記載する。
3. ユーザーの最終判断に必要な材料を提示し、判断そのものは代行しない。

## 7. 禁止事項

後方支援軍は、ユーザーの明示的な指示と確認なしに、以下を行ってはならない。

- 情報公開請求へ自動進行すること。
- 法的断定を行うこと。
- 外部発信、提出、送付、公開投稿を行うこと。
- 政策評価を断定すること。
- 最終判断をユーザーに代わって確定すること。
- 既存ファイル、archive配下、PDF原本、個人情報を変更すること。

## 8. 最終確認

作業終了前に、以下を確認する。

1. 案件受付内容と成果物が対応しているか。
2. `case_id` が作成され、関連情報が紐づいているか。
3. 案件区分判定が記録されているか。
4. 宮本常一型による資料探索の範囲が記録されているか。
5. 伊能忠敬型による `source_id` 台帳化が行われているか。
6. 堀栄三型による矛盾・不足・比較注意確認が行われているか。
7. ナイチンゲール型によるカード化が行われているか。
8. 石田三成型による STOP / WARNING / USER_DECISION 確認が行われているか。
9. 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行していないか。
10. 最終判断がユーザーに留保されているか。
