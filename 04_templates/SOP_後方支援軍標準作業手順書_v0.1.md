# SOP 後方支援軍標準作業手順書 v0.1

## 1. 目的

本標準作業手順書（SOP）は、後方支援軍が案件を受けたときに、どの順番で処理し、どの部門を動かし、どこで `STOP` / `WARNING` / `USER_DECISION` を出し、どの成果物へ進めるかを標準化するために定める。

後方支援軍は、資料探索、資料管理、情報分析、資料化、統制本陣確認を通じて、ユーザーが最終判断を行うための根拠・論点・未確認事項を整理する。最終判断は常にユーザーに留保し、後方支援軍が自動的に断定、提出、発信、請求、政策評価へ進行してはならない。

## 2. 適用範囲

本SOPは、後方支援軍が担当する以下の作業に適用する。

- 案件受付
- `case_id` 作成
- 案件区分判定
- 使用可能資料の確認
- 宮本常一型｜情報収集担当官による資料探索
- 伊能忠敬型｜資料管理室長による `source_id` 台帳化
- 堀栄三型｜情報分析官による矛盾・不足・比較注意確認
- ナイチンゲール型｜資料化担当官によるカード化・表化
- 石田三成型｜統制本陣による `STOP` / `WARNING` / `USER_DECISION` 確認
- 中核軍・法務官・前線部隊への引渡条件確認

## 3. 基本原則

1. `source_id` のない情報を正式根拠化しない。
2. PDF存在確認と本文確認を混同しない。
3. 事実、算出事項、推定、評価候補、未確認事項、断定不可事項を分離する。
4. 比率・構成比・増減率には、分母・分子を明記する。
5. 発言・答弁・公約・議決・予算・契約・実績を混同しない。
6. 情報公開請求へ自動進行しない。
7. 法的断定へ自動進行しない。
8. 外部発信へ自動進行しない。
9. 政策評価断定へ自動進行しない。
10. 最終判断は常にユーザーに留保する。

## 4. 成果物の基本単位

後方支援軍は、案件の性質に応じて以下の成果物へ進める。

- 受付票：案件目的、対象範囲、希望成果物、期限、禁止事項、確認待ち事項を記録する。
- `case_id` 管理メモ：案件単位の識別子と関連成果物の紐づけを記録する。
- 資料探索ログ：探索した場所、検索語、確認した資料、未発見事項を記録する。
- `source_id` 台帳：資料ごとの出所、取得日、本文確認状況、利用上の注意を記録する。
- 矛盾・不足・比較注意メモ：資料間の差異、未確認事項、比較不能条件を記録する。
- カード：1事実または1論点ごとに、根拠、確度、注意点、判断待ち事項を整理する。
- 表：数値、時系列、主体別比較、出典対応を、分母・分子・定義つきで整理する。
- 統制本陣確認票：`STOP` / `WARNING` / `USER_DECISION` の該当有無を記録する。
- 引渡メモ：中核軍、法務官、前線部隊へ引き渡せる条件または停止理由を記録する。

## 5. 標準作業手順

### Phase 0：案件受付

#### 0.1 担当

- 主担当：後方支援軍受付
- 確認担当：石田三成型｜統制本陣

#### 0.2 手順

1. ユーザーから案件を受け付ける。
2. 依頼内容から以下を抽出する。
   - 目的
   - 対象範囲
   - 希望成果物
   - 期限または優先度
   - 使用してよい資料範囲
   - 変更禁止対象
   - 想定される引渡先
3. 案件が以下の禁止領域へ直結しないか確認する。
   - 情報公開請求の提出、代理判断、文面確定
   - 法的断定
   - 外部発信、提出、送付、公開投稿
   - 政策評価断定
   - 個人情報の新規収集、加工、公開
4. 不明点が成果物の品質、安全性、引渡可否に影響する場合は、作業前に `USER_DECISION` としてユーザー確認へ戻す。

#### 0.3 出力

- 受付票
- 初期 `USER_DECISION` 一覧
- 作業継続可否メモ

#### 0.4 STOP / WARNING / USER_DECISION

- `STOP`：個人情報、黒塗り前資料、PDF原本追加、archive配下変更、既存ファイル削除が必要になりそうな場合。
- `WARNING`：目的、対象範囲、資料範囲が曖昧で、誤読や過度な一般化の可能性がある場合。
- `USER_DECISION`：成果物形式、対象範囲、追加調査の要否をユーザーが選ぶ必要がある場合。

### Phase 1：資料探索

#### 1.1 担当

- 主担当：宮本常一型｜情報収集担当官
- 連携担当：伊能忠敬型｜資料管理室長

#### 1.2 手順

1. 案件ごとに一意の `case_id` を作成する。
2. 推奨形式は `CASE-YYYYMMDD-短い案件名-連番` とする。
3. 案件区分を判定する。
   - 資料探索
   - 事実整理
   - 年表化
   - 論点整理
   - 比較検討
   - 手続確認
   - 文案作成
   - ユーザー判断支援
4. 使用可能資料を確認する。
   - 公式資料
   - 公開資料
   - ユーザー提供資料
   - 関連する周辺資料
   - 利用禁止資料
   - PDF原本の追加禁止対象
   - 黒塗り前資料の禁止対象
5. 宮本常一型｜情報収集担当官は、現地性、生活実感、周辺資料、見落とされやすい一次情報を重視して資料を探索する。
6. 資料の発行主体、作成年月日、公開日、対象地域、対象期間、資料種別を確認する。
7. PDFについては、PDFが存在することと、PDF本文を確認したことを分けて記録する。
8. 資料が見つからない場合も、探索範囲、検索語、確認先、未発見事項を記録する。
9. 推測で資料の存在や本文内容を補わない。

#### 1.3 出力

- `case_id` 管理メモ
- 案件区分判定メモ
- 使用可能資料確認メモ
- 資料探索ログ

#### 1.4 STOP / WARNING / USER_DECISION

- `STOP`：使用禁止資料、個人情報、黒塗り前資料、PDF原本追加が必要になる場合。
- `WARNING`：PDF存在確認のみで本文確認が未了の場合、または一次資料に到達していない場合。
- `USER_DECISION`：資料探索範囲を広げるか、現時点の資料で次工程へ進むかを決める場合。

### Phase 2：資料管理

#### 2.1 担当

- 主担当：伊能忠敬型｜資料管理室長
- 連携担当：宮本常一型｜情報収集担当官

#### 2.2 手順

1. 収集した各資料に `source_id` を付与する。
2. 推奨形式は `SRC-YYYYMMDD-連番` とする。
3. `source_id` 台帳には最低限、以下を記録する。
   - `source_id`
   - `case_id`
   - 資料名
   - 発行主体または作成者
   - 作成日または公開日
   - 取得日
   - URLまたは保存場所
   - 資料種別
   - PDF存在確認の有無
   - PDF本文確認の有無
   - 確認したページ、章、表、該当箇所
   - 信頼性メモ
   - 利用上の注意
4. `source_id` のない情報を正式根拠化しない。
5. 引用、要約、カード化、表化、比較検討では必ず `source_id` を参照する。
6. 本文未確認資料は、正式根拠ではなく候補資料または探索メモとして扱う。

#### 2.3 出力

- `source_id` 台帳
- 本文確認状況一覧
- 候補資料一覧

#### 2.4 STOP / WARNING / USER_DECISION

- `STOP`：出所不明資料を正式根拠化しそうな場合。
- `WARNING`：本文未確認、取得日不明、発行主体不明、資料種別不明の資料が含まれる場合。
- `USER_DECISION`：本文未確認資料を除外するか、追加確認するか、候補として残すかを決める場合。

### Phase 3：情報分析

#### 3.1 担当

- 主担当：堀栄三型｜情報分析官
- 連携担当：伊能忠敬型｜資料管理室長

#### 3.2 手順

1. 情報を以下に分類する。
   - 事実
   - 算出事項
   - 推定
   - 評価候補
   - 未確認事項
   - 断定不可事項
2. 複数資料間で、日時、数値、主体、対象範囲、定義、資料作成目的が一致しているか確認する。
3. 矛盾がある場合は、片方を直ちに採用せず、以下を記録する。
   - 矛盾内容
   - 関連する `source_id`
   - 可能な原因
   - 追加確認の必要性
   - 成果物への影響
4. 不足情報がある場合は、不足内容、追加確認方法、判断への影響を記録する。
5. 比較する場合は、制度、時期、地域、統計定義、母集団、資料作成目的の違いを明示する。
6. 比率、構成比、増減率を扱う場合は、必ず分母、分子、対象期間、算出式を明記する。
7. 発言、答弁、公約、議決、予算、契約、実績を区別し、相互に置き換えない。
8. 比較注意が大きい場合は、断定表現を避ける。

#### 3.3 出力

- 情報分類メモ
- 矛盾・不足・比較注意メモ
- 算出式メモ
- 断定不可事項一覧

#### 3.4 STOP / WARNING / USER_DECISION

- `STOP`：推定、評価候補、未確認事項を事実として扱いそうな場合。
- `WARNING`：比較条件、分母・分子、算出式、資料定義が揃っていない場合。
- `USER_DECISION`：矛盾資料のどちらを重視するか、追加確認するか、断定不可として扱うかを決める場合。

### Phase 4：資料化

#### 4.1 担当

- 主担当：ナイチンゲール型｜資料化担当官
- 連携担当：堀栄三型｜情報分析官

#### 4.2 手順

1. 1カード1事実または1論点を原則としてカード化する。
2. 各カードには以下を記載する。
   - `case_id`
   - カード番号
   - 分類（事実、算出事項、推定、評価候補、未確認事項、断定不可事項）
   - 要約
   - 根拠となる `source_id`
   - 確度
   - 関連論点
   - 矛盾・不足・比較注意
   - ユーザー判断に関係する点
3. 表化する場合は、列定義、単位、対象期間、分母、分子、算出式、関連する `source_id` を明記する。
4. 事実、推論、意見、提案を混在させない。
5. ユーザーが採否を判断しやすいよう、選択肢と影響を簡潔に整理する。
6. 外部発信用、提出用、法的主張用、政策評価断定用の文面へ自動変換しない。

#### 4.3 出力

- カード一覧
- 表
- ユーザー判断待ち事項一覧

#### 4.4 STOP / WARNING / USER_DECISION

- `STOP`：カードや表が、法的断定、政策評価断定、外部発信、情報公開請求文面へ進みそうな場合。
- `WARNING`：根拠の弱い情報が強い表現で資料化されている場合。
- `USER_DECISION`：どのカード、表、表現を採用するかをユーザーが決める場合。

### Phase 5：統制本陣確認

#### 5.1 担当

- 主担当：石田三成型｜統制本陣
- 連携担当：全担当官

#### 5.2 手順

1. 全成果物について、`STOP` / `WARNING` / `USER_DECISION` の該当有無を確認する。
2. 以下に該当する場合は `STOP` とする。
   - 個人情報の新規収集、加工、公開につながる可能性がある。
   - PDF原本の追加、既存ファイル削除、archive配下変更、黒塗り前資料の追加が必要になりそうである。
   - 情報公開請求の提出、代理判断、文面確定に進みそうである。
   - 法的断定または専門家判断の代替になりそうである。
   - 外部発信、公開投稿、送付、提出、通報などに進みそうである。
   - 政策評価を断定する結論に進みそうである。
3. 以下に該当する場合は `WARNING` とする。
   - 資料が古い、出所が弱い、または一次資料で確認できない。
   - PDF存在確認のみで本文確認が未了である。
   - 数値や定義の比較条件が揃っていない。
   - 推論が資料事実よりも大きい。
   - 対象者、団体、行政機関、政策への評価に見える表現が含まれる。
   - 誤読または過度な一般化の可能性がある。
4. ユーザーが決めるべき事項を `USER_DECISION` として列挙する。

#### 5.3 出力

- 統制本陣確認票
- `STOP` 一覧
- `WARNING` 一覧
- `USER_DECISION` 一覧

### Phase 6：USER_DECISION

#### 6.1 担当

- 主担当：ユーザー
- 支援担当：石田三成型｜統制本陣

#### 6.2 ユーザーが決定する事項

以下は必ずユーザーが決定する。

- 追加調査を行うかどうか。
- どの資料を重視するか。
- どのカード、表、表現を成果物に反映するか。
- 情報公開請求へ進むかどうか。
- 法務官へ引き渡すかどうか。
- 外部発信、提出、送付、公開投稿へ進むかどうか。
- 政策評価として扱うかどうか。
- 最終的な結論、採用判断、公開判断。

#### 6.3 手順

1. 後方支援軍は、判断材料、選択肢、影響、未確認事項を提示する。
2. 後方支援軍は、ユーザーの判断を代行しない。
3. ユーザー判断がない事項は、未決定として記録する。
4. ユーザー判断が得られないまま、請求、法的断定、外部発信、政策評価断定へ進めない。

#### 6.4 出力

- ユーザー判断結果メモ
- 未決定事項一覧
- 次工程可否メモ

### Phase 7：引渡又は停止

#### 7.1 担当

- 主担当：石田三成型｜統制本陣
- 引渡候補：中核軍、法務官、前線部隊

#### 7.2 中核軍への引渡条件

中核軍へ引き渡せるのは、以下を満たす場合に限る。

1. `case_id` が作成されている。
2. 案件区分が記録されている。
3. 使用資料に `source_id` が付与されている。
4. 事実、算出事項、推定、評価候補、未確認事項、断定不可事項が分離されている。
5. 矛盾・不足・比較注意が明示されている。
6. `STOP` が解消されている、または停止理由として記録されている。
7. 必要な `USER_DECISION` が得られている。

#### 7.3 法務官への引渡条件

法務官へ引き渡せるのは、以下を満たす場合に限る。

1. 後方支援軍が法的断定を行っていない。
2. 法的確認が必要な論点と、事実関係が分離されている。
3. 根拠資料に `source_id` が付与されている。
4. 未確認事項、断定不可事項、比較注意が明示されている。
5. ユーザーが法務官への引渡を明示的に決定している。

#### 7.4 前線部隊への引渡条件

前線部隊へ引き渡せるのは、以下を満たす場合に限る。

1. 外部発信、提出、送付、公開投稿へ進むことをユーザーが明示的に決定している。
2. 発信用または提出用の文面が、根拠資料、推定、評価候補、未確認事項を混同していない。
3. 比率、構成比、増減率の分母・分子が明記されている。
4. 発言、答弁、公約、議決、予算、契約、実績が区別されている。
5. `WARNING` が残る場合は、文面上または添付メモ上で注意事項として明示されている。
6. `STOP` が残っていない。

#### 7.5 停止条件

以下の場合は、引渡ではなく停止とする。

- `source_id` のない情報を正式根拠にしなければ成果物が成立しない場合。
- PDF本文確認が未了であるにもかかわらず、本文内容を断定する必要がある場合。
- 事実、算出事項、推定、評価候補、未確認事項、断定不可事項を分離できない場合。
- 比較条件、分母・分子、算出式が不明なまま数値判断を求められる場合。
- 発言、答弁、公約、議決、予算、契約、実績の区別ができない場合。
- 情報公開請求、法的断定、外部発信、政策評価断定へユーザー判断なしに進む必要がある場合。
- 個人情報、黒塗り前資料、PDF原本追加、archive配下変更、既存ファイル削除が必要になる場合。

#### 7.6 出力

- 引渡メモ
- 停止理由メモ
- 残存 `WARNING` 一覧
- 残存 `USER_DECISION` 一覧

## 6. 最終確認チェックリスト

作業終了前に、以下を確認する。

1. 案件受付内容と成果物が対応している。
2. `case_id` が作成され、関連成果物が紐づいている。
3. 案件区分判定が記録されている。
4. 使用可能資料と使用禁止資料が確認されている。
5. 宮本常一型｜情報収集担当官による資料探索範囲が記録されている。
6. 伊能忠敬型｜資料管理室長による `source_id` 台帳化が行われている。
7. 堀栄三型｜情報分析官による矛盾・不足・比較注意確認が行われている。
8. ナイチンゲール型｜資料化担当官によるカード化・表化が行われている。
9. 石田三成型｜統制本陣による `STOP` / `WARNING` / `USER_DECISION` 確認が行われている。
10. `source_id` のない情報を正式根拠化していない。
11. PDF存在確認と本文確認を混同していない。
12. 事実、算出事項、推定、評価候補、未確認事項、断定不可事項が分離されている。
13. 比率、構成比、増減率には分母・分子が明記されている。
14. 発言、答弁、公約、議決、予算、契約、実績を混同していない。
15. 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行していない。
16. 中核軍、法務官、前線部隊への引渡条件または停止条件が記録されている。
17. 最終判断がユーザーに留保されている。
