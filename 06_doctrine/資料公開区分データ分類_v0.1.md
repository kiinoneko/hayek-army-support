# 資料公開区分・データ分類規程 v0.1

## 1. 目的

本規程は、後方支援軍が扱う資料、データ、PDF、開示文書、黒塗り前資料、個人情報、成果物を、公開可能性と保存先に応じて分類するための資料公開区分・データ分類規程である。

GitHub public 運用において、PDF原本、個人情報、黒塗り前資料、未公開資料、APIキー、メール本文などを誤って公開しないことを目的とする。

## 2. 基本原則

- GitHub に置けるものは、公開可能性を確認済みの `PUBLIC` 資料に限る。
- `PUBLIC` 以外の資料を GitHub に置いてはならない。
- 公開可否が不明な資料は、公開してよい資料ではなく、確認待ち資料として扱う。
- AI 判断だけで公開区分を解除してはならない。
- 外部公開、提出、発信、追加資料取得、アップロードは、必要な確認と `USER_DECISION` を経て行う。
- archive 配下を自動変更しない。
- 旧版を自動削除しない。

## 3. 資料公開区分の定義

| 区分 | 意味 | 保存先 | 扱い |
| --- | --- | --- | --- |
| `PUBLIC` | 公開可能資料 | GitHub 可 | 公開してよい |
| `INTERNAL` | 内部作業資料 | Google Drive | 原則非公開 |
| `SENSITIVE` | 配慮資料 | Google Drive 制限付き | 公開前確認必須 |
| `CONFIDENTIAL` | 非公開・黒塗り前資料 | Google Drive 厳格管理 | GitHub 禁止 |
| `SECRET` | APIキー・パスワード等 | 専用管理 | GitHub・Drive 本文保存禁止 |
| `DO_NOT_UPLOAD` | アップロード禁止 | 保存禁止又は専用隔離 | AI・GitHub 投入禁止 |

## 4. GitHub に置いてよいもの

GitHub に置いてよいものは、個人情報、PDF原本、黒塗り前資料、秘密情報、未公開資料を含まない `PUBLIC` 資料に限る。

例として、以下を GitHub に置くことができる。

- ドクトリン
- SOP
- RACI
- Evals
- source_id 台帳仕様
- gap_id 台帳仕様
- warning_stop コード仕様
- テンプレート
- 公開可能な README
- 公開可能な成果物
- 個人情報や原本 PDF を含まない整理済み Markdown

## 5. GitHub に置いてはいけないもの

以下は GitHub に置いてはならない。

- PDF原本
- 個人情報
- 黒塗り前資料
- 未公開資料
- 開示文書の原本
- メール本文
- APIキー
- パスワード
- トークン
- 秘密情報
- 公開配慮が必要な資料
- 著作権上問題がある全文転載
- 外部公開前提の断定文

## 6. Google Drive に置くもの

以下は、GitHub ではなく Google Drive に置く。必要に応じて、共有範囲、閲覧権限、編集権限、保存フォルダを制限する。

- PDF原本
- 自治体資料
- 政府資料
- 議会資料
- WAM資料
- 情報公開請求結果
- 黒塗り前後資料
- 成果物PDF
- スプレッドシート台帳
- 作業中の未公開資料

## 7. 個人情報・黒塗り前資料・PDF原本の扱い

### 7.1 個人情報

- 個人情報は GitHub に置かない。
- 個人情報該当性が未確認の資料は、`SENSITIVE` 又は `CONFIDENTIAL` として扱う。
- 個人情報該当性が未確認の場合は、`WARN_PRIVACY` 又は `STOP_PRIVACY` を付与する。
- 外部公開前に、氏名、住所、電話番号、メールアドレス、識別番号、所属、発言者が特定される情報、その他個人を識別し得る情報が含まれていないか確認する。

### 7.2 黒塗り前資料

- 黒塗り前資料は GitHub に置かない。
- 黒塗り前資料は `CONFIDENTIAL` として Google Drive で厳格管理する。
- 黒塗り前資料又は黒塗り前後の混在リスクがある資料は、`WARN_SECURITY` 又は `STOP_SECURITY` を付与する。
- 黒塗り後資料であっても、復元可能性、メタデータ、注釈、非表示レイヤー、ファイル名、本文中の残存情報を確認する。

### 7.3 PDF原本

- PDF原本は GitHub に置かない。
- PDF原本は Google Drive に置き、GitHub には必要な範囲で整理済み Markdown、要約、台帳、参照情報のみを置く。
- PDF原本から作成した成果物を公開する場合は、原本が混入していないこと、著作権上問題がある全文転載ではないこと、個人情報が含まれていないことを確認する。

## 8. source_id / gap_id / warning_id / stop_id との関係

- `source_id` は資料の根拠を管理する。
- `gap_id` は未確認事項を管理する。
- `warning_id` は公開・根拠・比較・法務上の注意を管理する。
- `stop_id` は公開・提出・断定を止める条件を管理する。
- 公開区分は source_id 台帳にも記録する。
- 公開可否が不明な資料は、`gap_id` 又は `warning_id` として管理する。
- `PUBLIC` 以外の資料を GitHub に置かない。
- 根拠資料が `INTERNAL`、`SENSITIVE`、`CONFIDENTIAL`、`SECRET`、`DO_NOT_UPLOAD` の場合、GitHub には原本や本文を置かず、公開可能な範囲の参照情報のみを扱う。

## 9. WARNING / STOP / USER_DECISION コード

### 9.1 WARNING

- `WARN_PUBLIC`：公開可否未確認
- `WARN_SECURITY`：個人情報・黒塗り前資料・PDF原本混入リスク
- `WARN_PRIVACY`：個人情報該当性未確認

### 9.2 STOP

- `STOP_PUBLIC`：外部公開判断前
- `STOP_SECURITY`：個人情報・黒塗り前資料・PDF原本混入リスク
- `STOP_PRIVACY`：個人情報保護上の確認不足
- `STOP_SECRET`：APIキー・パスワード・トークン混入

### 9.3 USER_DECISION

- `USER_DECISION_PUBLIC`：外部発信するか
- `USER_DECISION_UPLOAD`：追加資料を取得・アップロードするか

## 10. 公開判断と USER_DECISION

- 外部公開、外部発信、提出、公開前提の断定、追加資料取得、アップロードは、必要な確認を行ったうえで `USER_DECISION` に戻す。
- `USER_DECISION_PUBLIC` なしに外部発信しない。
- `USER_DECISION_UPLOAD` なしに追加資料の取得、アップロード、AI 投入を進めない。
- 公開可否が不明な場合は、`WARN_PUBLIC` 又は `STOP_PUBLIC` を付与し、確認済みとして扱わない。
- AI は公開判断を補助できるが、AI 判断だけで公開区分を解除しない。

## 11. 外部公開前の確認事項

外部公開前に、少なくとも以下を確認する。

- 個人情報が含まれていないか
- 黒塗り前資料が含まれていないか
- PDF原本を GitHub へ置いていないか
- `source_id` があるか
- 未確認事項を確認済みのように書いていないか
- 法的断定、違法断定、不正断定になっていないか
- 名誉毀損、誤認、過大評価のリスクがないか
- 情報公開請求文案や外部発信文に自動進行していないか
- `USER_DECISION` を経ているか

## 12. 誤公開時の初動対応

誤公開又は誤公開のおそれがある場合は、以下の初動対応を行う。

1. 直ちに作業を停止する。
2. 該当ファイル・コミット・PRを特定する。
3. 公開範囲を確認する。
4. GitHub から削除又は非公開化を検討する。
5. 必要に応じて履歴削除・トークン無効化・権限変更を行う。
6. インシデント記録を残す。
7. 再発防止策を追加する。
8. ユーザー判断へ戻す。

## 13. 禁止事項

以下を禁止する。

- PDF原本を GitHub に置かない。
- 個人情報を GitHub に置かない。
- 黒塗り前資料を GitHub に置かない。
- APIキー、パスワード、トークンを GitHub に置かない。
- メール本文を GitHub に置かない。
- 未公開資料を GitHub に置かない。
- 公開可否未確認の資料を外部公開しない。
- `USER_DECISION` なしに外部発信しない。
- AI 判断だけで公開区分を解除しない。
- archive 配下を自動変更しない。
- 旧版を自動削除しない。

## 14. 運用メモ

- 新規資料を扱う場合は、最初に公開区分を仮設定する。
- 仮設定が `PUBLIC` 以外の場合は GitHub へ置かない。
- 公開区分が変わった場合は、source_id 台帳、関連する gap_id、warning_id、stop_id を更新する。
- 判断に迷う場合は、低い公開リスクではなく高い保護レベルを選択し、`USER_DECISION` に戻す。
