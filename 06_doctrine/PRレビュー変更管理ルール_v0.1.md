# PRレビュー変更管理ルール v0.1

## 1. 目的

本ルールは、後方支援軍において、Codexや手動編集で作成された変更を、`main`へ反映する前に確認・評価・差し戻し・承認・マージするためのPRレビュー・変更管理ルールを定める。

本規程の目的は、ドクトリン、SOP、RACI、Evals、`source_id`台帳仕様、`gap_id`台帳仕様、`warning_stop`コード仕様、資料公開区分、インシデント対応手順などの重要文書を、安全に更新することである。

## 2. 基本定義

| 用語 | 定義 |
|---|---|
| `main` | 正式軍令・正式仕様・正式テンプレートの本流。後方支援軍における正式版の基準線である。 |
| `branch` | 変更案・草案・作業枝。`main`へ反映する前の検討・作業領域である。 |
| `PR` | 変更案をレビューする軍議。差分、影響、リスク、整合性を確認し、承認・差し戻し・分割・破棄を判断する場である。 |
| `review` | 差分確認、リスク確認、整合性確認。変更の目的、範囲、安全性、既存文書との整合を確認する行為である。 |
| `merge` | 本流への正式反映。レビューとユーザー承認を経た変更を`main`へ取り込む行為である。 |
| `squash merge` | 複数コミットを1つにまとめて`main`へ反映する方法。文書・仕様・テンプレート追加の標準マージ方法である。 |
| `archive` | 旧版保存領域。原則として自動変更しない。 |

## 3. PRレビューの定義

PRレビューとは、`branch`上の変更案を`main`へ反映する前に、変更目的、変更範囲、安全性、根拠管理、既存文書との整合性、公開リスク、運用影響、USER_DECISION事項を確認する手続である。

PRレビューは、AI又は作業者が差分を提示し、ユーザーが最終判断を行うための確認手続であり、AI判断だけで`merge`を完了させるものではない。

## 4. 変更管理の定義

変更管理とは、後方支援軍の正式文書・仕様・テンプレート・運用ルールに対する変更について、変更理由、変更範囲、影響範囲、レビュー結果、マージ方法、コミット、備考を記録し、後から追跡可能にする管理手続である。

変更管理は、次の事項を対象とする。

- ドクトリン新規作成・改訂
- 統合運用プロンプト改訂
- SOP改訂
- RACI改訂
- Evals改訂
- `source_id` / `gap_id` / `warning_stop` 仕様改訂
- 資料公開区分・インシデント対応の変更
- GitHub / Google Drive運用ルール変更
- `archive`配下に関わる変更
- 外部公開ルールに関わる変更

## 5. `main`ブランチの位置づけ

`main`は、正式軍令・正式仕様・正式テンプレートの本流である。

`main`上の内容は、後方支援軍の正式な運用基準として扱う。したがって、重要文書の追加・改訂・削除は、原則として`branch`で変更案を作成し、PRレビューとユーザー承認を経てから反映する。

## 6. `branch`の位置づけ

`branch`は、変更案・草案・作業枝である。

`branch`では、PR目的に沿った差分を作成する。1つの`branch`には、原則として1つの目的のみを含める。複数目的の変更が混在する場合は、PR分割を検討する。

## 7. PRの位置づけ

PRは、変更案をレビューする軍議である。

PRでは、次の事項を明示する。

- 変更目的
- 変更対象ファイル
- 変更内容
- 影響範囲
- レビュー観点
- チェック結果
- USER_DECISION事項
- 想定マージ方法

## 8. `merge`の位置づけ

`merge`は、本流への正式反映である。

`merge`は、レビュー可能な差分について、レビュー観点を満たし、差し戻し条件に該当せず、ユーザーが承認した場合にのみ実施する。

## 9. squash merge方針

- 原則として文書・仕様・テンプレート追加は`squash merge`を標準とする。
- 1 PR = 1目的を原則とする。
- 複数目的の変更はPRを分割する。
- `merge commit`は大規模統合時のみ検討する。
- `rebase merge`は通常運用では原則使わない。
- `merge`後はPR番号とコミットメッセージを変更管理ログに記録する。

## 10. main直コミット方針

### 10.1 main直コミット可

次の軽微変更は、ユーザーの明示又は運用上の必要性が明確な場合に限り、`main`への直接コミットを許容できる。

- 空フォルダ作成
- `.gitkeep` の追加
- 明らかな誤字修正
- READMEの軽微な表記修正
- ユーザーが明示した軽微変更

### 10.2 PR必須

次の変更は、`main`への直接コミットを許さず、PRレビューを必須とする。

- ドクトリン新規作成・改訂
- 統合運用プロンプト改訂
- SOP改訂
- RACI改訂
- Evals改訂
- `source_id` / `gap_id` / `warning_stop` 仕様改訂
- 資料公開区分・インシデント対応の変更
- GitHub / Google Drive運用ルール変更
- `archive`配下に関わる変更
- 外部公開ルールに関わる変更
- 個人情報・黒塗り前資料・PDF原本・APIキー混入リスクがある変更

## 11. レビュー観点

| 観点 | 確認内容 |
|---|---|
| 範囲 | 変更対象がPR目的と一致しているか |
| 既存保護 | 既存ファイル・`archive`配下を不要に変更していないか |
| 根拠管理 | `source_id` / `gap_id` / `warning_id` / `stop_id` の思想と矛盾しないか |
| 禁止事項 | 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行していないか |
| 公開安全 | PDF原本、個人情報、黒塗り前資料、APIキーを含んでいないか |
| 整合性 | 既存ドクトリン、SOP、RACI、Evalsと矛盾しないか |
| 命名 | ファイル名、ID形式、バージョン番号が規則に合っているか |
| 影響 | 後方支援軍の運用に与える影響が明記されているか |
| テスト | 必要に応じてEvals又はチェックリストで確認したか |
| USER_DECISION | 最終判断がユーザーに留保されているか |

## 12. マージ許可条件

次の条件をすべて満たす場合に限り、`merge`を許可できる。

- 変更対象が明確である。
- 既存ファイルを不要に削除していない。
- `archive`配下を不要に変更していない。
- PDF原本、個人情報、黒塗り前資料、APIキーを追加していない。
- `source_id` / `gap_id` / `warning_stop` の運用思想と矛盾しない。
- 情報公開請求・法的断定・外部発信・政策評価断定へ自動進行していない。
- 差分がレビュー可能な大きさである。
- ユーザーがマージを承認している。

## 13. 差し戻し条件

次のいずれかに該当する場合は、PRを差し戻す。

- 変更範囲が広すぎる。
- 目的外のファイルを変更している。
- `archive`配下を変更している。
- PDF原本、個人情報、黒塗り前資料、APIキーを含む。
- `source_id`なしの事実化を許している。
- `gap_id`を根拠扱いしている。
- WARNING / STOP / USER_DECISIONを弱めている。
- 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行している。
- 既存ドクトリンと矛盾している。
- レビュー不能な大規模変更である。

## 14. 禁止事項

次の行為を禁止する。

- AI判断だけでPRをマージしない。
- AI判断だけで`archive`配下を変更しない。
- AI判断だけで旧版を削除しない。
- PDF原本、個人情報、黒塗り前資料、APIキーを追加しない。
- 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行しない。
- USER_DECISIONを省略しない。
- レビュー不能な巨大PRをそのままマージしない。
- コンフリクトがあるPRを無理にマージしない。

## 16. 変更管理ログ

変更管理ログは、PRごと又は重要変更ごとに記録する。`merge`後は、PR番号とコミットメッセージを変更管理ログに記録する。

| change_id | 日付 | PR番号 | 変更内容 | 変更種別 | 影響範囲 | レビュー結果 | マージ方法 | コミット | 備考 |
|---|---|---|---|---|---|---|---|---|---|
| CHG_0001 | YYYY-MM-DD | #000 | 変更内容を記載 | doctrine / sop / template / policy / tool / index | 影響範囲を記載 | 承認 / 差し戻し / 分割 / 破棄 | squash merge / merge commit / rebase merge | コミットハッシュ又はメッセージ | 補足事項 |

### 16.1 change_id命名規則

`change_id`は、必ず`CHG_`で始める。

使用例は次のとおりである。

- `CHG_0001`
- `CHG_DOCTRINE_001`
- `CHG_TEMPLATE_001`
- `CHG_POLICY_001`
- `CHG_TOOL_001`
- `CHG_INDEX_001`

## 17. USER_DECISION事項

次の事項は、AI又は作業者が自動決定せず、ユーザー判断に留保する。

- PRをマージするか。
- PRを差し戻すか。
- PRを分割するか。
- `archive`配下を変更するか。
- `main`直コミットを許可するか。
- 変更を破棄するか。
- インシデント対応へ移すか。
- 法務官確認へ移すか。

## 18. PR条件

PR作成時は、次の条件を満たす。

- 既存ファイルを削除しない。
- `archive`配下を変更しない。
- PDF原本を追加しない。
- 個人情報を追加しない。
- 黒塗り前資料を追加しない。
- APIキー、パスワード、トークンを追加しない。
- `main`へ直接コミットしない。
