# インシデント対応手順 v0.1

## 目的

本手順は、後方支援軍において、誤公開、個人情報混入、黒塗り前資料混入、PDF原本のGitHub追加、APIキー・トークン混入、`source_id`なしの断定、法的断定、外部発信誤作成、情報公開請求への自動進行などが起きた場合に、速やかに停止・隔離・修正・記録・再発防止を行うためのインシデント対応手順である。

## 1. インシデントの定義

インシデントとは、後方支援軍の作業、成果物、台帳、リポジトリ、共有ドライブ、外部発信、情報公開請求準備において、公開範囲、秘密管理、個人情報保護、根拠確認、評価表現、手続判断、権限設定、保存先管理に誤り又は疑いが発生し、停止・隔離・修正・記録・再発防止が必要となる事象をいう。

疑い段階であっても、個人情報、APIキー・パスワード・トークン、黒塗り前資料、非公開資料、PDF原本、外部発信、情報公開請求、archive配下変更に関係する場合は、インシデント又はインシデント候補として扱う。

## 2. インシデント区分

| incident_type | 内容 |
|---|---|
| PUBLIC_LEAK | 公開してはいけない資料の公開 |
| PRIVACY | 個人情報混入 |
| CONFIDENTIAL | 黒塗り前資料・非公開資料混入 |
| PDF_ORIGINAL | PDF原本のGitHub追加 |
| SECRET | APIキー・パスワード・トークン混入 |
| SOURCE_ERROR | `source_id`なしの事実化 |
| VALUE_ERROR | 数値誤記・分母分子誤り |
| QUOTE_ERROR | 発言原文未確認の言質化 |
| LEGAL_OVERREACH | 法的断定 |
| POLICY_OVERREACH | 政策評価断定 |
| PUBLICATION | 外部発信への誤進行 |
| REQUEST | 情報公開請求への誤進行 |
| ARCHIVE_CHANGE | archive配下の誤変更・旧版削除 |
| PERMISSION | 権限設定・共有設定ミス |

## 3. 重大度区分

| severity | 意味 | 対応 |
|---|---|---|
| S1 | 個人情報、APIキー、黒塗り前資料、非公開資料の公開 | 即時停止・削除・無効化・記録 |
| S2 | PDF原本、開示文書原本、公開配慮資料のGitHub追加 | 即時停止・削除・公開範囲確認 |
| S3 | `source_id`なし断定、誤数値、誤引用、法的断定 | 修正PR・訂正記録 |
| S4 | 表記ミス、軽微な分類ミス、リンク切れ | 通常修正 |
| S5 | 影響軽微な改善事項 | 次回改訂 |

## 4. 初動対応

インシデント又はインシデント疑いを発見した場合は、以下の順に対応する。

1. 直ちに作業を停止する。
2. 該当ファイル、PR、コミット、ブランチを特定する。
3. 公開範囲を確認する。
4. 個人情報・秘密情報・黒塗り前資料・PDF原本が含まれるか確認する。
5. 必要に応じてファイル削除、PR停止、ブランチ削除、リポジトリ非公開化を検討する。
6. APIキーやトークンが含まれる場合は即時無効化する。
7. 修正PRを作成する。
8. `incident_id`を発行して記録する。
9. 再発防止策を追加する。
10. USER_DECISIONへ戻す。

初動対応中は、影響範囲を広げる操作を避け、外部発信、情報公開請求、公開リンクの再共有、archive配下の自動変更を停止する。

## 5. GitHub上の誤公開対応

- GitHubにPDF原本、個人情報、黒塗り前資料、APIキー、未公開資料を置いてはならない。
- 誤って追加した場合は直ちにPR又はブランチを停止する。
- mainにマージ済みの場合は該当ファイルを削除する修正PRを作成する。
- 秘密情報が含まれる場合は履歴削除やトークン無効化を検討する。
- archive配下の旧版削除や改変は、原則として行わない。
- GitHub上で削除しても履歴に残る可能性があるため、SECRET系は無効化を優先する。
- PR作成前に、追加ファイルにPDF原本、個人情報、黒塗り前資料、APIキー・パスワード・トークン、未公開資料が含まれていないことを確認する。
- mainへ直接コミットせず、修正PRによって対応する。

## 6. Google Drive上の誤共有対応

Google Drive上で誤共有又は誤共有疑いが発生した場合は、以下を行う。

- 共有範囲を確認する。
- リンク共有を停止する。
- 不要な閲覧者・編集者を削除する。
- 黒塗り前資料と黒塗り後資料を分ける。
- 公開用成果物と原本資料を分ける。
- `source_id`台帳に保存先と公開区分を記録する。
- 外部共有済みの場合は、共有先、共有日時、共有ファイル名、共有権限、回収又は差替え状況をインシデント記録表に残す。

## 7. APIキー・トークン混入時の対応

APIキー、パスワード、トークン、認証情報、シークレットらしき文字列を発見した場合は、SECRETインシデントとして扱う。

1. 直ちに作業、PR、外部共有を停止する。
2. 該当ファイル、PR、コミット、ブランチ、共有先を特定する。
3. GitHub又は共有ドライブ上から該当ファイルを削除する修正PR又は削除対応を検討する。
4. 削除だけで済ませず、APIキー・トークンの即時無効化又はローテーションをUSER_DECISIONに戻す。
5. 必要に応じて履歴削除、権限変更、監査ログ確認をUSER_DECISIONに戻す。
6. `INC_SECRET_001`形式の`incident_id`を発行して記録する。
7. 再発防止策として、secret scan、レビュー手順、`.gitignore`、サンプル値化、ダミー値化を追加する。

## 8. 個人情報・黒塗り前資料混入時の対応

個人情報、個人情報該当性が未確認の情報、黒塗り前資料、非公開資料、公開配慮資料が混入した場合は、PRIVACY又はCONFIDENTIALインシデントとして扱う。

- 個人情報・黒塗り前資料・非公開資料をGitHubに残さない。
- 公開用成果物と原本資料を分離する。
- 黒塗り前資料と黒塗り後資料を分離する。
- ファイル名、本文、メタデータ、画像、添付資料、引用部分に個人情報又は黒塗り前情報が残っていないか確認する。
- 必要に応じてファイル削除、PR停止、共有停止、権限削除、リポジトリ非公開化をUSER_DECISIONに戻す。
- 影響範囲、公開範囲、閲覧者、ダウンロード可能性を記録する。
- 修正後も、再公開前に公開可否を確認する。

## 9. 誤断定・誤分析時の対応

誤断定、誤分析、根拠不足の事実化、数値誤記、引用誤り、法的断定、政策評価断定が発生した場合は、SOURCE_ERROR、VALUE_ERROR、QUOTE_ERROR、LEGAL_OVERREACH又はPOLICY_OVERREACHとして扱う。

- `source_id`なしで事実化した場合は修正する。
- 数値、分母、分子、算出式を再確認する。
- 発言原文・日付・媒体・文脈を再確認する。
- 法的断定、政策評価断定、違法・不正・癒着断定を取り消し、評価候補又は断定不可事項へ戻す。
- 必要に応じて`warning_id`、`gap_id`、`stop_id`を追加する。
- 修正履歴を残す。
- 誤分析が外部発信又は情報公開請求に進んでいた場合は、PUBLICATION又はREQUESTとしても記録する。

## 10. 外部発信・情報公開請求への誤進行時の対応

外部発信、公開文案、投稿、送付文、情報公開請求、照会、申入れ、通報、報告などに誤って進行した場合は、PUBLICATION又はREQUESTインシデントとして扱う。

1. 外部発信又は情報公開請求の作業を直ちに停止する。
2. 送信済み、予約投稿、下書き、共有リンク、添付資料、宛先を確認する。
3. 送信前であれば中止し、USER_DECISIONへ戻す。
4. 送信済みであれば、訂正、撤回、追加説明、法務官又は前線部隊への引き渡しをUSER_DECISIONに戻す。
5. `source_id`、数値、引用、法的表現、政策評価表現、個人情報、黒塗り状態、PDF原本添付の有無を再確認する。
6. 自動判断だけで外部訂正、再送、請求取下げ、追加請求を行わない。

## 11. incident_id命名規則

`incident_id`は必ず`INC_`で始める。

使用できる形式の例は以下のとおりである。

- `INC_0001`
- `INC_PUBLIC_001`
- `INC_PRIVACY_001`
- `INC_SECRET_001`
- `INC_SOURCE_001`
- `INC_VALUE_001`
- `INC_QUOTE_001`
- `INC_LEGAL_001`
- `INC_ARCHIVE_001`

区分別IDを使う場合は、incident_typeに対応する識別語を含め、連番を重複させない。

## 12. インシデント記録表

| incident_id | 発生日 | incident_type | severity | 内容 | 対象ファイル・PR・コミット | 公開範囲 | 初動対応 | 修正対応 | 再発防止策 | 状態 | USER_DECISION |
|---|---|---|---|---|---|---|---|---|---|---|---|
| INC_0001 | YYYY-MM-DD | PUBLIC_LEAK | S1 | 記入例 | path / PR / commit | public / internal / limited | 作業停止・範囲確認 | 削除PR・権限変更 | レビュー追加 | open / closed | 要 / 不要 |

## 13. WARNING / STOP

以下のWARNING及びSTOPを使用し、判断未了又は停止が必要な状態を明示する。

| id | 内容 |
|---|---|
| WARN_PUBLIC | 公開可否未確認 |
| WARN_SECURITY | 個人情報・黒塗り前資料・PDF原本混入リスク |
| WARN_PRIVACY | 個人情報該当性未確認 |
| WARN_SECRET | APIキー・トークン混入疑い |
| STOP_PUBLIC | 外部公開判断前 |
| STOP_SECURITY | 個人情報・黒塗り前資料・PDF原本混入リスク |
| STOP_PRIVACY | 個人情報保護上の確認不足 |
| STOP_SECRET | APIキー・パスワード・トークン混入 |
| STOP_ARCHIVE | archive配下変更又は旧版削除リスク |

WARNINGは注意喚起であり、STOPは作業停止又はUSER_DECISIONへの差戻しを意味する。STOPが付いた事項は、自動進行してはならない。

## 14. 再発防止

インシデント対応後は、少なくとも以下を検討し、必要なものを修正PR又は運用ルールに追加する。

- 公開可否確認チェックリストの追加。
- PDF原本、開示文書原本、黒塗り前資料、個人情報、APIキー・パスワード・トークンをGitHubに追加しないためのレビュー手順の追加。
- `source_id`必須確認、数値・分母・分子・算出式確認、引用原文確認の追加。
- 法的断定、政策評価断定、違法・不正・癒着断定を避ける表現ルールの追加。
- Google Driveの共有範囲、保存先、公開区分の台帳記録の徹底。
- archive配下変更及び旧版削除を自動実行しないためのSTOP設定。
- secret scan、pre-commit hook、`.gitignore`、サンプル値化、ダミー値化の導入。
- インシデント記録表の更新とレビュー。

## 15. USER_DECISION事項

以下はユーザー判断に留保し、AI又は作業者の自動判断だけで実行してはならない。

- リポジトリを一時非公開にするか。
- ファイルを削除するか。
- 履歴削除を行うか。
- APIキー・トークンを無効化するか。
- 外部訂正を出すか。
- 情報公開請求・外部発信を中止するか。
- 法務官へ引き渡すか。
- 前線部隊へ引き渡すか。
- 再発防止ルールを追加するか。

USER_DECISION事項は、判断理由、選択肢、期限、影響範囲、関連incident_idを添えて差し戻す。

## 16. 禁止事項

以下を禁止する。

- インシデントを隠さない。
- AI判断だけでインシデントを解消済みにしない。
- 個人情報、黒塗り前資料、PDF原本、APIキーをGitHubに残さない。
- SECRET系情報を削除だけで済ませない。
- `source_id`なし断定を放置しない。
- 法的断定・政策評価断定を放置しない。
- 外部発信や情報公開請求へ自動進行しない。
- archive配下を自動変更しない。
- 旧版を自動削除しない。
- USER_DECISIONを省略しない。

## 17. PR作成時の確認事項

インシデント対応手順又は関連ルールを変更するPRでは、以下を確認する。

- 既存ファイルを削除しない。
- archive配下を変更しない。
- PDF原本を追加しない。
- 個人情報を追加しない。
- 黒塗り前資料を追加しない。
- APIキー、パスワード、トークンを追加しない。
- mainへ直接コミットしない。
- 自動マージしない。
