# 後方支援軍Runbook接続図 v0.1
## 1. 位置づけ
この文書は、後方支援軍で作成済みのRunbook、テンプレート、ID台帳、および今後作成予定の検証Runbookを、案件処理の流れに沿って接続するための索引である。
この文書は詳細分析Runbookではない。
目的は、次の点を整理することである。
- どの場面でどのRunbookを使うか。
- どこで原文検証を行うか。
- どこで後方支援軍、ハイエク軍、前線部隊を参加させるか。
- どこでID台帳へ記録するか。
- どこで公開前チェックへ接続するか。
この文書で扱う既存文書は、次のとおりである。
- `05_tools/runbook_案件起票初動分類_v0.1.md`
- `05_tools/runbook_source_id根拠資料管理_v0.1.md`
- `05_tools/runbook_gap_warning_stop判定仕様_v0.1.md`
- `05_tools/runbook_公金支出類型判定_v0.1.md`
- `05_tools/runbook_補助金支出確認_v0.1.md`
- `05_tools/runbook_委託契約確認_v0.1.md`
- `05_tools/runbook_指定管理確認_v0.1.md`
- `05_tools/runbook_調達手続確認_v0.1.md`
- `05_tools/runbook_公開前チェック実行_v0.1.md`
- `04_templates/template_後方支援軍ID台帳_v0.1.md`
今後作成予定の接続候補は、次のとおりである。
- `05_tools/runbook_原文資料相互検証_v0.1.md`（今後作成予定）
- `05_tools/runbook_統合検証会議_v0.1.md`（今後作成予定）
上記2つは未作成の可能性があるため、既存文書として扱わない。
## 2. 全体処理フロー
後方支援軍の案件処理は、次の順序で接続する。
```text
case_id起票
→ ID台帳へ仮記録
→ source_id正式化
→ ID台帳更新
→ 原文資料相互検証
→ gap_id / warning_id / stop_id 判定
→ ID台帳更新
→ 公金支出類型判定
→ 専門Runbookへ接続
→ 統合検証会議
→ ID台帳確定更新
→ 公開前チェック
→ レポート・外部発信・情報公開請求候補へ接続
```
各段階の役割は、次のとおりである。
| 段階 | 主な目的 | 接続先 |
|---|---|---|
| case_id起票 | 案件を識別し、初動分類する | `runbook_案件起票初動分類_v0.1.md` |
| ID台帳へ仮記録 | case_id起票直後の状態を仮登録する | `template_後方支援軍ID台帳_v0.1.md`から作成した運用台帳 |
| source_id正式化 | 根拠資料を識別し、資料単位で管理する | `runbook_source_id根拠資料管理_v0.1.md` |
| ID台帳更新 | source_id正式化後の状態を更新する | 運用台帳 |
| 原文資料相互検証 | 原文、引用、数値、発言、表を確認する | `runbook_原文資料相互検証_v0.1.md`（今後作成予定） |
| gap / warning / stop 判定 | 未確認、注意、停止を分離する | `runbook_gap_warning_stop判定仕様_v0.1.md` |
| ID台帳更新 | gap_id、warning_id、stop_idの状態を更新する | 運用台帳 |
| 公金支出類型判定 | 支出の類型を決める | `runbook_公金支出類型判定_v0.1.md` |
| 専門Runbook接続 | 類型別に確認項目を深掘りする | 補助金、委託契約、指定管理、調達手続の各Runbook |
| 統合検証会議 | 後方支援軍、ハイエク軍、前線部隊で検証する | `runbook_統合検証会議_v0.1.md`（今後作成予定） |
| ID台帳確定更新 | 統合検証会議後の状態を確定更新する | 運用台帳 |
| 公開前チェック | 公開、公開用PDF化、外部発信前の最終確認を行う | `runbook_公開前チェック実行_v0.1.md` |
| 次工程接続 | レポート、外部発信、情報公開請求候補に渡す | 各案件の運用判断 |
この流れは、案件を急いでいる場合でも省略しない。
ID台帳は、処理の最後に記録するだけのメモではない。
ID台帳は、案件処理の各段階で状態を更新する管理台帳である。
初動時点では仮記録とし、source_id正式化、原文資料相互検証、gap_id / warning_id / stop_id 判定、専門Runbook確認、統合検証会議、公開前チェックの各段階で状態を更新する。
特に、source_id正式化、原文資料相互検証、gap_id / warning_id / stop_id 判定、公開前チェックは、断定表現や外部発信の前提である。
## 3. 最初に使うRunbook
新しい案件を扱う場合、最初に使うRunbookは次である。
- `05_tools/runbook_案件起票初動分類_v0.1.md`
この段階では、案件に仮の入口を与える。
確認する事項は、次のとおりである。
- 案件の対象自治体、対象部局、対象事業を把握する。
- 親`case_id`が必要か、子`case_id`で分けるべきかを検討する。
- 対象が予算、決算、契約、補助金、指定管理、調達、議会発言、監査、住民監査請求等のどれに近いかを仮分類する。
- 必要な根拠資料の候補を洗い出す。
- 重要案件か、軽微な整理案件かを仮判定する。
この段階で確定してよいのは、案件の入口と仮分類である。
この段階で確定してはいけないものは、次のとおりである。
- 不正、不当、違法、浪費などの評価語。
- 金額、引用、発言者、契約内容の断定。
- 原文未確認の資料に基づく結論。
- 支出類型が未確定のままの専門Runbook選択。
初動分類が終わったら、ID台帳へ仮記録したうえで、source_id根拠資料管理へ接続する。
## 4. source_id根拠資料管理への接続
根拠資料を扱う場合は、次のRunbookへ接続する。
- `05_tools/runbook_source_id根拠資料管理_v0.1.md`
source_idは、資料を一意に扱うための管理IDである。
source_id正式化の目的は、次のとおりである。
- 根拠資料を資料単位で識別する。
- 同一資料の重複登録を避ける。
- 後続の引用、数値、表、ページ確認の起点を作る。
- gap_id、warning_id、stop_idとの接続点を明確にする。
- ID台帳へ記録できる状態にする。
source_idに紐づけるべき情報は、案件ごとの必要性に応じて整理する。
- 資料名。
- 発行者または作成者。
- 作成日、公開日、取得日、確認日。
- URLまたは保存場所の説明。
- ページ、該当箇所、表番号、議事録の該当発言。
- 関係するcase_id。
- 未確認点、注意点、停止条件の有無。
source_idは、単にURLや資料名を登録するだけではなく、資料の確認状態を管理するために使う。
特に次を区別する。

- source_id登録済み
- URL確認済み
- 本文確認済み
- ページ確認済み
- 引用確認済み
- 数値確認済み
- 別資料照合済み
- 反証確認済み
- 公開使用可

`source_id`が存在するだけでは、原文確認済み、引用確認済み、数値確認済み、公開使用可とは扱わない。
原文、ページ、日時、発言者、資料名、金額、表、引用箇所が未確認の場合は、後続の原文資料相互検証または`gap_id / warning_id / stop_id`判定へ接続する。
source_id正式化の次には、ID台帳を更新したうえで、原文資料相互検証へ接続する。

## 5. 原文資料相互検証への接続候補
原文資料相互検証は、今後作成予定の次のRunbookへ接続する候補である。
- `05_tools/runbook_原文資料相互検証_v0.1.md`（今後作成予定）
この段階は、source_idの存在と原文確認を切り分けるための検証ゲートである。
重要な原則は、次のとおりである。
- source_idがあるだけでは原文確認済みと扱わない。
- 原文、ページ、日時、発言者、資料名、金額、表、引用箇所を確認する。
- 要約資料、議会だより、動画切り取り、SNS投稿だけで重要断定しない。
- 原文未確認の場合はgap_idまたはstop_idへ接続する。
- この段階は今後作成予定の`runbook_原文資料相互検証_v0.1.md`に接続する。
原文資料相互検証で確認する対象は、次のとおりである。
| 確認対象 | 確認内容 | 未確認時の接続 |
|---|---|---|
| 原文 | 一次資料、公式資料、議事録、契約書、仕様書等に到達しているか | gap_idまたはstop_id |
| ページ | 引用箇所のページ、表、項目が特定できるか | gap_id |
| 日時 | 会議日、契約日、交付決定日、支出日、公開日が確認できるか | gap_id |
| 発言者 | 議事録や会議録で発言者が特定できるか | gap_idまたはstop_id |
| 資料名 | 正式名称、版、年度、対象部局が確認できるか | gap_id |
| 金額 | 単位、税込税抜、予算決算、契約額、精算額の区別ができるか | warning_idまたはstop_id |
| 表 | 表番号、行、列、注記が確認できるか | gap_id |
| 引用箇所 | 引用が原文の趣旨を損なっていないか | warning_idまたはstop_id |
原文未確認のまま、次の作業をしてはならない。
- 重要断定。
- 金額の確定。
- 発言者名を伴う批判。
- 契約違反、制度不備、浪費などの評価。
- 公開用PDF化。
- 外部発信。
- 情報公開請求候補化。
内部確認用の下書きPDFを作る場合も、未確認箇所を明示する。
内部確認用PDFを公開用資料として扱ってはならない。
内部確認用資料には、未確認、要確認、仮置き、原文未照合などの状態を明記する。
原文資料相互検証が完了したら、gap_id / warning_id / stop_id 判定へ接続する。
## 6. gap_id / warning_id / stop_id 判定への接続
未確認、注意、停止条件を整理する場合は、次のRunbookへ接続する。
- `05_tools/runbook_gap_warning_stop判定仕様_v0.1.md`
gap_id / warning_id / stop_id は、案件の不確実性とリスクを分離するために使う。
使い分けの基本は、次のとおりである。
| ID | 使う場面 | 次の処理 |
|---|---|---|
| gap_id | 未確認情報、追加確認が必要な箇所 | 確認待ちとして記録し、断定しない |
| warning_id | 注意喚起が必要だが作業継続は可能な箇所 | 表現を弱め、条件付きで進める |
| stop_id | 作業停止が必要な重大リスク | 公開、断定、公開用PDF化、外部発信を止める |
この判定は、原文資料相互検証の直後に行う。
判定を省略すると、未確認情報が分析本文に混入する。
特に、次の場合はstop_id候補として扱う。
- 原文未確認のまま重要断定をしようとしている。
- 個人情報や非公開情報の混入疑いがある。
- 黒塗り前資料やsecretの混入疑いがある。
- PDF原本や画像をGitHubへ追加しようとしている。
- 旧本文と新本文が混在している。
- 変更対象外ファイルが差分に入っている。
判定結果は、ID台帳に記録できる形にしておく。
## 7. 公金支出類型判定への接続
支出類型が不明な場合は、次のRunbookへ接続する。
- `05_tools/runbook_公金支出類型判定_v0.1.md`
公金支出類型判定の目的は、補助金、委託契約、指定管理、調達手続などを混同しないことである。
支出類型を誤ると、確認すべき資料、見るべき論点、判断基準がずれる。
代表的な分岐は、次のとおりである。
| 類型 | 主な確認対象 | 接続先 |
|---|---|---|
| 補助金 | 交付要綱、交付決定、実績報告、精算、返還条件 | `runbook_補助金支出確認_v0.1.md` |
| 委託契約 | 契約書、仕様書、成果物、検収、再委託 | `runbook_委託契約確認_v0.1.md` |
| 指定管理 | 協定書、指定管理料、利用料金、モニタリング | `runbook_指定管理確認_v0.1.md` |
| 調達手続 | 入札、随意契約、プロポーザル、落札率 | `runbook_調達手続確認_v0.1.md` |
類型が複数にまたがる場合は、親case_idと子case_idを分ける。
類型不明のまま専門Runbookへ進めてはならない。
## 8. 専門Runbookへの接続
公金支出類型判定が終わったら、該当する専門Runbookへ接続する。
専門Runbookは、詳細分析を行うための接続先である。
### 8.1 補助金支出確認
補助金、交付決定、実績報告、精算、返還条件を見る場合は、次へ接続する。
- `05_tools/runbook_補助金支出確認_v0.1.md`
このRunbookでは、補助対象経費、交付条件、精算、返還、実績報告の整合性を確認する。
### 8.2 委託契約確認
契約書、仕様書、成果物、検収、再委託を見る場合は、次へ接続する。
- `05_tools/runbook_委託契約確認_v0.1.md`
このRunbookでは、契約内容、業務仕様、成果物、検収、履行確認を確認する。
### 8.3 指定管理確認
公の施設、指定管理料、利用料金、モニタリングを見る場合は、次へ接続する。
- `05_tools/runbook_指定管理確認_v0.1.md`
このRunbookでは、協定、指定管理料、利用料金制、モニタリング、指定期間を確認する。
### 8.4 調達手続確認
入札、随意契約、プロポーザル、落札率を見る場合は、次へ接続する。
- `05_tools/runbook_調達手続確認_v0.1.md`
このRunbookでは、公告、仕様書、参加資格、入札結果、随意契約理由、落札率を確認する。
専門Runbookで得た確認結果は、ID台帳へ記録する。
重要案件、外部発信予定案件、情報公開請求候補案件は、統合検証会議へ接続する。
## 9. 統合検証会議への接続候補
統合検証会議は、今後作成予定の次のRunbookへ接続する候補である。
- `05_tools/runbook_統合検証会議_v0.1.md`（今後作成予定）
この段階は、後方支援軍、ハイエク軍、前線部隊が参加する検証ゲートである。
参加者ごとの役割は、次のとおりである。
- 後方支援軍は原文、数値、引用、source_idを確認する。
- ハイエク軍は自由、財産権、納税者負担、制度効果、権力集中を評価する。
- 前線部隊はレポート化、公開用PDF化、外部発信、情報公開請求に使えるか確認する。
- 反証役は読み違い、別解釈、矛盾、断定過剰を確認する。
- この段階は今後作成予定の`runbook_統合検証会議_v0.1.md`に接続する。
統合検証会議に接続する目安は、次のとおりである。
- 外部発信を予定している。
- 公開用PDF化を予定している。
- 情報公開請求候補にする予定がある。
- 金額、契約、補助金、指定管理、調達の論点が重要である。
- 複数部隊の視点で表現を確認する必要がある。
- 反証役による読み違い確認が必要である。
統合検証会議の結果は、ID台帳へ記録する。
必要に応じて、gap_id、warning_id、stop_idを更新する。
## 10. ID台帳への接続
ID台帳へ記録する場合は、次のテンプレートから作成した運用台帳へ接続する。
- `04_templates/template_後方支援軍ID台帳_v0.1.md`
ID台帳は、案件処理の履歴と根拠を追跡するための管理表である。
記録対象の例は、次のとおりである。
- `case_id`
- 親`case_id`
- 子`case_id`
- `source_id`
- `gap_id`
- `warning_id`
- `stop_id`
- 支出類型
- 接続した専門Runbook
- 原文資料相互検証の状態
- 統合検証会議の状態
- 公開前チェックの状態
ID台帳は、単なるメモではない。
後続のレビュー、公開前チェック、情報公開請求候補化、外部発信判断のための接続点である。
ID台帳に記録する際は、未確認情報と確認済み情報を混在させない。
ID台帳への記録は、テンプレート本体ではなく、テンプレートから作成した運用台帳ファイルに行う。
`04_templates/template_後方支援軍ID台帳_v0.1.md`は雛形であり、案件記録の直接入力先ではない。
案件ごとの記録は、別途作成した運用台帳、案件台帳、または当該案件用の管理ファイルに記録する。
ID台帳は、案件処理の各段階で更新する。
初動時点では仮記録とし、source_id正式化、原文資料相互検証、gap_id / warning_id / stop_id 判定、専門Runbook確認、統合検証会議、公開前チェックの各段階で状態を更新する。
| 状態 | 意味 |
|---|---|
| 仮記録 | case_id起票直後の初期状態 |
| 確認中 | source_id、原文、金額、引用等を確認している状態 |
| 要追加確認 | gap_idがあり、追加確認が必要な状態 |
| 注意付き継続 | warning_idがあり、表現調整や条件付きで継続する状態 |
| 停止中 | stop_idがあり、公開・断定・外部発信を止める状態 |
| 検証済み | 原文確認、ID判定、専門Runbook確認が完了した状態 |
| 公開前確認済み | 公開前チェックを通過した状態 |
## 11. 公開前チェックへの接続
公開、公開用PDF化、外部発信、情報公開請求候補化の前には、次のRunbookへ接続する。
- `05_tools/runbook_公開前チェック実行_v0.1.md`
公開前チェックの目的は、次のとおりである。
- 原文確認済みの情報だけで断定しているか確認する。
- ID台帳が各段階で更新されているか確認する。
- 内部確認用PDFと公開用PDFが混同されていないか確認する。
- source_idが正式化されているか確認する。
- gap_id、warning_id、stop_idの判定が反映されているか確認する。
- 個人情報、非公開資料、secret、黒塗り前資料が混入していないか確認する。
- PDF原本や画像をGitHubへ追加していないか確認する。
- README.md、archive、04_templates、05_tools既存Runbookを目的外に変更していないか確認する。
- 旧本文と新本文が混在していないか確認する。
公開前チェックが未完了の場合、公開、公開用PDF化、外部発信、情報公開請求候補化に進めない。
公開前チェック後に変更が入った場合は、再度公開前チェックへ戻る。
## 12. 親case_idと子case_idの使い分け
親`case_id`は、案件全体をまとめる単位である。
子`case_id`は、論点、支出類型、資料群、年度、部局、契約単位を分けるために使う。
親`case_id`を使う場面は、次のとおりである。
- 1つの政策、事業、施設、制度を大きく追跡する場合。
- 複数年度、複数資料、複数支出をまとめる場合。
- レポート全体の管理単位が必要な場合。
子`case_id`を使う場面は、次のとおりである。
- 補助金、委託契約、指定管理、調達手続が混在する場合。
- 年度ごとに資料や金額が異なる場合。
- 契約ごとに仕様書、成果物、検収が異なる場合。
- source_id、gap_id、warning_id、stop_idを細かく管理する必要がある場合。
親子関係を作る場合は、ID台帳に必ず記録する。
子`case_id`で確認した結果を親`case_id`へ戻す場合は、未確認情報を混ぜない。
## 13. 後方支援軍・ハイエク軍・前線部隊の役割分担
案件処理では、各部隊の役割を分ける。
| 部隊 | 主な役割 | 主な接続先 |
|---|---|---|
| 後方支援軍 | 資料管理、source_id、原文確認、数値確認、引用確認、ID台帳管理 | source_id管理、原文資料相互検証、ID台帳、公開前チェック |
| ハイエク軍 | 自由、財産権、納税者負担、制度効果、権力集中の評価 | 統合検証会議 |
| 前線部隊 | レポート化、公開用PDF化、外部発信、情報公開請求候補化の実用性確認 | 統合検証会議、公開前チェック |
| 反証役 | 読み違い、別解釈、矛盾、断定過剰の確認 | 原文資料相互検証、統合検証会議 |
後方支援軍だけで公開判断してよいのは、軽微で、原文確認済みで、stop_idがなく、外部発信リスクが低い場合に限る。
重要案件は、統合検証会議へ接続する。

## 15. 作業停止条件
次のいずれかに該当する場合は、作業を停止し、gap_id、warning_id、stop_idのいずれかに接続する。
- 類型不明のまま専門Runbookへ進めようとしている。
- source_id未確定のまま分析を確定しようとしている。
- 原文未確認のまま引用・数値・発言を確定しようとしている。
- gap_id / warning_id / stop_id 判定を省略している。
- 補助金、委託契約、指定管理、調達手続を混同している。
- 統合検証会議が必要な重要案件で、後方支援軍だけで公開判断しようとしている。
- 公開前チェック前に外部発信・公開用PDF化・断定表現を行おうとしている。
- 個人情報、黒塗り前資料、非公開資料、secretの混入疑いがある。
- PDF原本や画像をGitHubへ追加しようとしている。
- 旧本文と新本文が混在している。
- 変更対象外ファイルが差分に入っている。
- README.md、archive、04_templates、05_tools既存Runbookが目的外に変更されている。
停止後は、次のいずれかを行う。
- 原文資料相互検証へ戻る。
- source_id根拠資料管理へ戻る。
- 公金支出類型判定へ戻る。
- ID台帳へ未確認点を記録する。
- 公開前チェックをやり直す。
- ID台帳の状態を更新する。
- 内部確認用PDFと公開用PDFの区別を確認する。
- テンプレート本体に案件記録を直接書き込んでいないか確認する。
- 対象外差分を除外する。
## 16. 禁止事項
この接続図の運用では、次を禁止する。
- mainへ直接コミットすること。
- 既存ファイルを目的外に変更すること。
- README.mdを変更すること。
- archiveを変更すること。
- 04_templatesを変更すること。
- 05_tools内の既存Runbook本文を変更すること。
- PDF原本、画像、個人情報、secret、APIキーを追加すること。
- 既存Runbook本文を長くコピーすること。
- 過去の壊れ差分や未完成差分を再利用すること。
- 旧本文と新本文を併記すること。
- 接続図である本ファイルを詳細分析Runbookとして肥大化させること。
- 今後作成予定のRunbookを既存文書として扱うこと。
- 原文未確認のまま重要断定すること。
- 公開前チェック前に外部発信すること。
`04_templates/template_後方支援軍ID台帳_v0.1.md`から作成した運用台帳ファイルに案件記録を行うことは、禁止事項には該当しない。
禁止されるのは、テンプレート本体を案件記録用に直接変更することである。
内部確認用の下書きPDFを作成すること自体は、直ちに禁止事項とはしない。
ただし、未確認箇所を明示せずに公開用資料として扱うこと、または公開前チェック前に外部発信することは禁止する。
禁止事項に該当する場合は、stop_id候補として扱う。
## 17. PR前確認条件
PR前には、次の条件を確認する。
- 変更ファイルは`00_index/index_後方支援軍Runbook接続図_v0.1.md`の1個のみである。
- 新規追加なので差分は追加行のみである。
- README.mdを変更していない。
- archiveを変更していない。
- 04_templatesを変更していない。
- 05_tools既存Runbookを変更していない。
- PDFや画像を追加していない。
- secretや個人情報を追加していない。
- 章番号の重複を作っていない。
- Markdown表を壊していない。
- 第17章まで完成している。
- ID台帳を最後だけでなく、各段階で更新する運用になっている。
- テンプレート本体に案件記録を直接書き込む運用になっていない。
- 内部確認用PDFと公開用PDFの区別が明記されている。
- 公開前チェック前に禁止されるのは、公開用PDF化、外部発信、断定表現であることが明確になっている。
- ID台帳の仮記録、確認中、要追加確認、注意付き継続、停止中、検証済み、公開前確認済みの状態区分が明記されている。
- 今後作成予定の2文書を既存文書として扱っていない。
- 原文資料相互検証と統合検証会議を接続候補として明記している。
- 全体処理フローが案件処理の順序に沿っている。
- 作業停止条件を明記している。
PR前確認で問題が見つかった場合は、公開前チェックへ戻る。
