# gap_id台帳仕様 v0.1

## 目的

この仕様は、後方支援軍が、未確認事項、不足資料、確認不能事項、本文未確認資料、比較条件不足、法的確認不足、公開可否未確認などを、推測で埋めずに追跡管理するための `gap_id` 台帳仕様を定める。

`gap_id` は、後方支援軍における「情報ギャップ管理」の中心である。不明点を曖昧にせず、何が不足しているか、なぜ必要か、誰が確認するか、次工程は何かを明示するために使用する。

## 1. gap_id の定義

`gap_id` とは、案件内で確認できていない情報、不足している資料、本文未確認の根拠、比較条件の不足、法的根拠の未確認、公開可否の未確認などを、個別に識別・追跡するための管理IDである。

`gap_id` は、未確認事項を事実として扱うためのIDではない。未確認事項を未確認のまま明示し、正式判断、カード化、外部発信、情報公開請求、自動処理などへ進む前に必要な確認工程を止めるための管理IDである。

## 2. gap_id の命名規則

`gap_id` は、必ず `GAP_` で始める。

基本形式は以下のいずれかとする。

| 形式 | 用途 |
|---|---|
| `GAP_0001` | 汎用連番 |
| `GAP_{対象}_{連番}` | 対象別の不足事項 |
| `GAP_{対象}_{種別}_{連番}` | 対象とギャップ種別を明示する場合 |

形式例：

- `GAP_0001`
- `GAP_TAKETOYO_001`
- `GAP_MIHAMA_001`
- `GAP_WAM_001`
- `GAP_CONTRACT_001`
- `GAP_SUBSIDY_001`
- `GAP_QUOTE_001`
- `GAP_LEGAL_001`

運用上の注意：

- 同一案件内で重複する `gap_id` を作成しない。
- 対象名、自治体名、資料種別、論点種別を必要に応じて含める。
- 連番は原則として3桁又は4桁で管理する。
- 一度発行した `gap_id` は、削除せず、状態を `解消済み`、`解消不能`、又は `保留` に変更して履歴を残す。

| 項目 | 内容 | 必須 |

| case_id | 案件ID | 必須 |
| gap_type | ギャップ種別 | 必須 |
| 不足内容 | 何が足りないか | 必須 |
| 必要理由 | なぜ必要か | 必須 |
| 影響 | 未解消の場合の影響 | 必須 |
| 優先度 | A / B / C / HOLD / CLOSED | 必須 |
| 状態 | 未着手、探索中、解消済み等 | 必須 |
| 必要資料 | 確認に必要な資料名 | 必須 |
| 想定取得先 | 公式サイト、WAM、自治体、議会、情報公開請求等 | 任意 |
| 関連source_id | 関連するsource_id | 任意 |
| 関連warning_id | 関連するwarning_id | 任意 |
| 関連stop_id | 関連するstop_id | 任意 |
| 関連request_id | 情報公開請求候補ID | 任意 |
| 担当部門 | 宮本、伊能、堀、ナイチンゲール、石田等 | 必須 |
| 次工程 | 次に何をするか | 必須 |

| 備考 | 注意点 | 任意 |

## 5. 優先度区分

| 優先度 | 意味 |
|---|---|
| A | 正式判断・カード化・次工程に必須 |
| B | 精度向上に必要 |
| C | 参考確認 |
| HOLD | ユーザー判断待ち |
| CLOSED | 解消済み |

## 6. 解消状態区分

`gap_id` の状態は、以下から選択する。

- 未着手
- 探索中
- 資料候補確認
- source_id候補あり
- 本文確認待ち
- ユーザーアップロード待ち
- 情報公開請求候補
- 法務官確認待ち
- USER_DECISION待ち
- 解消済み
- 解消不能
- 保留

## 7. source_id / warning_id / stop_id / request_id との関係

- `source_id` は確認済み根拠を管理する。
- `gap_id` は未確認事項を管理する。
- `warning_id` は注意事項を管理する。
- `stop_id` は停止条件を管理する。
- `request_id` は情報公開請求候補を管理する。

`gap_id` は、`source_id` が不足している部分を補うものではない。`gap_id` は、不足を不足として明示し、正式化を止めるための管理IDである。

関係整理：

- `source_id` が付与できない場合、必要に応じて `gap_id` を登録する。
- 未確認事項に注意喚起が必要な場合、関連 `warning_id` を記録する。
- 未確認事項により次工程を止める必要がある場合、関連 `stop_id` を記録する。
- 公開資料では解消できない可能性がある場合、関連 `request_id` 候補を記録する。

## 8. gap_id 登録条件

以下の場合は `gap_id` を登録する。

- 必要資料が未取得
- PDF本文が未確認
- 該当ページ・該当欄が未確認
- 主要数値が未確認
- 分母・分子が不明
- 補助金・委託料・交付金・制度収入が分離できない
- 発言原文・媒体・日付・文脈が未確認
- 比較条件が不足している
- 法的根拠・条例・要綱が未確認
- 職員数・FTE・人件費内訳が未確認
- 公開可否・個人情報・黒塗り前資料リスクが残る
- 情報公開請求候補として残す必要がある

## 9. gap_id 解消条件

`gap_id` は、以下の条件を満たした場合に `解消済み` とする。

- 必要資料を取得した
- `source_id` が付与された
- 本文確認が完了した
- 該当ページ・該当欄が確認された
- 数値、分母、分子、算出式が確認された
- 発言原文、媒体、日付、文脈が確認された
- 法的根拠又は確認先が明確になった
- ユーザーが保留又は終了を判断した
- 情報公開請求の対象として `request_id` へ移管した

解消時は、関連する `source_id`、確認した資料名、確認ページ、確認欄、確認日、判断者を可能な範囲で記録する。

## 10. 情報公開請求候補との関係

- `gap_id` が公開資料で解消できない場合、`request_id` 候補へ移管できる。
- ただし、情報公開請求の提出判断は `USER_DECISION` とする。
- `gap_id` から自動的に情報公開請求へ進行しない。
- `request_id` へ移す場合も、請求対象文書、保有機関、対象期間、確認目的、不存在・不開示リスクを整理する。

`request_id` へ移管した後も、元の `gap_id` には移管先 `request_id`、移管日、移管理由、残存リスクを記録する。

## 12. 運用例

### 例1：武豊町社協 補助金額未確認

- gap_id：`GAP_TAKETOYO_SUBSIDY_001`
- gap_type：`SUBSIDY`
- 不足内容：令和6年度の武豊町から社協への補助金交付額が未確認
- 必要理由：町依存度、公的財源比率、納税者負担整理に必要
- 次工程：武豊町決算資料・主要施策成果報告書・補助金一覧の確認
- 解消条件：`source_id` 付きで補助金額、年度、該当ページ、該当欄を確認

### 例2：WAM計算書類 本文未確認

- gap_id：`GAP_WAM_BODY_001`
- gap_type：`BODY`
- 不足内容：WAM計算書類のPDF本文が未確認
- 必要理由：サービス活動収益、人件費、資産負債を正式化できない
- 次工程：PDF本文確認又はユーザーアップロード
- 解消条件：本文確認済み `source_id` へ移行

### 例3：議員発言 原文未確認

- gap_id：`GAP_QUOTE_001`
- gap_type：`QUOTE`
- 不足内容：発言原文、媒体、日付、文脈が未確認
- 必要理由：`quote_id` として正式言質化できない
- 次工程：議事録、動画、SNS原文、公式会議録の確認
- 解消条件：発言原文、媒体、日付、文脈を確認

## 13. 禁止事項

- `gap_id` を根拠として事実化しない。
- `gap_id` を使って推定を事実化しない。
- 未確認事項を確認済み事項のように扱わない。
- `gap_id` がある状態で評価断定しない。
- `gap_id` がある状態で外部発信へ自動進行しない。
- `gap_id` がある状態で情報公開請求を自動提出しない。
- `gap_id` を放置せず、状態と次工程を記録する。
- PDF原本、個人情報、黒塗り前資料をGitHubに追加しない。
- `archive` 配下を変更しない。
