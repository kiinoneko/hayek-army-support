# ID体系一覧・相互接続仕様 v0.2
# ID命名規則・接続ルール・停止ゲート

## 1. 目的

本仕様書は、既存の `ID体系一覧_相互接続仕様_v0.1.md` を基礎として、後方支援軍で使用する各種IDを、単なる一覧ではなく実務運用できるルール体系として整理するための仕様書である。

後方支援軍が調査、資料確認、発言整理、政策分析、情報公開請求、判断停止、引継ぎを行う際に、AIが未確認情報を正式化したり、ユーザー判断なしに次工程へ進んだりすることを防ぐため、ID命名規則、ID接続ルール、正式化禁止ルール、処理停止ゲートを明文化する。

本仕様におけるIDは、証拠そのものではなく、証拠・判断・処理を追跡する索引である。正式な事実認定、政策評価、発言整理、情報公開請求、外部発信は、必ず根拠資料、判断記録、処理状態に戻って確認する。

## 2. 基本原則

| 原則 | 内容 |
|---|---|
| 追跡性 | すべての案件は、`case_id`を中心に根拠、未確認点、注意、停止、判断、請求、引継ぎ、変更を追跡できる状態にする。 |
| 根拠優先 | `source_id`なしに事実、数値、発言、政策内容を正式化してはならない。 |
| 判断分離 | AIによる整理、提案、下書きと、ユーザーによる判断を分離する。 |
| 停止尊重 | `stop_id`が発生した場合、ユーザー判断または追加資料確認まで自動進行してはならない。 |
| 未確認明示 | `gap_id`が残る場合、評価断定してはならない。 |
| 注意保持 | `warning_id`は処理停止ではないが、最終評価や外部発信時に無視してはならない。 |
| 接続必須 | 各IDは孤立させず、案件、資料、政策、判断、請求、引継ぎ、変更のいずれかへ接続する。 |
| 索引性 | IDは証拠そのものではなく、証拠・判断・処理を追跡する索引として扱う。 |

## 3. ID命名規則

### 3.1 共通ルール

- IDは原則として、`種別_YYYYMMDD_連番`、または対象を含む固定フォーマットで命名する。
- 日付はIDを発行した日、または案件管理上の基準日を `YYYYMMDD` 形式で記録する。
- 連番は同一日・同一種別内で重複しない3桁以上の番号を使用する。
- 一度発行したIDは、別の意味で再利用しない。
- 廃止、統合、修正が発生した場合は、削除ではなく `change_id` で変更履歴を残す。
- IDは検索・照合・引継ぎのための索引であり、IDだけを根拠として扱ってはならない。

### 3.2 表記上の注意

- ID名、接頭辞、コード表記は、台帳間で機械的に照合できるよう原表記を維持する。
- 日本語名称を含める場合は、自治体名、分野名、対象名を過度に長くしない。
- 個人情報、秘匿情報、未加工資料の内容をID文字列へ直接埋め込まない。

## 4. IDごとの命名フォーマット

| ID種別 | 命名フォーマット | 用途 | 例 |
|---|---|---|---|
| `case_id` | `case_YYYYMMDD_連番` | 案件全体の管理 | `case_20260625_001` |
| `source_id` | `source_YYYYMMDD_連番` | 根拠資料、公式資料、開示資料の管理 | `source_20260625_001` |
| `gap_id` | `gap_YYYYMMDD_連番` | 未確認点、不足資料、照合未了の管理 | `gap_20260625_001` |
| `warning_id` | `warning_YYYYMMDD_連番` | 注意喚起、文脈注意、比較注意の管理 | `warning_20260625_001` |
| `stop_id` | `stop_YYYYMMDD_連番` | 自動進行を止める停止理由の管理 | `stop_20260625_001` |
| `actor_id` | `actor_自治体名_氏名` | 政治行政アクター、発言者、担当者の管理 | `actor_武豊町_山田太郎` |
| `quote_id` | `quote_YYYYMMDD_連番` | 発言、答弁、公約、投稿等の管理 | `quote_20260625_001` |
| `policy_id` | `policy_自治体名_分野_連番` | 政策、事業、制度、契約、補助金の管理 | `policy_武豊町_福祉_001` |
| `request_id` | `request_YYYYMMDD_連番` | 情報公開請求、照会、確認依頼等の管理 | `request_20260625_001` |
| `decision_id` | `decision_YYYYMMDD_連番` | ユーザー判断の記録 | `decision_20260625_001` |
| `handoff_id` | `handoff_YYYYMMDD_連番` | 別部門、別AI、別案件への引渡管理 | `handoff_20260625_001` |
| `incident_id` | `incident_YYYYMMDD_連番` | 誤公開、誤断定、情報混入等の事故管理 | `incident_20260625_001` |
| `change_id` | `change_YYYYMMDD_連番` | 仕様、判断、資料、評価、請求文案等の変更管理 | `change_20260625_001` |

## 5. ID接続ルール

### 5.1 基本接続表

| ID | 必須または主要接続先 | 接続目的 |
|---|---|---|
| `case_id` | 全関連ID | 案件全体を束ねる中心IDとして使用する。 |
| `source_id` | `case_id`, `quote_id`, `policy_id`, `request_id` | 事実、数値、発言、政策整理の根拠資料を示す。 |
| `gap_id` | `case_id`, `request_id`, `decision_id` | 未確認事項を表示し、解消方法を追跡する。 |
| `warning_id` | `case_id`, `source_id`, `policy_id` | 注意喚起を評価・報告時に引き継ぐ。 |
| `stop_id` | `case_id`, `decision_id`, `source_id` | 自動進行禁止の理由と解除条件を示す。 |
| `actor_id` | `quote_id`, `policy_id`, `case_id` | 関与者、発言者、責任線を追跡する。 |
| `quote_id` | `source_id`, `actor_id`, `policy_id` | 発言原文と発言者、関連政策を結ぶ。 |
| `policy_id` | `source_id`, `quote_id`, `decision_id`, `case_id` | 政策・事業を根拠資料、発言、判断、影響範囲へ接続する。 |
| `request_id` | `gap_id`, `decision_id`, `source_id`, `case_id` | 情報公開請求、照会、確認依頼等の外部アクションを管理する。 |
| `decision_id` | `case_id`, `request_id`, `handoff_id`, `change_id` | ユーザー判断を必要工程へ接続する。 |
| `handoff_id` | `case_id`, `decision_id`, `source_id` | 別部門・別AI・別案件への引渡条件を示す。 |
| `incident_id` | `case_id`, `change_id`, `decision_id` | 誤公開、誤断定、個人情報混入等の処理経緯を示す。 |
| `change_id` | `decision_id`, `incident_id`, `case_id` | 仕様、判断、資料、評価、請求文案の変更履歴を示す。 |

### 5.2 必須接続ルール

- `quote_id`は必ず`source_id`に接続する。
- `policy_id`は、根拠資料、発言、判断、影響範囲のいずれかに接続する。
- `request_id`は、情報公開請求、照会、確認依頼など外部アクションに使う。
- `handoff_id`は、別部門・別AI・別案件へ引き渡すときに使う。
- `change_id`は、仕様、判断、資料、評価、請求文案に変更があった場合に使う。
- `source_id`と接続していない情報は、仮置き、未確認、下書きとして扱う。

## 6. 正式化禁止ルール

以下の状態では、事実、数値、発言、政策評価、請求文案、外部発信を正式化してはならない。

| 禁止条件 | 禁止される処理 | 必要な対応 |
|---|---|---|
| `source_id`がない | 事実・数値・発言・政策内容の正式化 | 根拠資料を確認し、`source_id`を発行・接続する。 |
| `decision_id`がない | ユーザー判断を要する工程への進行 | ユーザー判断を取得し、`decision_id`へ記録する。 |
| `gap_id`が未解消 | 評価断定、外部発信、請求文案確定 | 未確認点を解消、保留、請求化、判断待ちに分類する。 |
| `quote_id`が`source_id`未接続 | 発言原文の正式言質化 | 議事録、会見録、投稿、動画等の根拠を確認する。 |
| `stop_id`が未解除 | 自動進行、提出、公開、引渡 | ユーザー判断または追加資料確認を待つ。 |
| 黒塗り・個人情報の確認不足 | 資料公開、リポジトリ追加、外部共有 | 加工状態、公開区分、個人情報混入有無を確認する。 |

## 7. 処理停止ゲート

処理停止ゲートとは、AIが安全に次工程へ進めない状態を明示し、自動進行を止めるための運用ゲートである。

### 7.1 停止ゲートを発動する場面

- 根拠資料が未確認であるにもかかわらず、正式評価や断定を求められた場合。
- 情報公開請求、外部照会、リポジトリ公開、法務引渡など、ユーザー判断が必要な工程へ進む場合。
- `gap_id`が残っており、評価断定や外部発信に影響する場合。
- 発言原文が確認できないまま、`quote_id`を正式化しようとする場合。
- 個人情報、黒塗り、未加工資料、非公開情報が混入している可能性がある場合。
- 既存判断、仕様、請求文案、評価を変更するが、変更理由が記録されていない場合。

### 7.2 停止中に許される処理

- 未確認点の列挙。
- 必要資料の整理。
- 請求文案または照会文案の下書き作成。
- ユーザーへ判断が必要な選択肢を提示すること。
- `source_id`、`gap_id`、`warning_id`、`stop_id`、`decision_id`の記録準備。

### 7.3 停止中に禁止される処理

- 正式評価として断定すること。
- 外部提出用文案を確定すること。
- ユーザー判断なしに情報公開請求へ進むこと。
- `stop_id`の解除条件を満たさずに次工程へ進むこと。
- 未加工資料、黒塗り前資料、個人情報を不用意に公開すること。

## 8. decision_id が必要な場面

`decision_id`は、ユーザー判断をAI処理から分離し、判断責任と処理履歴を明確にするために使用する。

| 場面 | `decision_id`が必要な理由 |
|---|---|
| 情報公開請求を提出する | 外部アクションであり、請求範囲、文案、提出先の判断が必要であるため。 |
| 照会・確認依頼を外部へ送る | 外部接触に該当し、文面と宛先の判断が必要であるため。 |
| 政策評価を断定調でまとめる | 評価の強さ、公開範囲、根拠十分性の判断が必要であるため。 |
| 法務官、中核軍、前線部隊、別AIへ引き渡す | 引渡先、目的、添付資料、未確認点の扱いを判断するため。 |
| リポジトリへ資料や成果物を追加する | 公開区分、個人情報、黒塗り状態の確認が必要であるため。 |
| `stop_id`を解除する | 停止理由が解消されたか、保留して進むかの判断が必要であるため。 |
| 仕様、判断、評価、請求文案を変更する | 変更理由と変更責任を明確化する必要があるため。 |

**必須ルール:** `decision_id`なしにユーザー判断を要する工程へ進んではならない。

## 9. source_id が必要な場面

`source_id`は、根拠資料を追跡するための索引であり、正式化の前提条件である。

| 場面 | 必要な根拠資料の例 |
|---|---|
| 事実を正式化する | 公式資料、議事録、開示資料、告示、契約書、予算書、決算書。 |
| 数値を確定する | 決算カード、予算書、統計資料、事業報告、開示資料。 |
| 発言を`quote_id`化する | 議事録、会見録、本人投稿、動画、音声、公式記録。 |
| 政策・事業を`policy_id`化する | 事業計画、予算資料、条例、要綱、契約資料、議会資料。 |
| 情報公開請求の根拠を整理する | 不足資料一覧、既存公開資料、過去回答、制度資料。 |
| 引継ぎ資料を作る | 原資料、確認済み資料、未確認点、判断ログ。 |

**必須ルール:** `source_id`なしに事実を正式化してはならない。

## 10. gap_id / warning_id / stop_id の使い分け

| ID | 意味 | 処理状態 | 自動進行 |
|---|---|---|---|
| `gap_id` | 未確認、不足、照合未了 | 解消、保留、請求化、判断待ちに分類する。 | 評価断定は不可。調査整理は可。 |
| `warning_id` | 注意喚起、文脈注意、比較注意 | 最終評価や外部発信時に参照する。 | 停止ではないため進行可。ただし無視不可。 |
| `stop_id` | 自動進行禁止 | ユーザー判断または追加資料確認まで停止する。 | 不可。解除条件が必要。 |

### 10.1 運用上の判断

- `gap_id`が残る場合、評価断定してはならない。
- `warning_id`は注意喚起であり、処理停止ではない。
- `stop_id`が発生した場合、ユーザー判断または追加資料確認まで自動進行してはならない。
- `gap_id`を根拠として扱ってはならない。
- `warning_id`を無視した最終評価をしてはならない。

## 11. handoff_id / change_id の使い分け

| ID | 使う場面 | 記録すべき内容 |
|---|---|---|
| `handoff_id` | 別部門、別AI、別案件、中核軍、法務官、前線部隊へ引き渡す場合 | 引渡先、引渡目的、関連ID、未確認点、停止事項、必要判断。 |
| `change_id` | 仕様、判断、資料、評価、請求文案、台帳、PR内容に変更があった場合 | 変更対象、変更前、変更後、変更理由、関連`decision_id`、関連`incident_id`。 |

### 11.1 使い分けの基準

- 担当や処理単位が移る場合は `handoff_id` を使う。
- 内容、判断、文案、資料、仕様が変わる場合は `change_id` を使う。
- 引渡と同時に資料や評価が変更される場合は、`handoff_id` と `change_id` の両方を発行し、相互接続する。

## 13. 禁止事項

後方支援軍の実務運用では、以下を禁止する。

- 根拠資料なしの正式評価。
- 出典不明の数値確定。
- 発言原文なしの `quote_id` 正式化。
- ユーザー判断なしの情報公開請求文案確定。
- `stop_id` 発生後の自動進行。
- `warning_id` を無視した最終評価。
- `gap_id` を未解消のまま断定評価。
- IDだけを根拠として扱うこと。
- 黒塗り・個人情報・未加工資料の不用意な公開。
- `source_id`なしに事実を正式化すること。
- `decision_id`なしにユーザー判断を要する工程へ進むこと。
- `policy_id`だけで政策の違法性、不正、癒着、効果、失敗を断定すること。
- `actor_id`だけで個人責任、故意、違法、不正を断定すること。
- `request_id`なしに情報公開請求、照会、確認依頼などの外部アクションを管理すること。
- `change_id`なしに重要な仕様変更、判断変更、資料差替え、請求文案変更を処理済みにすること。

## 14. 運用例

### 14.1 議会発言を整理する場合

| 手順 | 処理 | 必要ID |
|---|---|---|
| 1 | 議事録、動画、会見録等を確認する。 | `source_id` |
| 2 | 発言者を整理する。 | `actor_id` |
| 3 | 発言原文または要旨を登録する。 | `quote_id` |
| 4 | 関連政策へ接続する。 | `policy_id` |
| 5 | 文脈上の注意があれば記録する。 | `warning_id` |

**停止条件:** 発言原文が確認できない場合、`quote_id`を正式化せず、`gap_id`または`stop_id`を発行する。

### 14.2 政策分析を行う場合

| 手順 | 処理 | 必要ID |
|---|---|---|
| 1 | 予算書、決算書、事業資料、条例、要綱を確認する。 | `source_id` |
| 2 | 政策・事業単位を登録する。 | `policy_id` |
| 3 | 不足資料を整理する。 | `gap_id` |
| 4 | 比較上の注意を記録する。 | `warning_id` |
| 5 | 断定評価が必要な場合、ユーザー判断を取得する。 | `decision_id` |

**停止条件:** `gap_id`が評価結論に影響する場合、断定評価へ進まず、情報公開請求または追加確認へ回す。

### 14.3 情報公開請求を行う場合

| 手順 | 処理 | 必要ID |
|---|---|---|
| 1 | 既存公開資料を確認する。 | `source_id` |
| 2 | 不足資料を明確化する。 | `gap_id` |
| 3 | 請求または照会の目的を整理する。 | `request_id` |
| 4 | 請求文案を下書きする。 | `request_id`, `change_id` |
| 5 | 提出可否、宛先、文案確定についてユーザー判断を得る。 | `decision_id` |

**停止条件:** `decision_id`がない場合、情報公開請求文案を確定してはならない。

### 14.4 別AIまたは別部門へ引き渡す場合

| 手順 | 処理 | 必要ID |
|---|---|---|
| 1 | 案件概要を整理する。 | `case_id` |
| 2 | 根拠資料、未確認点、注意、停止を列挙する。 | `source_id`, `gap_id`, `warning_id`, `stop_id` |
| 3 | 引渡先と引渡目的を明確にする。 | `handoff_id` |
| 4 | ユーザー判断が必要な場合は記録する。 | `decision_id` |
| 5 | 引渡後の変更があれば記録する。 | `change_id` |

**停止条件:** `stop_id`が未解除の場合、引渡資料には停止理由と解除条件を明記し、自動進行可能な案件として扱ってはならない。

## v0.3で追加すべき項目

- 各ID台帳の必須項目と任意項目の統一フォーマット。
- `stop_id`解除手順と解除後レビューの標準様式。
- `warning_id`の重大度分類と最終評価への反映方法。
- `gap_id`から`request_id`へ変換する判断基準。
