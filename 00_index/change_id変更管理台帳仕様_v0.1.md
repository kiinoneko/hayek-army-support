# change_id変更管理台帳仕様 v0.1

## 目的

本仕様は、後方支援軍において、仕様書、ドクトリン、SOP、RACI、Evals、ID台帳、案件台帳、請求文案、発言台帳、政策台帳、資料化テンプレート等に変更が発生した場合に、その変更内容、変更理由、変更前後、PR番号、判断記録、関連IDを `change_id` で管理するための仕様である。

`change_id` は、何を、いつ、なぜ、誰の判断で、どのPR又は案件で変更したかを追跡するためのIDである。

変更管理は、後方支援軍の軍令・台帳・テンプレートが無秩序に増殖したり、旧版が消えたり、判断根拠が不明になることを防ぐために行う。

---

## 1. `change_id` の定義

- `change_id` は、後方支援軍の仕様、台帳、判断、文案、資料、評価、PR、運用ルールに変更があった場合に発行する変更管理IDである。
- `change_id` は変更そのものの根拠ではなく、変更履歴を追跡するための索引である。
- 変更内容の根拠は、関連 `source_id`、`decision_id`、PR番号、`case_log`、`incident_id` 等へ戻って確認する。
- 重要変更は `change_id` なしに処理済みにしない。

---

## 2. 変更管理の位置づけ

- PRは変更提案である。
- `main` は正式運用版である。
- squash merge後の内容は正式変更として扱う。
- `change_id` はPR、仕様更新、判断変更、文案修正、差し戻し、再作成、インシデント対応を追跡する。
- 変更管理は、旧版削除、上書き事故、判断不明、根拠不明、責任線不明を防ぐために行う。
- 変更は原則として削除ではなく履歴化する。

---

## 3. `change_id` 命名規則

| 形式 | 用途 | 例 |
|---|---|---|
| `CHG_YYYYMMDD_連番` | 汎用変更 | `CHG_20260625_001` |
| `CHG_PR_連番` | PR単位の変更 | `CHG_PR_034` |
| `CHG_SPEC_連番` | 仕様書変更 | `CHG_SPEC_001` |
| `CHG_TEMPLATE_連番` | テンプレート変更 | `CHG_TEMPLATE_001` |
| `CHG_DECISION_連番` | 判断変更 | `CHG_DECISION_001` |
| `CHG_INCIDENT_連番` | インシデント対応変更 | `CHG_INCIDENT_001` |
| `CHG_REQUEST_連番` | 情報公開請求文案変更 | `CHG_REQUEST_001` |

- 必ず `CHG_` で始める。
- 一度発行した `change_id` は別変更に再利用しない。
- 変更対象名や個人名を長くIDに埋め込まない。
- 詳細は台帳項目に記録する。

---

`change_id` の管理対象は、以下を含む。

- ドクトリン
- SOP
- RACI
- Evals
- ID体系仕様
- `source_id` 台帳仕様
- `gap_id` 台帳仕様
- `warning_stop` コード仕様
- `actor_id` 台帳仕様
- `quote_id` 台帳仕様
- `policy_id` 台帳仕様
- `request_id` 台帳仕様
- 案件台帳テンプレート
- PRレビュー・マージ判定チェックリスト
- 情報公開請求文案
- 発言整理
- 政策評価候補
- 資料化テンプレート
- README
- `archive` 配下の旧版資料
- GitHub公開資料
- Google Drive保存先情報

## 5. 変更区分

| change_type | 内容 |
|---|---|
| ADD | 新規追加 |
| UPDATE | 既存内容の更新 |
| FIX | 誤記、接頭辞、表記揺れ等の修正 |
| REFACTOR | 構成整理、重複整理 |
| RENAME | ファイル名、見出し、ID名の変更 |
| SPLIT | ファイル又は案件の分割 |
| MERGE | ファイル又は案件の統合 |
| ARCHIVE | 旧版保存 |
| REVERT | 差し戻し、取り消し |
| INCIDENT_FIX | インシデント対応修正 |
| USER_DECISION | ユーザー判断による変更 |
| OTHER | その他 |

---

## 6. `change_id` 台帳の必須項目

| 項目 | 内容 | 必須 |
|---|---|---|
| change_id | 変更管理ID | 必須 |
| change_type | 変更区分 | 必須 |
| 変更対象 | ファイル、台帳、文案、判断、資料等 | 必須 |
| 対象パス | GitHub上のファイルパス等 | 任意 |
| PR番号 | 関連PR番号 | 任意 |
| case_id | 関連案件ID | 任意 |
| decision_id | 関連ユーザー判断ID | 必要に応じ必須 |
| incident_id | インシデント関連ID | 必要に応じ必須 |
| source_id | 変更根拠資料 | 必要に応じ必須 |
| 変更前 | 変更前の状態又は要約 | 必須 |
| 変更後 | 変更後の状態又は要約 | 必須 |
| 変更理由 | なぜ変更したか | 必須 |
| 変更日 | 変更日 | 必須 |
| 変更者・担当 | 担当部門又は担当者 | 必須 |
| レビュー状態 | 未レビュー、レビュー済み、差し戻し等 | 必須 |
| マージ状態 | 未マージ、squash済み、差し戻し等 | 任意 |
| 備考 | 注意事項 | 任意 |

---

## 7. PRとの関係

- 重要なPRには `change_id` を接続する。
- PR番号、PRタイトル、対象ファイル、変更行数、削除行数、レビュー結果、マージ方法を記録する。
- Squash mergeした場合、PR単位で1つの正式変更として記録する。
- コンフリクトがあった場合、修正経緯を記録する。
- 差し戻したPRも必要に応じて `change_id` を付与する。
- PR説明文だけでなく、ファイル本文を確認する。

---

## 8. `decision_id` との関係

- ユーザー判断が必要な変更は `decision_id` と接続する。
- `archive` 変更、既存ファイル削除、重要仕様上書き、外部公開、STOP解除、インシデント終了判断は `decision_id` を必要とする。
- AI判断だけで重要変更を確定しない。
- `decision_id` なしに `USER_DECISION` 事項を変更済みにしない。

---

## 9. `incident_id` との関係

- 誤公開、個人情報混入、黒塗り前資料混入、APIキー混入、誤断定、誤マージ、`archive` 破壊等が関係する変更は `incident_id` と接続する。
- インシデント対応として変更した場合、`change_type` は `INCIDENT_FIX` とする。
- インシデント対応変更は、初動、修正、再発防止、終了判断を記録する。
- `incident_id` なしに重大インシデントを処理済みにしない。

---

## 10. `archive` 配下との関係

- `archive` 配下は旧版保存領域である。
- `archive` 配下を自動変更しない。
- `archive` 配下の削除、上書き、移動は `USER_DECISION` 対象とする。
- `archive` 変更がある場合は `change_id` と `decision_id` を必須とする。
- 旧版は原則として削除ではなく保存する。

---

## 11. 変更前後の記録ルール

- 変更前と変更後を要約で記録する。
- 小さな誤記修正でも、重要仕様に関係する場合は記録する。
- ID接頭辞修正、禁止事項追加、STOP条件追加、公開区分修正は重要変更として扱う。
- 変更理由を「修正」だけで済ませず、なぜ必要だったかを書く。
- 変更前全文を長く転載しない。
- 必要ならPR差分、`source_id`、`case_log` を参照する。

---

## 12. 変更レビュー手順

| 手順 | 内容 |
|---|---|
| 1 | 変更対象を確認する |
| 2 | 変更理由を確認する |
| 3 | 既存仕様との整合性を確認する |
| 4 | 影響範囲を確認する |
| 5 | archive、PDF原本、個人情報、黒塗り前資料、APIキー混入を確認する |
| 6 | USER_DECISIONが必要か確認する |
| 7 | change_idを発行する |
| 8 | PRレビュー結果と接続する |
| 9 | マージ又は差し戻しを記録する |
| 10 | 必要に応じてcase_log、decision_id、incident_idへ接続する |

## 14. WARNING / STOP

| ID | 条件 | 対応 |
|---|---|---|
| `WARN_CHANGE_SCOPE` | 影響範囲が広い | ユーザー確認 |
| `WARN_VERSION` | v0.1、v0.2等の版関係が不明 | 版関係を確認 |
| `WARN_RENAME` | ファイル名変更を含む | 参照切れ確認 |
| `WARN_EXISTING_FILE` | 既存ファイル変更あり | 差分確認 |
| `STOP_ARCHIVE` | archive配下変更 | USER_DECISIONまで停止 |
| `STOP_SECRET` | APIキー等混入 | 即停止、incident_id検討 |
| `STOP_PRIVACY` | 個人情報混入 | 即停止 |
| `STOP_PUBLIC` | 公開区分未確認 | 公開判断まで停止 |
| `STOP_DELETE` | 削除差分あり | 削除理由とUSER_DECISION確認 |
| `STOP_CONFLICT` | マージコンフリクトあり | マージ禁止 |

---

## 15. 禁止事項

- `change_id` なしに重要変更を処理済みにしない。
- 変更理由を記録せずに仕様を更新しない。
- 既存ファイル削除を `USER_DECISION` なしに行わない。
- `archive` 配下を `USER_DECISION` なしに変更しない。
- PDF原本、個人情報、黒塗り前資料、APIキーをGitHubに追加しない。
- コンフリクトがあるPRをマージしない。
- 差し戻し理由を記録せずに再PRを作らない。
- `incident_id` が必要な事故修正を通常変更として扱わない。
- ID接頭辞の不整合を放置しない。
- `main` へ直接コミットしない。
