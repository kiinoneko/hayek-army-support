# 案件台帳テンプレート v0.3

## 1. 目的

本仕様書は、後方支援軍の `ID体系一覧_相互接続仕様_v0.1.md` と `01_ID命名規則_接続ルール_停止ゲート_v0.2.md` を、実際の案件運用へ落とし込むための標準テンプレートである。

後方支援軍が自治体資料、議会発言、補助金、委託契約、政策、情報公開請求、判断停止、引継ぎを扱う際に、案件ごとの状態、根拠資料、未確認事項、警告事項、停止事項、ユーザー判断、次アクションを一目で確認できる台帳形式として使用する。

本仕様におけるIDは、証拠そのものではなく、証拠・判断・処理を追跡する索引である。正式な事実認定、政策評価、発言整理、情報公開請求、外部発信は、必ず根拠資料、判断記録、処理状態に戻って確認する。

## 2. 案件台帳の基本原則

| 原則 | 内容 |
|---|---|
| 案件中心 | 案件台帳は `case_id` を中心に管理する。 |
| 根拠優先 | `source_id` なしに `confirmed_facts` へ記載してはならない。 |
| 未確認分離 | 未確認事項は `gap_id` として管理し、確認済み事実と混在させない。 |
| 警告保持 | `warning_id` がある場合、最終評価や引継ぎで影響範囲を明記する。 |
| 停止尊重 | `stop_id` がある場合、自動進行せず、ユーザー判断又は追加資料確認を待つ。 |
| 判断分離 | `decision_id` なしに情報公開請求、外部提出、評価断定、引継ぎ確定を行わない。 |
| 接続必須 | `quote_id`、`request_id`、`policy_id` 等は孤立させず、関連IDへ接続する。 |
| 変更追跡 | 文案・評価・資料・状態・判断が変わった場合は `change_id` を付与する。 |
| 終了時保持 | 案件終了時も `remaining_gaps` と `remaining_warnings` を残す。 |
| 索引性 | IDは証拠ではなく索引である。 |

## 3. 案件区分

案件区分は、調査の重さ、外部提出の有無、政策影響、緊急性、継続監視の必要性に応じて設定する。

| 区分 | 用途 |
|---|---|
| 軽量案件 | 単発確認、簡易整理、短時間で完了する資料確認。 |
| 通常案件 | 複数資料を照合し、通常の調査・整理・報告を行う案件。 |
| 重要案件 | 政策評価、予算、契約、議会発言、外部発信に影響する案件。 |
| 緊急案件 | 期限、提出、議会日程、公開リスク等により即時対応が必要な案件。 |
| 継続監視案件 | 定期的に資料、発言、契約、予算、政策動向を確認する案件。 |
| 情報公開請求案件 | 情報公開請求、照会、確認依頼を伴う案件。 |
| 議会発言確認案件 | 議事録、動画、会見、SNS等の発言原文を確認する案件。 |
| 補助金・委託契約確認案件 | 補助金、委託料、契約書、仕様書、実績報告等を確認する案件。 |
| 政策評価案件 | 政策目的、受益者、負担者、納税者負担、効果を評価する案件。 |
| 資料整理案件 | 原資料、開示資料、保存先、公開区分、台帳接続を整理する案件。 |

## 4. 案件状態

案件状態は、現在どの工程で停止又は進行しているかを示す。状態は必ず `updated_at` と併せて更新する。

| 状態 | 意味 |
|---|---|
| 未着手 | 案件登録のみで、実作業に入っていない。 |
| 調査中 | 資料探索、整理、初期確認を行っている。 |
| 根拠資料確認中 | `source_id` 付与前後の根拠資料確認を行っている。 |
| 数値整合性確認中 | 予算、決算、契約額、補助金額等の整合性を確認している。 |
| 未確認事項あり | `gap_id` が残っている。 |
| warningあり | `warning_id` があり、評価又は外部発信時に注意が必要である。 |
| stop中 | `stop_id` が発生し、自動進行を停止している。 |
| ユーザー判断待ち | `decision_id` 付与前の判断待ちである。 |
| 請求文案作成前停止 | 情報公開請求文案の作成前に、判断又は資料確認で停止している。 |
| 請求文案作成済み | 請求文案の下書きが作成済みである。 |
| 提出待ち | 提出判断又は提出作業を待っている。 |
| 提出済み | 情報公開請求、照会、確認依頼等を提出済みである。 |
| 回答待ち | 外部機関からの回答を待っている。 |
| 回答受領 | 回答又は開示資料を受領した。 |
| 分析完了 | 根拠、未確認、警告、判断を整理し分析を完了した。 |
| 引継ぎ済み | `handoff_id` により別部門、別AI、別案件へ引き継いだ。 |
| 終了 | 案件を終了した。未解消事項がある場合も残置して記録する。 |
| 再調査 | 新資料、矛盾、追加判断により再調査へ戻した。 |

## 6. ID接続欄

ID接続欄は、案件に関連する全IDを集約し、台帳間の照合を可能にする欄である。

| ID | 記録単位 | 必須接続 |
|---|---|---|
| `case_id` | 案件単位 | 全関連ID。 |
| `source_id` | 資料単位 | `case_id`、必要に応じて `quote_id`、`policy_id`、`request_id`。 |
| `gap_id` | 未確認事項単位 | `case_id`、必要に応じて `request_id`、`decision_id`。 |
| `warning_id` | 警告事項単位 | `case_id`、必要に応じて `source_id`、`policy_id`。 |
| `stop_id` | 停止理由単位 | `case_id`、必要に応じて `decision_id`、`source_id`。 |
| `actor_id` | 発言者、担当者、関与者単位 | `quote_id`、`policy_id`、`case_id`。 |
| `quote_id` | 発言、答弁、引用単位 | 必ず `source_id` と接続する。 |
| `policy_id` | 政策、事業、補助金、委託契約単位 | `case_id`、`source_id`、必要に応じて `quote_id`、`decision_id`。 |
| `request_id` | 外部アクション単位 | `case_id`、`gap_id`、`decision_id`、必要に応じて `source_id`。 |
| `decision_id` | ユーザー判断単位 | `case_id`、必要に応じて `request_id`、`handoff_id`、`change_id`。 |
| `handoff_id` | 引継ぎ単位 | `case_id`、`decision_id`、`source_id`。 |
| `incident_id` | 事故、誤公開、誤断定単位 | `case_id`、`change_id`、`decision_id`。 |
| `change_id` | 変更単位 | `case_id`、必要に応じて `decision_id`、`incident_id`。 |

## 7. 根拠資料欄

根拠資料欄は、確認済み事実、数値、発言、政策整理の根拠を記録する欄である。

- `source_id` を付与した資料のみを根拠資料欄に記録する。
- 資料名、年度、公開元、取得日、URL又は保存先を必ず記録する。
- 信頼度は、公式資料、開示資料、議会資料、一次資料、二次資料の別が分かるように記録する。
- 内容確度は、本文確認済み、数値照合済み、要再確認、黒塗りあり等を明記する。
- 黒塗り、個人情報、未加工資料の扱いは、公開区分を確認してから記録する。

## 8. 未確認事項欄

未確認事項欄は、資料不足、照合未了、原文未確認、数値不一致、対象範囲不明などを `gap_id` として管理する欄である。

- `gap_id` が残る場合、`final_summary` で断定評価してはならない。
- 未確認事項は、必要資料、優先度、解消条件を併記する。
- 未確認事項を削除する場合は、解消根拠又は判断記録を残す。
- 情報公開請求で解消する場合は、対応する `request_id` と接続する。

## 9. 警告事項欄

警告事項欄は、処理停止までは要しないが、評価、公開、引継ぎ、外部提出時に無視できない注意点を `warning_id` として管理する欄である。

- `warning_id` がある場合、`final_summary` に影響範囲を明記する。
- 比較条件の違い、年度差、資料の粒度差、推定混入リスク、文脈注意を記録する。
- 警告事項は停止事項ではないが、最終評価で無視してはならない。

## 10. 停止事項欄

停止事項欄は、自動進行を止める必要がある理由を `stop_id` として管理する欄である。

- `stop_id` がある場合、`next_action` は原則として「ユーザー判断待ち」又は「追加資料確認」とする。
- 停止理由、停止工程、再開条件、必要な `decision_id`、必要な `source_id` を明記する。
- `stop_id` 発生後に、ユーザー判断なしで情報公開請求文案、外部提出、評価断定を確定してはならない。

## 11. ユーザー判断欄

ユーザー判断欄は、AIによる整理や提案と、ユーザーによる判断を分離するための欄である。

- `decision_id` なしに情報公開請求の提出判断を確定してはならない。
- 情報公開請求、照会、外部提出、引継ぎ、評価断定、公開範囲、停止解除は、原則としてユーザー判断欄に記録する。
- 判断内容、判断日、影響範囲、次工程を明記する。

## 12. 次アクション欄

次アクション欄は、案件を次にどの工程へ進めるかを示す欄である。

- `next_action` は一文で明確に記録する。
- `required_source` には次工程に必要な資料又は `source_id` を記録する。
- `required_decision` には次工程に必要な判断又は `decision_id` を記録する。
- `deadline`、`priority`、`handoff_target` を併記し、引継ぎ漏れを防止する。
- `stop_id` がある場合は、原則として「ユーザー判断待ち」又は「追加資料確認」以外へ進めない。

## 13. 引継ぎ欄

引継ぎ欄は、別部門、別AI、中核軍、法務官、前線部隊、外部レビューへ案件を渡す場合に使用する。

- 引継ぎ時は `handoff_id` を付与する。
- 引継ぎ内容には、案件目的、根拠資料、未確認事項、警告事項、停止事項、ユーザー判断、次アクションを含める。
- 受領確認の有無を記録し、引継ぎ後の責任線を明確にする。

## 14. 変更履歴欄

変更履歴欄は、文案・評価・資料・状態・判断が変わった場合に使用する。

- 変更が発生した場合は `change_id` を付与する。
- 変更対象、変更前、変更後、理由を記録する。
- 判断変更を伴う場合は、対応する `decision_id` と接続する。
- 誤記、誤公開、誤断定、個人情報混入等が関係する場合は、必要に応じて `incident_id` と接続する。

## 15. 運用例

## 16. 禁止事項

以下の行為を禁止する。

- 出典不明の数値を `confirmed_facts` に入れること。
- 未確認事項を消さずに解消扱いすること。
- `warning_id` を無視して最終評価すること。
- `stop_id` 発生後に自動で請求文案を確定すること。
- ユーザー判断なしに外部提出へ進むこと。
- `quote_id` に原文未確認の発言を正式登録すること。
- IDだけを根拠として扱うこと。
- 黒塗り・個人情報・未加工資料を不用意に公開すること。
- 推定と事実を同じ欄に混在させること。
