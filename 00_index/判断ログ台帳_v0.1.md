# 判断ログ台帳 v0.1

## 1. 判断ログ台帳の定義

判断ログ台帳は、後方支援軍において、`USER_DECISION`としてユーザーが行った重要判断を記録するための監査証跡台帳である。

情報公開請求へ進むか、法務官へ引き渡すか、中核軍へ引き渡すか、前線部隊へ引き渡すか、外部発信するか、政策評価断定へ進むか、追加資料を取得するか、案件を保留又は終了するか等の判断について、後から確認可能な形で管理する。

`case_log`が案件全体の進行履歴を記録するものであるのに対し、判断ログ台帳はユーザー判断そのものを記録する。

## 2. decision_id命名規則

`decision_id`は、必ず`DEC_`で始める。

基本形式は以下のいずれかとする。

- 汎用連番: `DEC_0001`
- 判断区分別連番: `DEC_{decision_type}_{連番}`

形式例:

- `DEC_0001`
- `DEC_REQUEST_001`
- `DEC_LEGAL_001`
- `DEC_CORE_001`
- `DEC_FRONT_001`
- `DEC_PUBLIC_001`
- `DEC_POLICY_001`
- `DEC_UPLOAD_001`
- `DEC_HOLD_001`
- `DEC_CLOSE_001`
- `DEC_GITHUB_001`
- `DEC_ARCHIVE_001`

命名時の注意:

- 同一案件内で重複しないこと。
- 後から参照できるよう、`case_log`、`handoff_id`、`request_id`候補等に同じ`decision_id`を記録すること。
- 判断を取消又は更新する場合も、元の`decision_id`を上書きせず、取消又は更新の判断を新しい`decision_id`で記録すること。

## 3. USER_DECISIONの対象

| 判断対象 | 内容 |
|---|---|
| 情報公開請求 | 情報公開請求へ進むか、文案作成へ進むか、提出を保留するか |
| 法務官引渡 | ダイシー型法務官へ条例、要綱、契約、同意人事、行政手続等の確認を渡すか |
| 中核軍引渡 | ハイエク中核軍へ制度評価、自由秩序評価、公共選択論評価を渡すか |
| 前線部隊引渡 | 請願、陳情、一般質問、公開質問、住民説明へ展開するか |
| 資料化 | 決算カード、比較カード、A4概要カード、発言カード等を作るか |
| 外部発信 | SNS、ブログ、公開資料、配布資料、議会向け資料として外部公開するか |
| 政策評価断定 | 評価候補から評価断定へ進むか |
| 追加資料 | 資料探索、アップロード、原文確認、PDF本文確認を行うか |
| GitHub公開 | GitHubへ仕様、テンプレート、成果物を公開するか |
| archive変更 | archive配下を変更するか |
| 保留 | 案件を一時停止するか |
| 終了 | 案件を終了するか |

## 4. 判断区分

| decision_type | 内容 |
|---|---|
| REQUEST | 情報公開請求関連 |
| LEGAL | 法務官確認関連 |
| CORE | 中核軍引渡関連 |
| FRONT | 前線部隊引渡関連 |
| DOC | 資料化関連 |
| PUBLIC | 外部発信関連 |
| POLICY | 政策評価断定関連 |
| UPLOAD | 資料取得・アップロード関連 |
| GITHUB | GitHub公開・PR・マージ関連 |
| ARCHIVE | archive配下変更関連 |
| HOLD | 保留 |
| CLOSE | 終了 |
| INCIDENT | インシデント対応関連 |

## 5. 判断状態

| 状態 | 内容 |
|---|---|
| 未判断 | まだ判断していない |
| 判断待ち | ユーザー判断を待っている |
| 承認 | 進行を承認した |
| 却下 | 進行しないと判断した |
| 保留 | 一時停止した |
| 差し戻し | 追加確認又は修正を求めた |
| 条件付き承認 | 条件を満たした場合のみ進行可 |
| 完了 | 判断に基づく処理が完了 |
| 取消 | 以前の判断を取り消した |

## 6. 判断ログ台帳の必須項目

| 項目 | 内容 | 必須 |
|---|---|---|
| decision_id | 判断ID | 必須 |
| case_id | 関連案件ID | 必須 |
| decision_type | 判断区分 | 必須 |
| 判断日 | 判断した日 | 必須 |
| 判断者 | 原則ユーザー | 必須 |
| 判断事項 | 何を判断したか | 必須 |
| 選択肢 | 取り得る選択肢 | 必須 |
| 判断結果 | 承認、却下、保留、差し戻し等 | 必須 |
| 判断理由 | 判断理由又は判断時の根拠 | 任意 |
| 関連source_id | 根拠資料 | 任意 |
| 関連gap_id | 未確認事項 | 任意 |
| 関連warning_id | 注意事項 | 任意 |
| 関連stop_id | 停止事項 | 任意 |
| 関連actor_id | 政治行政アクター | 任意 |
| 関連quote_id | 発言ID | 任意 |
| 関連handoff_id | 引渡ID | 任意 |
| 関連incident_id | インシデントID | 任意 |
| 次工程 | 判断後に行う作業 | 必須 |
| 備考 | 注意事項 | 任意 |

## 7. 判断前チェック

判断前に以下を確認する。

- `case_id`があるか。
- 判断対象が明確か。
- 判断に必要な`source_id`が整理されているか。
- `gap_id`が残っている場合、その影響が明記されているか。
- `warning_id` / `stop_id`が整理されているか。
- 情報公開請求、法的断定、外部発信、政策評価断定に該当しないか確認しているか。
- 外部公開の場合、資料公開区分と個人情報混入リスクを確認しているか。
- GitHub公開の場合、PDF原本、個人情報、黒塗り前資料、APIキーが含まれていないか確認しているか。
- archive変更の場合、旧版保護と復旧可能性を確認しているか。

## 8. 判断後の処理

判断後に以下を行う。

- 判断結果を`decision_id`で記録する。
- `case_log`に関連`decision_id`を記録する。
- 引渡が発生する場合は`handoff_id`を作成する。
- 情報公開請求へ進む場合は`request_id`候補を作成する。
- 外部発信へ進む場合は`PUBLIC`判断済みであることを明記する。
- 保留の場合は`HOLD`理由と再開条件を記録する。
- 終了の場合は`CLOSED`理由を記録する。
- 差し戻しの場合は必要な修正内容を明記する。

## 9. case_logとの関係

- `case_log`は案件全体の進行履歴を記録する。
- 判断ログ台帳はユーザー判断のみを記録する。
- `case_log`には関連`decision_id`を記録する。
- `decision_id`には関連`case_id`を必ず記録する。
- `USER_DECISION`が必要な案件は、判断ログ台帳なしに次工程へ進めない。

## 10. source_id / gap_id / warning_id / stop_idとの関係

- `source_id`は、判断時に参照した根拠資料を示す。
- `gap_id`は、判断時点で残っている未確認事項を示す。
- `warning_id`は、判断に影響し得る注意事項を示す。
- `stop_id`は、判断又は処理を停止すべき事項を示す。
- `gap_id`、`warning_id`、`stop_id`が存在する場合は、判断理由又は備考に判断への影響を記録する。
- `stop_id`がある判断事項は、自動で承認扱いにしない。
- `gap_id`が残る事項は、確認済みとして扱わない。

## 11. 引渡判断の記録

引渡判断は、以下の形式で記録する。

| decision_id | case_id | 引渡先 | 判断結果 | 引渡条件 | 関連handoff_id | 備考 |
|---|---|---|---|---|---|---|
|  |  | ハイエク中核軍 |  |  |  |  |
|  |  | ダイシー型法務官 |  |  |  |  |
|  |  | 前線部隊 |  |  |  |  |
|  |  | ナイチンゲール型資料化担当官 |  |  |  |  |
|  |  | ユーザー |  |  |  |  |

引渡先には、少なくとも以下を使用できる。

- ハイエク中核軍
- ダイシー型法務官
- 前線部隊
- ナイチンゲール型資料化担当官
- ユーザー

## 12. 外部発信判断の記録

外部発信判断は、以下の形式で記録する。

| decision_id | case_id | 発信対象 | 発信媒体 | 公開区分 | 判断結果 | 関連source_id | 関連stop_id | 備考 |
|---|---|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |  |  |

外部発信に関する必須ルール:

- 外部発信は`USER_DECISION`なしに進めない。
- 公開資料であっても、切り取り、誤認、個人攻撃、法的断定に注意する。
- 個人情報、黒塗り前資料、PDF原本、APIキーを含む場合は`STOP`とする。
- `actor_id`を用いる場合、個人責任、違法、不正、癒着、汚職を断定しない。

## 13. 情報公開請求判断の記録

情報公開請求判断は、以下の形式で記録する。

| decision_id | case_id | 請求対象 | 判断結果 | request_id候補 | 関連gap_id | 備考 |
|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |

情報公開請求に関する必須ルール:

- 情報公開請求は`USER_DECISION`なしに提出へ進まない。
- 後方支援軍は請求候補の整理まで行い、提出判断はユーザーが行う。
- 請求文案、提出日、受付番号、開示期限、開示結果は別途`request_id`で管理する。

## 14. 禁止事項

- `USER_DECISION`をAI判断で代替しない。
- 判断ログなしに情報公開請求へ進まない。
- 判断ログなしに外部発信へ進まない。
- 判断ログなしに法的断定へ進まない。
- 判断ログなしに政策評価断定へ進まない。
- 判断ログなしにarchive配下を変更しない。
- 判断ログなしにGitHub公開判断を行わない。
- `stop_id`がある判断事項を自動で承認扱いにしない。
- `gap_id`が残る事項を確認済みとして扱わない。
- 個人責任、違法、不正、癒着、汚職を`actor_id`だけで断定しない。
- PDF原本、個人情報、黒塗り前資料、APIキーをGitHubに追加しない。
- archive配下を自動変更しない。
