# request_id 情報公開請求台帳仕様 v0.1

## 1. request_idの定義

`request_id` は、後方支援軍における情報公開請求案件を一意に識別するための管理IDである。

`request_id` は、情報公開請求に関する以下の一連の過程を台帳化する。

- 請求前調査
- 請求候補の整理
- 請求文案の作成補助
- 提出判断
- 提出日
- 受付番号
- 開示期限
- 開示結果
- 受領資料の `source_id` 化
- 追加請求又は再請求
- 異議申立て又は不服申立て検討
- 関連する `source_id` / `gap_id` / `decision_id` との接続

`request_id` は単なる提出記録ではなく、公開情報で確認できた範囲、なお未確認の範囲、請求によって確認したい文書、提出判断、開示後の次工程を連結して管理するための台帳単位である。

## 2. 情報公開請求の位置づけ

情報公開請求は、公開情報で確認できる範囲を整理した後、なお見えない文書・根拠・内訳・契約・補助金・決裁・実績を確認するための制度的手段である。

情報公開請求は情報収集の開始点ではない。まず、公開資料、公式サイト、議会資料、予算書、決算書、主要施策成果報告書、監査資料、WAM、入札公告、契約情報、議事録、審議会資料等を確認する。

公開情報で確認できる範囲は `source_id` 化する。見えない部分、不明点、本文未確認、内訳不明、契約書未確認、補助金実績未確認等は `gap_id` 化する。その `gap_id` をもとに請求候補文書を特定する。

後方支援軍は、請求候補の整理、請求文案の作成補助、`gap_id` との接続までは行う。ただし、情報公開請求に進むか、実際に提出するか、請求範囲をどう設定するかの提出判断は、必ず `USER_DECISION` に留保する。

## 3. 請求前調査の原則

情報公開請求前には、公開情報で確認できる範囲を先に確認し、確認済み資料を `source_id` として整理する。

請求前調査では、少なくとも以下を確認する。

- 公式サイト
- 予算書
- 決算書
- 主要施策成果報告書
- 議会資料、議事録、議案資料
- 監査資料
- WAM、法人公開資料
- 入札公告、契約情報、指定管理情報
- 審議会資料、委員会記録
- 既存の `source_id`
- 既存の `gap_id`

請求前調査の結果、公開情報だけでは確認できない事項を `gap_id` として明確化し、その `gap_id` を解消するために必要な文書を請求候補として整理する。

## 4. request_id命名規則

`request_id` は必ず `REQ_` で始める。

命名は、案件横断の連番、自治体名、請求対象、制度区分等を用いて、後から検索しやすい形式にする。

形式例：

- `REQ_0001`
- `REQ_TAKETOYO_001`
- `REQ_MIHAMA_001`
- `REQ_CONTRACT_001`
- `REQ_SUBSIDY_001`
- `REQ_APPOINTMENT_001`
- `REQ_DISCLOSURE_001`
- `REQ_SHAKYO_001`
- `REQ_POLICY_001`

## 5. 請求状態区分

| 状態 | 内容 |
|---|---|
| 請求前調査中 | 公開情報を確認中 |
| 請求候補 | `gap_id` から請求候補文書を整理した状態 |
| 文案作成中 | 請求文案を作成中 |
| USER_DECISION待ち | 提出判断待ち |
| 提出予定 | ユーザーが提出を承認した状態 |
| 提出済み | 実際に請求を提出した状態 |
| 受付済み | 受付番号又は受付確認を得た状態 |
| 開示期限待ち | 開示決定期限を待っている状態 |
| 開示決定済み | 開示、部分開示、不開示等の決定が出た状態 |
| 資料受領済み | 開示資料を受領した状態 |
| source_id化中 | 受領資料を `source_id` 化している状態 |
| 追加請求候補 | 受領資料から追加請求が必要な状態 |
| 不服申立て検討 | 不開示又は部分開示に対する対応検討 |
| CLOSED | 請求案件を終了した状態 |
| HOLD | 保留 |

## 8. 請求前チェック

情報公開請求に進む前に、以下を確認する。

- 公開資料を確認したか
- 公式サイトを確認したか
- 予算書、決算書、主要施策成果報告書を確認したか
- 議会資料、議事録、議案資料を確認したか
- 監査資料を確認したか
- WAMや法人公開資料を確認したか
- 入札公告、契約情報、指定管理情報を確認したか
- 既存 `source_id` を確認したか
- `gap_id` を作成したか
- 請求対象文書名をできるだけ具体化したか
- 対象年度又は対象期間を明確にしたか
- 請求先機関を確認したか
- 請求文案が広すぎないか
- 個人情報、非公開情報、第三者情報が含まれる可能性を確認したか
- `USER_DECISION` を得たか

請求文案は、提出用最終案と区別して管理する。文案作成は後方支援軍が補助できるが、提出用最終案とする判断は `USER_DECISION` とする。

| request_id | 文案作成日 | 文案状態 | 対象文書 | 対象期間 | 注意事項 | USER_DECISION |
|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |

- 草案
- レビュー中
- 修正中
- USER_DECISION待ち
- 提出用最終案
- 提出済み
- 破棄

|  |  |  |  |  |  |

- 不服申立て検討
- 保留
- 終了

## 15. case_log / decision_id / gap_idとの関係

- `request_id` は必ず `case_id` と接続する。
- 情報公開請求へ進む判断は `decision_id` で記録する。
- `request_id` は `gap_id` を解消するために作成する。
- 開示資料を受領したら `source_id` 化する。
- 開示後に残った不明点は新たな `gap_id` 又は既存 `gap_id` の継続として管理する。
- `case_log` には `request_id` の状態を記録する。

## 16. USER_DECISION事項

以下はユーザー判断に留保する。

- 情報公開請求に進むか
- 請求文案を提出用最終案にするか
- 実際に提出するか
- 請求範囲を広げるか狭めるか
- 不開示・部分開示に対して追加請求するか
- 不服申立てを検討するか
- 法務官へ引き渡すか
- 外部発信するか
- 案件を保留又は終了するか

## 17. 禁止事項

- 公開情報の確認前に請求へ自動進行しない
- `gap_id` なしに請求候補を作らない
- `USER_DECISION` なしに請求提出へ進まない
- AI判断だけで提出済みにしない
- 請求文案を提出用最終案と混同しない
- 請求前調査と開示資料を混同しない
- 文書不存在を「事実不存在」と断定しない
- 不開示又は部分開示を直ちに不正と断定しない
- 開示資料を `source_id` 化せずに正式根拠化しない
- 開示資料に個人情報、黒塗り前資料、非公開情報が含まれる場合、GitHubに追加しない
- 外部発信へ自動進行しない
- 法的断定へ自動進行しない
- 政策評価断定へ自動進行しない
- `archive` 配下を変更しない
