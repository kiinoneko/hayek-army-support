# case_log 案件処理ログ仕様 v0.1

## 1. case_logの定義

case_logは、後方支援軍が扱う各案件について、受付、資料探索、source_id化、gap_id整理、分析、資料化、統制確認、USER_DECISION、引渡、停止、終了までの処理履歴を記録するための監査証跡である。

case_logは、案件が「いつ、誰により、何を根拠に、どこまで進み、どこで止まり、次に何をすべきか」を後から確認できるようにするために作成する。

case_logは評価断定文ではなく、案件の現在状態、処理履歴、根拠ID、未確認事項、停止理由、ユーザー判断、引渡内容を管理する台帳である。

## 2. case_id命名規則

case_idは、案件を一意に識別するためのIDであり、必ず `case_` で始める。

### 基本形式

- `case_YYYYMMDD_001`
- `case_対象団体_論点_YYYYMMDD_連番`
- `case_対象団体_論点_連番`
- `case_案件種別_連番`

### 形式例

- `case_YYYYMMDD_001`
- `case_taketoyo_shakyo_20260619_001`
- `case_mihama_shakyo_budget_001`
- `case_chita_council_quote_001`
- `case_gov_policy_001`
- `case_contract_001`
- `case_subsidy_001`
- `case_appointment_001`

### 命名上の注意

- case_idは重複させない。
- case_idは途中で変更しない。
- 日本語、空白、機種依存文字は避ける。
- 個人情報又は秘匿情報をcase_idに含めない。
- 案件名が変わってもcase_idは維持する。

## 3. 案件状態の区分

| 状態 | 内容 |
|---|---|
| 受付 | 案件を受け付けた状態 |
| 資料探索中 | 宮本常一型が資料候補を探索している状態 |
| source_id整理中 | 伊能忠敬型が資料台帳化している状態 |
| 本文確認中 | PDF又は資料本文を確認している状態 |
| 分析中 | 堀栄三型が矛盾・不足・比較注意を確認している状態 |
| 資料化中 | ナイチンゲール型がカード・表・概要資料化している状態 |
| 統制確認中 | 石田三成型がWARNING / STOP / USER_DECISIONを確認している状態 |
| USER_DECISION待ち | ユーザー判断が必要な状態 |
| 引渡済み | 中核軍、法務官、前線部隊、資料化担当へ引渡済み |
| STOP | 根拠不足、公開可否、法務確認等により停止 |
| HOLD | 保留 |
| CLOSED | 終了 |

## 4. 案件区分

| 案件区分 | 内容 |
|---|---|
| 軽量案件 | メモ、簡易確認、単一資料の確認 |
| 通常案件 | 数値、資料、議会発言、団体分析、公的財源を扱う案件 |
| 重要案件 | 情報公開請求、法務論点、外部発信、個人名、補助金、委託契約、政策評価に関わる案件 |
| 停止案件 | source_id不足、本文未確認、公開可否未確認、法務確認待ち等で停止している案件 |
| 終了案件 | 目的を達成し、追加作業が不要になった案件 |
| 保留案件 | ユーザー判断又は資料不足により一時保留している案件 |

## 5. 処理フェーズ

| phase | 内容 | 主担当 |
|---|---|---|
| Phase 0 | 案件受付 | 石田三成型｜統制本陣 |
| Phase 1 | 資料探索 | 宮本常一型｜情報収集担当官 |
| Phase 2 | 資料管理・source_id化 | 伊能忠敬型｜資料管理室長 |
| Phase 3 | 本文確認・数値確認 | 伊能忠敬型 / 堀栄三型 |
| Phase 4 | 分析・矛盾確認 | 堀栄三型｜情報分析官 |
| Phase 5 | 資料化 | ナイチンゲール型｜資料化担当官 |
| Phase 6 | 統制確認 | 石田三成型｜統制本陣 |
| Phase 7 | USER_DECISION | ユーザー |
| Phase 8 | 引渡又は停止 | 石田三成型｜統制本陣 |
| Phase 9 | 終了・継続管理 | 石田三成型｜統制本陣 |

## 7. 状態遷移ルール

- source_id未整理のまま正式資料化へ進めない。
- PDF存在確認のみで本文確認済みにしない。
- gap_idがある場合は解消、保留、又はUSER_DECISION待ちに分類する。
- warning_idがある場合は注意事項として明記する。
- stop_idがある場合は次工程へ自動進行しない。
- USER_DECISION事項はユーザー判断なしに解消済みにしない。
- 引渡済みの場合は引渡先、引渡日、引渡内容を記録する。
- CLOSEDにする場合は終了理由を記録する。

## 8. 停止条件

以下に該当する場合、案件はSTOP又はUSER_DECISION待ちにする。

- source_idがない。
- PDF本文未確認。
- 該当ページ又は該当欄が未確認。
- 数値の分母・分子が不明。
- 予算値、決算値、契約額、支払実績が混同されている。
- 補助金、委託料、交付金、制度収入が混同されている。
- quote_idなしで発言を正式言質化しようとしている。
- actor_idを根拠に個人責任、違法、不正、癒着、汚職を断定しようとしている。
- 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行しようとしている。
- PDF原本、個人情報、黒塗り前資料、APIキー混入リスクがある。
- archive配下を変更しようとしている。

## 9. USER_DECISION記録

USER_DECISIONは、ユーザー判断が必要な事項を記録するための台帳である。USER_DECISION事項は、ユーザー判断なしに解消済みにしてはならない。

| decision_id | case_id | 判断事項 | 選択肢 | 判断日 | 判断結果 | 備考 |
|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |

### USER_DECISIONが必要な事項

- 情報公開請求へ進むか。
- 法務官へ引き渡すか。
- 中核軍へ引き渡すか。
- 前線部隊へ引き渡すか。
- 外部発信するか。
- 政策評価断定へ進むか。
- 追加資料を取得又はアップロードするか。
- 案件を保留又は終了するか。
- GitHubへ公開するか。
- archive配下を変更するか。

## 10. 引渡記録

引渡記録は、案件を中核軍、法務官、前線部隊、資料化担当等へ渡した事実、根拠、範囲、残課題を記録する。

| handoff_id | case_id | 引渡日 | 引渡元 | 引渡先 | 引渡内容 | 関連source_id | 関連gap_id | USER_DECISION | 備考 |
|---|---|---|---|---|---|---|---|---|---|
|  |  |  |  |  |  |  |  |  |  |

case_logでは、以下の関連IDを管理する。

| ID | 内容 |
|---|---|
| source_id | 根拠資料ID |
| gap_id | 未確認事項ID |
| warning_id | 注意事項ID |
| stop_id | 停止事項ID |
| actor_id | 政治行政アクターID |
| quote_id | 発言ID |
| policy_id | 政策ID |
| decision_id | USER_DECISION ID |
| handoff_id | 引渡ID |
| incident_id | 事故、逸脱、混入、誤処理等の記録ID |
| change_id | 仕様変更、台帳変更、修正履歴ID |

## 12. 禁止事項

- case_logを評価断定文として使わない。
- source_idなしの事実を正式化しない。
- gap_idを根拠として扱わない。
- USER_DECISIONを省略しない。
- STOP状態の案件を自動で進めない。
- 情報公開請求へ自動進行しない。
- 法的断定へ自動進行しない。
- 外部発信へ自動進行しない。
- 政策評価断定へ自動進行しない。
- 個人責任を断定しない。
- PDF原本、個人情報、黒塗り前資料をGitHubに追加しない。
- archive配下を変更しない。
