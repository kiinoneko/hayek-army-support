# incident_idインシデント台帳仕様 v0.1

## 目的

本仕様は、後方支援軍において、誤公開、個人情報混入、黒塗り前資料混入、APIキー・パスワード・トークン等の秘密情報混入、PDF原本のGitHub追加、誤断定、誤マージ、archive破壊、情報公開請求の自動進行、外部発信事故などが発生した場合に、`incident_id` で記録・追跡・封じ込め・修正・再発防止を行うための仕様である。

`incident_id` は、事故を隠さず、何が起きたか、どの範囲に影響したか、どの対応を行ったか、どの判断を経たか、どの変更で修正したかを追跡するためのIDである。

---

## 1. `incident_id` の定義

- `incident_id` は、後方支援軍の運用中に発生した事故、誤処理、誤公開、秘密情報混入、誤断定、誤マージ等を記録するIDである。
- `incident_id` は事故そのものの根拠ではなく、事故対応履歴を追跡する索引である。
- インシデントの根拠は、関連 `case_id`、`source_id`、`change_id`、`decision_id`、PR番号、GitHub差分、case_log等に戻って確認する。
- 重大インシデントは `incident_id` なしに処理済みにしてはならない。
- インシデントは隠さず、初動、封じ込め、修正、再発防止、終了判断を記録する。

---

## 2. インシデント台帳の位置づけ

- 後方支援軍は事故をゼロにするだけでなく、事故発生時に復旧できる仕組みを持つ。
- `incident_id` は失敗の記録であり、責任追及ではなく再発防止を目的とする。
- ただし、個人情報、秘密情報、黒塗り前資料、誤公開、誤断定は重大事故として扱う。
- `incident_id` は `decision_id`、`change_id`、`case_id` と接続する。
- 重大インシデントの終了判断は、原則としてUSER_DECISIONを必要とする。

---

## 3. `incident_id` 命名規則

| 形式 | 用途 | 例 |
|---|---|---|
| `INC_YYYYMMDD_連番` | 汎用インシデント | `INC_20260625_001` |
| `INC_SECRET_連番` | APIキー・トークン等の秘密情報混入 | `INC_SECRET_001` |
| `INC_PRIVACY_連番` | 個人情報混入 | `INC_PRIVACY_001` |
| `INC_PUBLIC_連番` | 誤公開・外部発信事故 | `INC_PUBLIC_001` |
| `INC_GITHUB_連番` | GitHub誤追加・誤マージ | `INC_GITHUB_001` |
| `INC_ARCHIVE_連番` | archive破壊・旧版削除 | `INC_ARCHIVE_001` |
| `INC_QUOTE_連番` | 発言誤引用・原文未確認の言質化 | `INC_QUOTE_001` |
| `INC_VALUE_連番` | 数値誤記・誤計算 | `INC_VALUE_001` |
| `INC_LEGAL_連番` | 法的断定事故 | `INC_LEGAL_001` |
| `INC_REQUEST_連番` | 情報公開請求の誤提出・自動進行 | `INC_REQUEST_001` |

- 必ず `INC_` で始める。
- 一度発行した `incident_id` は別インシデントに再利用しない。
- 個人名、住所、電話番号、メールアドレス、秘密情報をID文字列に埋め込まない。
- 詳細は台帳項目に記録する。

---

## 4. インシデント分類

| incident_type | 内容 |
|---|---|
| PUBLIC | 外部公開事故、公開範囲誤り |
| PRIVACY | 個人情報混入 |
| SECRET | APIキー、パスワード、トークン等の秘密情報混入 |
| REDACTION | 黒塗り前資料、未加工資料の混入 |
| PDF | PDF原本をGitHubへ追加 |
| GITHUB | 誤PR、誤マージ、不要ファイル追加 |
| ARCHIVE | archive配下の削除、上書き、破壊 |
| SOURCE | source_idなしで正式化 |
| VALUE | 数値誤記、誤計算、分母分子誤り |
| QUOTE | 発言原文未確認の言質化、誤引用 |
| ACTOR | actor_id誤接続、人物誤認 |
| POLICY | policy_id誤接続、政策評価誤断定 |
| LEGAL | 違法、不正、癒着等の法的断定事故 |
| REQUEST | 情報公開請求の誤提出、自動進行 |
| HANDOFF | 引渡先、引渡内容、未確認事項の誤り |
| CHANGE | change_idなしの重要変更 |
| DECISION | decision_idなしのUSER_DECISION事項進行 |
| OTHER | その他 |

---

## 5. 重大度分類

| severity | 意味 | 例 | 初動 |
|---|---|---|---|
| SEV1 | 重大 | APIキー漏えい、個人情報公開、黒塗り前資料公開 | 即停止、封じ込め、USER_DECISION |
| SEV2 | 高 | 誤公開、法的断定、archive破壊、誤マージ | STOP、影響範囲確認、修正PR |
| SEV3 | 中 | source_idなし正式化、数値誤記、発言文脈誤り | 修正、warning又はstop |
| SEV4 | 低 | 表記揺れ、軽微なリンク切れ、テンプレ不整合 | 修正記録 |
| SEV5 | 記録のみ | 実害なしのヒヤリハット | 再発防止メモ |

- SEV1、SEV2は `decision_id` を必要とする場合がある。
- SEV1は削除だけで終わらせず、秘密情報無効化、履歴対応、影響範囲確認を検討する。
- SEV3以上は `change_id` による修正履歴を残す。

| 項目 | 内容 | 必須 |
|---|---|---|
| incident_id | インシデントID | 必須 |
| incident_type | インシデント分類 | 必須 |
| severity | 重大度 | 必須 |
| 発生日 | 発生又は発見日 | 必須 |
| 発見者・担当 | 発見者又は対応担当 | 必須 |

| 影響範囲 | 影響したファイル、PR、案件、外部公開範囲 | 必須 |
| 関連case_id | 関連案件 | 任意 |
| 関連source_id | 関連資料 | 任意 |
| 関連decision_id | 判断記録 | 必要に応じ必須 |
| 関連change_id | 修正変更 | 必要に応じ必須 |
| 関連PR番号 | GitHub PR番号 | 任意 |
| 初動対応 | 最初に行った対応 | 必須 |

| 再発防止策 | 今後の防止策 | 必須 |
| 終了判断 | 終了、継続監視、保留 | 必須 |
| 備考 | 注意事項 | 任意 |

## 7. 初動対応

| 手順 | 内容 |
|---|---|
| 1 | 作業を停止する |
| 2 | incident_idを発行する |
| 3 | 事故分類と重大度を仮判定する |
| 4 | 影響範囲を確認する |
| 5 | SECRET、PRIVACY、REDACTIONの場合は即時封じ込めを優先する |
| 6 | USER_DECISIONが必要か確認する |
| 7 | change_idが必要な修正か確認する |
| 8 | case_log又はPRレビュー記録に残す |

---

## 8. 封じ込め

- 誤公開ファイルを追加で拡散しない。
- 秘密情報をチャットやGitHubに再掲しない。
- APIキー、トークン、パスワードが関係する場合は無効化・再発行を検討する。
- 個人情報、黒塗り前資料が関係する場合は公開範囲を確認する。
- GitHub上のPRは閉じる、修正する、又はrevert PRを検討する。
- mainに入った場合は `change_id` と `decision_id` を接続して対応する。
- archive配下の破壊は復旧可否を確認する。

## 10. 修正・復旧

- 修正は原則としてPRで行う。
- 修正PRには `change_id` を接続する。
- 誤公開、個人情報、secret、黒塗り前資料が関係する場合は `decision_id` を検討する。
- mainへ入った誤りは、revert、修正PR、履歴対応のいずれかを検討する。
- 履歴削除や秘密情報無効化はUSER_DECISION対象とする。
- 修正後も再発防止策を記録する。

---

## 11. `decision_id` との関係

- 重大インシデント対応は `decision_id` と接続する。
- SECRET、PRIVACY、REDACTION、PUBLIC、ARCHIVE、LEGAL系はUSER_DECISIONを必要とする場合がある。
- インシデント終了判断はAI判断だけで行わない。
- `decision_id` は対応方針、終了判断、公開判断、履歴対応判断を記録する。

---

## 12. `change_id` との関係

- インシデント修正は `change_id` と接続する。
- `change_type` は原則 `INCIDENT_FIX` とする。
- 修正対象、変更前、変更後、変更理由を記録する。
- incident_idなしに重大事故修正を通常変更として扱わない。

---

## 13. `case_id` / `source_id` との関係

- インシデントが特定案件に関係する場合は `case_id` と接続する。
- 誤資料、誤数値、誤引用が関係する場合は `source_id` と接続する。
- source_idなしで正式化した事故は、関連 `gap_id` 又は `stop_id` も確認する。
- インシデントによって source_id の信頼度や内容確度に影響する場合は更新を検討する。

---

## 14. GitHub事故対応

| 事故 | 対応 |
|---|---|
| 誤PR | PRを閉じる又は修正する |
| 誤マージ | revert又は修正PRを検討 |
| コンフリクト放置 | マージ禁止、Codexへ解消指示 |
| PDF原本追加 | 削除、公開範囲確認、decision_id検討 |
| 個人情報追加 | 即停止、公開範囲確認、incident_id発行 |
| APIキー追加 | 即停止、無効化・再発行検討 |
| archive破壊 | 復旧、decision_id、change_id接続 |

---

## 15. Google Drive事故対応

- Drive上の共有範囲を確認する。
- 誤共有があれば共有解除又は権限変更を検討する。
- 黒塗り前資料、個人情報、開示資料の保存先を確認する。
- GitHubに置くべきでない資料がDriveから混入していないか確認する。
- Drive資料の公開URLをQRコード化する場合は公開区分を確認する。

---

## 16. 情報公開請求・外部発信事故対応

- 情報公開請求をUSER_DECISIONなしに提出済みにしない。
- 誤提出、宛先誤り、請求範囲誤りは `INC_REQUEST_連番` で管理する。
- 外部発信で誤断定、誤引用、個人情報混入があった場合は `INC_PUBLIC_連番` 又は該当分類で管理する。
- 訂正、撤回、再提出、補足説明はUSER_DECISION対象とする。

---

## 17. 再発防止

| 再発防止項目 | 内容 |
|---|---|
| チェックリスト追加 | PRレビュー、公開前チェック等へ反映 |
| テンプレート修正 | 禁止事項、STOP条件を追加 |
| Evals追加 | 同じ失敗をテストケース化 |
| SOP修正 | 手順の抜けを補正 |
| ID接続修正 | source_id、gap_id、decision_id等を再接続 |
| 教育メモ | 後方支援軍運用メモとして記録 |

## 19. WARNING / STOP

| ID | 条件 | 対応 |
|---|---|---|
| `WARN_INCIDENT_MINOR` | 軽微な誤記、表記揺れ | 修正記録 |
| `WARN_INCIDENT_SCOPE` | 影響範囲が不明 | 調査継続 |
| `WARN_INCIDENT_REPEAT` | 同種事故の再発 | Evals又はSOP修正 |
| `STOP_INCIDENT_SEV1` | SEV1事故 | 即停止、USER_DECISION |
| `STOP_SECRET` | APIキー等secret混入 | 無効化・再発行検討 |
| `STOP_PRIVACY` | 個人情報混入 | 公開停止、影響範囲確認 |
| `STOP_REDACTION` | 黒塗り前資料混入 | 公開停止 |
| `STOP_PUBLIC` | 外部公開事故 | 公開範囲確認 |
| `STOP_GITHUB` | GitHub誤公開・誤マージ | 修正PR又はrevert検討 |
| `STOP_ARCHIVE` | archive破壊 | 復旧・USER_DECISION |
| `STOP_LEGAL` | 法的断定事故 | 法務官確認 |
| `STOP_REQUEST` | 情報公開請求誤提出 | USER_DECISION |

---

## 20. 禁止事項

- 事故を記録せずに処理済みにしない。
- SEV1、SEV2をAI判断だけで終了扱いにしない。
- secretをチャットやGitHubに再掲しない。
- 個人情報を再掲しない。
- 黒塗り前資料を再アップロードしない。
- 誤公開を隠さない。
- incident_idなしに重大事故修正を通常変更として扱わない。
- change_idなしに修正済みにしない。
- decision_idなしに重大対応を終了しない。
- コンフリクトがあるPRをマージしない。
- archive配下をUSER_DECISIONなしに変更しない。
- PDF原本、個人情報、黒塗り前資料、APIキーをGitHubに追加しない。
- 法的断定事故を自己判断で修正完了扱いにしない。
- 情報公開請求の誤提出を記録せず放置しない。
