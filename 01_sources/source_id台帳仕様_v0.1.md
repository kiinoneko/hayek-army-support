# source_id台帳仕様 v0.1

## 1. 目的

本仕様は、後方支援軍が扱う資料・データ・PDF・Webページ・議事録・発言記録・契約資料・補助金資料・委託資料を、後から検証可能な形で管理するための `source_id` 台帳仕様である。

`source_id` は、後方支援軍における根拠管理の中心である。`source_id` のない情報は、正式な事実、正式数値、正式引用、正式成果物として扱わない。

## 2. source_id の定義

`source_id` とは、後方支援軍が根拠として扱う資料、データ、PDF、Webページ、議事録、発言記録、契約資料、補助金資料、委託資料等に対して付与する一意の資料IDである。

`source_id` は、次の目的で使用する。

- 根拠資料の所在、取得日、保存先、確認状態を追跡する
- 数値、引用、評価、成果物がどの資料に基づくかを明確にする
- 後日、第三者が根拠を再確認できる状態を維持する
- quote_id、actor_id、policy_id、gap_id、warning_id 等の関連IDと接続する
- GitHub上の成果物とGoogle Drive上の原本保存先を分離しつつ接続する

## 3. source_id の命名規則

`source_id` は必ず `SRC_` で始める。`S-0001` のような短縮形式は使用しない。

基本形式は次のいずれかとする。

- `SRC_0001`
- `SRC_TAKETOYO_001`
- `SRC_MIHAMA_001`
- `SRC_WAM_001`
- `SRC_GOV_001`
- `SRC_DIET_001`
- `SRC_PREF_AICHI_001`

命名規則は次のとおりである。

| 形式 | 用途 |
| --- | --- |
| `SRC_0001` | 汎用資料ID |
| `SRC_<組織又は領域>_<連番>` | 自治体、制度、組織、調査領域ごとの資料ID |
| `SRC_TAKETOYO_001` | 武豊町関連資料 |
| `SRC_MIHAMA_001` | 美浜町関連資料 |
| `SRC_WAM_001` | WAM関連資料 |
| `SRC_GOV_001` | 国、省庁、政府関連資料 |
| `SRC_DIET_001` | 国会、議事録、会議録関連資料 |
| `SRC_PREF_AICHI_001` | 愛知県関連資料 |

運用上の注意は次のとおりである。

- `source_id` は一度正式化したら原則として再利用しない
- 資料差替、年度違い、版違いは別の `source_id` として扱う
- 同一資料のPDF版とHTML版を同一根拠として扱う場合でも、形式差、取得日差、本文差がある場合は別ID又は備考で差異を明記する
- 未確認資料に仮IDを付与する場合も `SRC_` 形式を使用し、資料状態を「候補確認」又は「取得不可」等にする

## 4. source_id 台帳の必須項目

| 項目 | 内容 | 必須 |
| --- | --- | --- |
| source_id | 資料ID | 必須 |
| 資料名 | 正式資料名 | 必須 |
| 年度・対象期間 | 年度、日付、対象期間 | 必須 |
| 公開元 | 省庁、自治体、議会、団体等 | 必須 |
| URL又は取得元 | Web URL、取得経路 | 必須 |
| 取得日 | YYYY-MM-DD | 必須 |
| 保存先 | Google Drive等の保存場所 | 原本保存時必須 |
| ファイル形式 | PDF、HTML、Excel、CSV等 | 必須 |
| 該当ページ | ページ番号 | 数値・引用時必須 |
| 該当欄 | 科目、表、項目、発言箇所等 | 数値・引用時必須 |
| 本文確認状態 | 未確認、確認済 | 必須 |
| 資料状態 | 候補確認、存在確認済、原本保存済、本文確認済、正式化済等 | 必須 |
| 信頼度 | S / A / B / C / D / E | 必須 |
| 内容確度 | A / B / C | 必須 |
| 関連ID | quote_id、actor_id、policy_id、gap_id、warning_id等 | 任意 |
| 備考 | 差替可能性、注意点、未確認事項 | 任意 |

## 5. 資料状態区分

| 資料状態 | 意味 | 扱い |
| --- | --- | --- |
| 候補確認 | 資料候補として認識したが、存在又は取得可能性が未確定 | 正式根拠化不可 |
| 存在確認済 | 資料の存在、URL、公開元等を確認した状態 | 本文確認前のため数値・引用には使用不可 |
| 原本保存済 | PDF、Excel、CSV等の原本をGoogle Drive等に保存済み | 本文確認が別途必要 |
| 本文確認済 | 本文を開き、該当箇所を確認済み | 内容確度と該当ページ・該当欄に応じて根拠候補化可能 |
| 正式source_id化済 | 正式化条件を満たし、根拠管理上の正式IDとして確定 | 正式根拠候補 |
| 差替確認中 | 最新版、改訂版、差替版の有無を確認中 | 断定使用を避ける |
| 廃止 | 資料が廃止、撤回、又は根拠として不適切になった状態 | 新規成果物への使用禁止 |
| 取得不可 | URL切れ、非公開化、入手不能等により取得不可 | 代替資料探索又は備考記録が必要 |
| 要アップロード | 原本保存が必要だがGoogle Drive等への保存が未完了 | `WARN_DRIVE` 又は `STOP_PUBLIC` の対象 |

## 6. 信頼度区分

| 信頼度 | 情報源 | 扱い |
| --- | --- | --- |
| S | 原資料・一次資料 | 正式根拠候補 |
| A | 公式要約・公式概要・公式説明資料 | 根拠候補。ただし原資料確認優先 |
| B | 公的機関資料・制度資料・統計資料 | 補助根拠 |
| C | 報道・専門家記事・団体資料 | 参考情報 |
| D | SNS・ブログ・動画・演説・公開発言 | 言質候補・補助情報 |
| E | AI回答・未出典メモ・伝聞 | 事実化禁止 |

## 7. 内容確度区分

| 内容確度 | 意味 |
| --- | --- |
| A | 本文確認済み。該当ページ・該当欄を特定済み |
| B | 資料存在確認済みだが、該当箇所未特定又は本文確認不十分 |
| C | 候補資料又は取得未了。正式根拠化不可 |

## 8. PDF存在確認と本文確認の分離

PDFの存在確認と本文確認は別工程として扱う。

- PDFのURL、ファイル名、公開元、保存先を確認しただけでは「存在確認済」であり、「本文確認済」ではない
- PDFをダウンロード又はGoogle Driveに保存しただけでは「原本保存済」であり、「本文確認済」ではない
- 本文確認済とするには、PDF本文を開き、該当ページ、該当欄、該当表、該当発言箇所等を確認する
- 数値、引用、断定表現、政策評価、外部発信に使用する場合は、本文確認状態、該当ページ、該当欄を必ず記録する
- PDF本文未確認のまま主要数値又は引用を成果物に入れる場合は `STOP_DOC` とする

## 9. source_id と関連IDの関係

`source_id` は関連IDの根拠資料として機能する。

| 関連ID | source_id との関係 |
| --- | --- |
| quote_id | 引用文、発言、議事録抜粋等がどの資料のどの箇所に基づくかを接続する |
| actor_id | 発言者、組織、契約主体、補助金交付先等の根拠資料を接続する |
| policy_id | 政策、制度、事業、補助金、委託事業の根拠資料を接続する |
| gap_id | 不整合、未確認、欠落、制度上の隙間を示す際の根拠資料を接続する |
| warning_id | WARNING又はSTOPの発生理由となった資料不足、確認不足、公開リスクを接続する |

関連IDを作成する場合、可能な限り `source_id`、該当ページ、該当欄、本文確認状態を併記する。

## 10. Google Drive保存先との接続

GitHubには仕様、台帳テンプレート、公開可能な成果物を置き、PDF原本等はGoogle Driveに保存する。

Google Drive保存先は、台帳の「保存先」に記録する。原本保存時は保存先記録を必須とする。

Google Drive保存先には次を記録することが望ましい。

- 共有ドライブ名又はフォルダ名
- フォルダパス
- ファイル名
- 保存日
- 版又は取得日
- 公開可否、黒塗り有無、個人情報有無

Google Driveに置くものは次のとおりである。

- PDF原本
- 開示資料
- 黒塗り前後資料
- 成果物PDF
- 自治体資料
- WAM資料
- 議会資料

## 11. GitHubに置いてよいもの・置いてはいけないもの

### GitHubに置いてよいもの

- source_id台帳仕様
- 台帳テンプレート
- ドクトリン
- SOP
- RACI
- Evals
- 公開可能な成果物

### GitHubに置いてはいけないもの

- PDF原本
- 個人情報
- 黒塗り前資料
- 未公開資料
- メール本文
- APIキー
- 公開配慮が必要な開示文書

## 12. source_id正式化条件

`source_id` を正式化するには、次の条件をすべて満たす必要がある。

- 資料名が確認されている
- 公開元が確認されている
- 年度又は対象期間が確認されている
- URL又は取得元が確認されている
- 取得日が記録されている
- 本文確認状態が明記されている
- 数値・引用に使う場合は該当ページと該当欄が記録されている
- 原本を扱う場合はGoogle Drive保存先が記録されている
- 信頼度と内容確度が記録されている

正式化後の `source_id` は「正式source_id化済」とし、差替確認中、廃止、取得不可等へ状態変更する場合は備考に理由と日付を記録する。

## 13. source_id未確定時のWARNING / STOP

`source_id` が未確定、確認不足、又は公開リスクを含む場合は、WARNING又はSTOPを付与する。

### WARNING

| コード | 意味 | 典型例 |
| --- | --- | --- |
| `WARN_SRC` | source_id不足 | 根拠資料IDが未付与 |
| `WARN_PAGE` | 該当ページ不足 | 数値のページ番号が未記録 |
| `WARN_FIELD` | 該当欄不足 | 表、科目、発言箇所が未記録 |
| `WARN_DATE` | 取得日不明 | URLはあるが取得日がない |
| `WARN_VERSION` | 最新版・差替未確認 | 改訂版や差替版の有無が未確認 |
| `WARN_DRIVE` | Google Drive保存先未確認 | 原本保存先が未記録 |
| `WARN_CONFIDENCE` | 内容確度不足 | 内容確度B又はCのまま断定に近い使用をしている |

### STOP

| コード | 意味 | 典型例 |
| --- | --- | --- |
| `STOP_DOC` | PDF本文未確認 | PDFの存在だけで本文確認済として扱っている |
| `STOP_SOURCE` | 根拠資料不足 | 正式根拠に必要な資料がない |
| `STOP_VALUE` | 主要数値未取得 | 主要数値の取得、ページ、欄が未確定 |
| `STOP_PUBLIC` | 公開可否判断前 | 公開してよいか未判断の資料又は成果物が含まれる |
| `STOP_SECURITY` | 個人情報・黒塗り前資料・PDF原本混入リスク | GitHubに置いてはいけない資料が混入するおそれがある |

STOPが付与された項目は、解消されるまで正式成果物、外部発信、政策評価断定、情報公開請求、法的断定へ進めない。

## 15. 運用例

### 例1: 自治体PDFを根拠に数値を使う場合

1. 自治体WebサイトでPDFの存在を確認する
2. 台帳に `SRC_TAKETOYO_001` を作成し、資料状態を「存在確認済」とする
3. PDF原本をGoogle Driveに保存し、保存先を記録して「原本保存済」とする
4. PDF本文を開き、該当ページと該当欄を確認する
5. 本文確認状態を「確認済」、資料状態を「本文確認済」、内容確度をAにする
6. 正式化条件を満たしたら「正式source_id化済」とする

### 例2: 議事録の発言を引用する場合

1. 議事録の公開元、会議日、URL、取得日を記録する
2. `SRC_DIET_001` 又は自治体議会用の `SRC_` IDを付与する
3. 発言者、発言箇所、ページ、段落又は時刻を該当欄に記録する
4. quote_idを作成し、関連ID欄に記録する
5. 該当ページ又は発言箇所が未特定の場合は `WARN_PAGE` 又は `WARN_FIELD` とする

### 例3: WAM資料を補助根拠として扱う場合

1. `SRC_WAM_001` を付与する
2. 公開元、制度名、年度、URL、取得日を記録する
3. 原本を扱う場合はGoogle Drive保存先を記録する
4. 原資料であれば信頼度S、公式説明資料であれば信頼度A、公的制度資料であれば信頼度Bを付与する
5. 本文未確認の場合は内容確度B以下とし、正式根拠化しない

## 16. 禁止事項

次の行為を禁止する。

- source_idなしの情報を正式事実化しない
- PDF存在確認のみで本文確認済みとしない
- URLのみで正式source_id化しない
- 該当ページ不明の数値を正式化しない
- 該当欄不明の引用を正式化しない
- AI回答をsource_idにしない
- GitHubにPDF原本、個人情報、黒塗り前資料を追加しない
- 情報公開請求、法的断定、外部発信、政策評価断定へ自動進行しない

## 17. PR運用上の注意

本仕様の追加又は改訂にあたっては、次を守る。

- 既存ファイルを削除しない
- archive配下を変更しない
- PDF原本を追加しない
- 個人情報を追加しない
- 黒塗り前資料を追加しない
- mainへ直接コミットしない
